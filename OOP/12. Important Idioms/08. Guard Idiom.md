# Guard Idiom

> **One sentence:** Wrap a condition, resource, or critical section in an object whose constructor checks/acquires and whose destructor automatically reverses — so cleanup always happens, even when exceptions fly.

Also called: **Scope Guard**, **Lock Guard**, **Resource Guard**, **Finally idiom**.

## 🟢 Beginner — What Is a Guard and Why Does It Matter?

### The Problem: Every Exit Path Needs Cleanup

Imagine you lock a mutex, open a file, and then do some work. Any exit from that function — normal return, early return, or exception — must release the lock and close the file:

```cpp
void processFile(const std::string& path) {
    mtx.lock();
    FILE* f = fopen(path.c_str(), "r");

    if (!f) {
        mtx.unlock();      // ❌ must remember
        return;
    }

    if (parseError(f)) {
        fclose(f);         // ❌ must remember
        mtx.unlock();      // ❌ must remember — in right order!
        return;
    }

    doWork(f);             // what if this throws?
                           // ❌ fclose and unlock are NEVER called

    fclose(f);
    mtx.unlock();
}
```

Every new exit path means more cleanup calls to remember. Miss one → **deadlock, leak, or corruption**.

---

### The Guard Fix: Cleanup Is Automatic

A guard object ties cleanup to scope exit — you can't forget it:

```cpp
void processFile(const std::string& path) {
    std::lock_guard<std::mutex> lock(mtx);    // locks now, unlocks at scope exit ✓
    std::ifstream f(path);                    // opens now, closes at scope exit ✓

    if (!f) return;          // ✓ both cleaned up automatically

    if (parseError(f)) return; // ✓ both cleaned up automatically

    doWork(f);               // throws? ✓ both cleaned up during unwinding
}
// ✓ lock released, file closed — guaranteed, no matter how we leave
```

The function went from 4 manual cleanup calls across 3 exit paths to **zero** — the guard handles everything.

---

### How a Guard Works

A guard is just RAII applied to cleanup:

```
Constructor runs  →  acquire / save state
...object lives...
Destructor runs   →  release / restore state   ← always happens
```

```cpp
struct MutexGuard {
    explicit MutexGuard(std::mutex& m) : mtx_(m) {
        mtx_.lock();       // acquire in constructor
    }
    ~MutexGuard() {
        mtx_.unlock();     // release in destructor — even on exception
    }
    MutexGuard(const MutexGuard&) = delete;
    MutexGuard& operator=(const MutexGuard&) = delete;
private:
    std::mutex& mtx_;
};
```

This is exactly what `std::lock_guard` does.

---

### Guards vs Manual Cleanup

```
Manual cleanup:
  - Every new return/throw path needs its own cleanup
  - Order matters (unlock after fclose? or before?)
  - One forgotten call → bug in production
  - Code review must check every path

Guard:
  - Cleanup defined once, at construction
  - Order is guaranteed (reverse construction order)
  - Impossible to forget — destructor always runs
  - Code review only needs to check the guard itself
```

## 🟡 Intermediate — Writing Your Own Guards

### Pattern: Resource Guard

Use when you acquire a C-style resource that the standard library doesn't wrap:

```cpp
class FileGuard {
public:
    explicit FileGuard(const char* path, const char* mode)
        : file_(fopen(path, mode)) {
        if (!file_)
            throw std::runtime_error(std::string("Cannot open: ") + path);
    }

    ~FileGuard() {
        if (file_) fclose(file_);
    }

    FILE* get() const { return file_; }

    // No copy — ownership is exclusive
    FileGuard(const FileGuard&) = delete;
    FileGuard& operator=(const FileGuard&) = delete;

    // Allow move
    FileGuard(FileGuard&& other) noexcept : file_(other.file_) {
        other.file_ = nullptr;
    }

private:
    FILE* file_;
};

void readData(const std::string& path) {
    FileGuard f(path.c_str(), "r");
    // use f.get() ...
}   // fclose called automatically ✓
```

---

### Pattern: State Restore Guard

Use when you need to temporarily change a value and restore it afterward — regardless of how the function exits:

```cpp
template<typename T>
class RestoreGuard {
public:
    explicit RestoreGuard(T& value)
        : ref_(value), saved_(value) {}   // save current state

    ~RestoreGuard() {
        ref_ = saved_;                     // restore on exit — always
    }

    RestoreGuard(const RestoreGuard&) = delete;
    RestoreGuard& operator=(const RestoreGuard&) = delete;

private:
    T& ref_;
    T  saved_;
};

// Helper to deduce T automatically
template<typename T>
RestoreGuard<T> makeRestoreGuard(T& value) {
    return RestoreGuard<T>(value);
}
```

Usage:

```cpp
bool debugMode = false;

void diagnose() {
    auto guard = makeRestoreGuard(debugMode);
    debugMode = true;          // temporarily enable debug mode

    runDiagnostics();          // might throw — doesn't matter

}   // debugMode restored to false here ✓

// Real example: temporarily change floating-point rounding mode
void withRoundDown(auto fn) {
    auto guard = makeRestoreGuard(fegetround());   // save rounding mode
    fesetround(FE_DOWNWARD);
    fn();
}   // rounding mode restored ✓
```

---

### Pattern: Flag Guard

Use when entering/exiting a state needs to be tracked — like preventing recursive calls or signalling "in progress":

```cpp
class FlagGuard {
public:
    explicit FlagGuard(bool& flag) : flag_(flag) {
        if (flag_) throw std::logic_error("Re-entrant call detected!");
        flag_ = true;    // set on entry
    }

    ~FlagGuard() {
        flag_ = false;   // clear on exit
    }

    FlagGuard(const FlagGuard&) = delete;
    FlagGuard& operator=(const FlagGuard&) = delete;

private:
    bool& flag_;
};

class Widget {
public:
    void update() {
        FlagGuard guard(updating_);   // prevents re-entrant calls
        // ... update logic that might indirectly call update() again ...
    }

private:
    bool updating_ = false;
};
```

---

### Pattern: Transaction Guard

Use to roll back a multi-step operation if anything fails midway:

```cpp
class TransactionGuard {
public:
    explicit TransactionGuard(Database& db) : db_(db), committed_(false) {
        db_.beginTransaction();
    }

    ~TransactionGuard() {
        if (!committed_) {
            db_.rollback();    // auto-rollback if commit() was never called
        }
    }

    void commit() {
        db_.commit();
        committed_ = true;
    }

    TransactionGuard(const TransactionGuard&) = delete;
    TransactionGuard& operator=(const TransactionGuard&) = delete;

private:
    Database& db_;
    bool      committed_;
};

void transferFunds(Database& db, int from, int to, double amount) {
    TransactionGuard txn(db);

    db.debit(from, amount);    // might throw
    db.credit(to, amount);     // might throw

    txn.commit();              // only reached if both succeed
}   // if anything threw, rollback runs automatically ✓
```

## 🔴 Advanced — Scope Guards, Dismiss, and Condition Guards

### Generic Scope Guard

A scope guard runs **any callable** at scope exit — perfect for one-off cleanup without writing a class:

```cpp
template<typename F>
class ScopeGuard {
public:
    explicit ScopeGuard(F fn)
        : fn_(std::move(fn)), active_(true) {}

    ~ScopeGuard() {
        if (active_) fn_();
    }

    // Dismiss cancels the cleanup
    void dismiss() noexcept { active_ = false; }

    ScopeGuard(ScopeGuard&&) = delete;
    ScopeGuard(const ScopeGuard&) = delete;
    ScopeGuard& operator=(const ScopeGuard&) = delete;

private:
    F    fn_;
    bool active_;
};

// Deduction guide / factory
template<typename F>
ScopeGuard<F> makeScopeGuard(F&& fn) {
    return ScopeGuard<F>(std::forward<F>(fn));
}
```

Usage:

```cpp
void setupNetwork() {
    auto socket = createSocket();
    auto guard = makeScopeGuard([&]{ closeSocket(socket); });

    bindSocket(socket, port);    // might throw
    listenSocket(socket);        // might throw

    // success — keep the socket open, cancel cleanup
    guard.dismiss();
    activeSocket_ = socket;
}   // if anything threw, closeSocket called automatically ✓
```

---

### `dismiss()` — The Commit/Rollback Pattern

`dismiss()` is what separates a scope guard from simple RAII. It lets you:
- **Set up cleanup on entry** (as a safety net)
- **Cancel cleanup on success** (commit the work)

```cpp
void installPlugin(PluginManager& mgr, const std::string& path) {
    // Safety net: if anything fails, uninstall
    mgr.install(path);
    auto guard = makeScopeGuard([&]{ mgr.uninstall(path); });

    mgr.validate(path);       // might throw: validation failed
    mgr.initialize(path);     // might throw: init failed

    guard.dismiss();          // all steps succeeded — keep the plugin installed
}
// If validate() or initialize() threw → uninstall() runs automatically ✓
// If everything succeeded → uninstall() is cancelled ✓
```

### Three Flavours of Scope Guard

Different cleanup needs depending on *how* the scope exits:

```cpp
// Runs cleanup ALWAYS (on both success and failure)
template<typename F>
class ScopeExit {
public:
    explicit ScopeExit(F fn) : fn_(std::move(fn)) {}
    ~ScopeExit() { fn_(); }
    ScopeExit(const ScopeExit&) = delete;
private:
    F fn_;
};

// Runs cleanup only on FAILURE (exception in flight)
template<typename F>
class ScopeFailure {
public:
    explicit ScopeFailure(F fn)
        : fn_(std::move(fn))
        , exceptions_(std::uncaught_exceptions()) {}

    ~ScopeFailure() {
        if (std::uncaught_exceptions() > exceptions_) fn_();
    }
    ScopeFailure(const ScopeFailure&) = delete;
private:
    F   fn_;
    int exceptions_;
};

// Runs cleanup only on SUCCESS (no exception in flight)
template<typename F>
class ScopeSuccess {
public:
    explicit ScopeSuccess(F fn)
        : fn_(std::move(fn))
        , exceptions_(std::uncaught_exceptions()) {}

    ~ScopeSuccess() {
        if (std::uncaught_exceptions() == exceptions_) fn_();
    }
    ScopeSuccess(const ScopeSuccess&) = delete;
private:
    F   fn_;
    int exceptions_;
};
```

Usage:

```cpp
void processOrder(Order& order) {
    // Always: log that we attempted
    ScopeExit logAttempt([&]{
        logger.log("Processed order: " + order.id);
    });

    // Only on failure: send alert
    ScopeFailure alertOnFail([&]{
        alerting.send("Order failed: " + order.id);
    });

    // Only on success: update metrics
    ScopeSuccess updateMetrics([&]{
        metrics.increment("orders.success");
    });

    chargePayment(order);   // might throw
    shipItems(order);       // might throw
}
// Success: logAttempt + updateMetrics run
// Failure: logAttempt + alertOnFail run
```

### Condition Guard — Check-and-Set Atomically

Guards can enforce preconditions and postconditions around a block:

```cpp
template<typename Predicate>
class ConditionGuard {
public:
    explicit ConditionGuard(Predicate pre, Predicate post,
                            const char* msg = "Condition violated")
        : post_(post), msg_(msg) {
        if (!pre()) throw std::logic_error(
            std::string("Precondition failed: ") + msg);
    }

    ~ConditionGuard() noexcept(false) {
        if (!post_()) throw std::logic_error(
            std::string("Postcondition failed: ") + msg_);
    }

private:
    Predicate   post_;
    const char* msg_;
};

class BankAccount {
public:
    void withdraw(double amount) {
        ConditionGuard guard(
            [&]{ return balance_ >= amount; },      // pre:  enough funds
            [&]{ return balance_ >= 0; },           // post: not overdrawn
            "withdraw"
        );
        balance_ -= amount;
    }

private:
    double balance_ = 1000.0;
};
```

> ⚠️ Throwing from a destructor is dangerous (terminates if another exception is active). Use `noexcept(false)` with great care and only in debug/test builds.

## 🔵 Expert — Guard Composition, Thread Safety & C++23 Additions

### Composing Multiple Guards

Guards compose naturally — they clean up in **reverse construction order** (LIFO), which is usually exactly what you want:

```cpp
void openConnection() {
    // Construction order: socket → TLS → protocol
    SocketGuard    socket(host_, port_);   // 1st acquired
    TLSGuard       tls(socket.get());      // 2nd acquired
    ProtocolGuard  proto(tls.get());       // 3rd acquired

    negotiate(proto);   // might throw

    // Destruction order (on success or failure):
    // ~ProtocolGuard   ← 3rd destroyed (shutdown protocol)
    // ~TLSGuard        ← 2nd destroyed (teardown TLS)
    // ~SocketGuard     ← 1st destroyed (close socket)
    // Exactly the right order ✓
}
```

This LIFO order is guaranteed by the C++ standard for local variables in the same scope.

### Thread-Safe Guards

#### `std::lock_guard` vs `std::unique_lock` vs `std::scoped_lock`

```cpp
// lock_guard — simplest, single mutex, no unlock before scope end
{
    std::lock_guard<std::mutex> lock(mtx);
    sharedData.modify();
}   // unlocked ✓

// unique_lock — flexible: can unlock early, try-lock, defer-lock
{
    std::unique_lock<std::mutex> lock(mtx);
    if (someCondition) {
        lock.unlock();         // early unlock
        expensiveWork();       // don't hold lock during this
        lock.lock();           // re-lock before modifying
    }
    sharedData.modify();
}   // unlocked if still locked ✓

// scoped_lock (C++17) — deadlock-safe locking of MULTIPLE mutexes
{
    std::scoped_lock lock(mtx1, mtx2);   // locks both atomically — no deadlock ✓
    data1.modify();
    data2.modify();
}   // both unlocked ✓
```

#### `shared_lock` for Read-Write Locks (C++17)

```cpp
std::shared_mutex rwMutex;

void read() {
    std::shared_lock lock(rwMutex);    // multiple readers at once ✓
    return data_.value();
}

void write(int v) {
    std::unique_lock lock(rwMutex);    // exclusive write access ✓
    data_.setValue(v);
}
```

### Guard for `std::unique_lock` with Condition Variables

```cpp
class BoundedQueue {
public:
    void push(int val) {
        std::unique_lock<std::mutex> lock(mtx_);
        notFull_.wait(lock, [&]{ return queue_.size() < capacity_; });
        queue_.push(val);
        notEmpty_.notify_one();
    }   // lock released ✓

    int pop() {
        std::unique_lock<std::mutex> lock(mtx_);
        notEmpty_.wait(lock, [&]{ return !queue_.empty(); });
        int val = queue_.front();
        queue_.pop();
        notFull_.notify_one();
        return val;
    }   // lock released ✓

private:
    std::queue<int>         queue_;
    std::mutex              mtx_;
    std::condition_variable notFull_, notEmpty_;
    std::size_t             capacity_ = 10;
};
```

`unique_lock` must be used here because `condition_variable::wait()` unlocks and relocks the mutex internally — `lock_guard` can't do that.

### C++23: `std::scope_exit`, `std::scope_failure`, `std::scope_success`

C++23 standardises the three scope guard flavours as library types in `<scope>`:

```cpp
#include <scope>

void processFile(const std::string& path) {
    FILE* f = fopen(path.c_str(), "r");

    // Always close the file — even on exception
    std::scope_exit closeFile([&]{ if (f) fclose(f); });

    // Only log on failure
    std::scope_failure logFail([&]{
        std::cerr << "Failed to process: " << path << "\n";
    });

    // Only update cache on success
    std::scope_success updateCache([&]{
        cache_.markProcessed(path);
    });

    parse(f);    // might throw
    validate(f); // might throw
}
```

Before C++23, use the Boost equivalents (`boost::scope_exit`) or roll your own (see Advanced section).

### Implementing a Reusable Guard Library

A small, header-only guard library you can drop into any project:

```cpp
// guards.h
#pragma once
#include <utility>
#include <exception>

// Run fn() on scope exit — always
template<typename F>
struct [[nodiscard]] ScopeExit {
    explicit ScopeExit(F fn) : fn_(std::move(fn)) {}
    ~ScopeExit() noexcept { try { fn_(); } catch (...) {} }
    ScopeExit(ScopeExit&&) = delete;
    ScopeExit(const ScopeExit&) = delete;
private: F fn_;
};

// Run fn() only on failure (exception)
template<typename F>
struct [[nodiscard]] ScopeFailure {
    explicit ScopeFailure(F fn)
        : fn_(std::move(fn)), ex_(std::uncaught_exceptions()) {}
    ~ScopeFailure() noexcept {
        if (std::uncaught_exceptions() > ex_)
            try { fn_(); } catch (...) {}
    }
    ScopeFailure(ScopeFailure&&) = delete;
    ScopeFailure(const ScopeFailure&) = delete;
private: F fn_; int ex_;
};

// Run fn() only on success (no exception)
template<typename F>
struct [[nodiscard]] ScopeSuccess {
    explicit ScopeSuccess(F fn)
        : fn_(std::move(fn)), ex_(std::uncaught_exceptions()) {}
    ~ScopeSuccess() noexcept {
        if (std::uncaught_exceptions() == ex_)
            try { fn_(); } catch (...) {}
    }
    ScopeSuccess(ScopeSuccess&&) = delete;
    ScopeSuccess(const ScopeSuccess&) = delete;
private: F fn_; int ex_;
};

// Factory helpers — use these instead of constructing directly
template<typename F> auto onExit   (F&& f){ return ScopeExit    <std::decay_t<F>>(std::forward<F>(f)); }
template<typename F> auto onFailure(F&& f){ return ScopeFailure <std::decay_t<F>>(std::forward<F>(f)); }
template<typename F> auto onSuccess(F&& f){ return ScopeSuccess <std::decay_t<F>>(std::forward<F>(f)); }
```

Usage:

```cpp
#include "guards.h"

void doWork() {
    startTimer();
    auto _ = onExit([&]{ stopTimer(); });      // always stop timer

    openConnection();
    auto _2 = onFailure([&]{ closeConnection(); });  // rollback on fail

    processData();
    auto _3 = onSuccess([&]{ logSuccess(); }); // log on success

    heavyWork();   // might throw
}
```

### Common Pitfalls

```cpp
// ❌ Pitfall 1: Guard not given a name — destroyed immediately
makeScopeGuard([&]{ cleanup(); });   // temporary — destructs on this line!
auto guard = makeScopeGuard([&]{ cleanup(); });   // ✅ must name it

// ❌ Pitfall 2: Capturing dangling reference in lambda
ScopeGuard g([&value]{ process(value); });
// if 'value' is destroyed before 'g' — undefined behaviour
// Fix: capture by value, or ensure lifetimes

// ❌ Pitfall 3: Throwing from a destructor during stack unwinding
~ScopeGuard() {
    fn_();   // if fn_() throws while another exception is active → std::terminate
}
// Fix: wrap in try/catch inside the destructor

// ❌ Pitfall 4: Using lock_guard when you need to unlock mid-scope
std::lock_guard<std::mutex> lock(mtx);
expensiveIO();   // holds lock during slow I/O — bad for concurrency
// Fix: use unique_lock and call lock.unlock() before expensiveIO()

// ❌ Pitfall 5: Locking the same mutex twice with lock_guard
std::lock_guard<std::mutex> l1(mtx);
helper();   // helper() also tries to lock mtx → deadlock!
// Fix: use std::recursive_mutex, or restructure to avoid nested locking

// ❌ Pitfall 6: Forgetting [[nodiscard]] allows silently discarding guards
ScopeExit([&]{ cleanup(); });   // compiles without [[nodiscard]] — bug!
// Fix: mark guard types [[nodiscard]] so the compiler warns
```

## Standard Library Guards Cheat Sheet

| Guard | Header | Acquires | Releases | Notes |
|-------|--------|----------|----------|-------|
| `std::lock_guard` | `<mutex>` | `mutex.lock()` | `mutex.unlock()` | Simplest, no early unlock |
| `std::unique_lock` | `<mutex>` | `mutex.lock()` | `mutex.unlock()` | Flexible: defer, try, early unlock |
| `std::shared_lock` | `<shared_mutex>` | `mutex.lock_shared()` | `mutex.unlock_shared()` | Multiple readers |
| `std::scoped_lock` | `<mutex>` | Multiple mutexes | All mutexes | Deadlock-free multi-mutex |
| `std::scope_exit` | `<scope>` (C++23) | — | Calls lambda | Always runs |
| `std::scope_failure` | `<scope>` (C++23) | — | Calls lambda | Runs on exception only |
| `std::scope_success` | `<scope>` (C++23) | — | Calls lambda | Runs on success only |

## Quick Reference

```
🟢 Beginner    → Guards tie cleanup to scope exit — impossible to forget
                  Constructor acquires / checks, destructor releases / restores
                  std::lock_guard, std::ifstream are built-in guards
                  Cleanup runs even when exceptions unwind the stack

🟡 Intermediate → FileGuard, RestoreGuard, FlagGuard, TransactionGuard
                  commit() / dismiss() = cancel the cleanup on success
                  TransactionGuard: rollback unless commit() is called

🔴 Advanced    → Generic ScopeGuard<F> — run any lambda at scope exit
                  Three flavours: always / on-failure / on-success
                  Use std::uncaught_exceptions() to detect exception in flight
                  ConditionGuard: enforce pre/postconditions around a block

🔵 Expert      → Guards compose in LIFO order — reverse construction guaranteed
                  scoped_lock for deadlock-free multi-mutex acquisition
                  shared_lock + unique_lock for read-write separation
                  C++23 std::scope_exit / scope_failure / scope_success
                  [[nodiscard]] on guard types prevents accidental discard
```
