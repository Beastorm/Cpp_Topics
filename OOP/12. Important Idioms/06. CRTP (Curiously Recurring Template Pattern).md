# CRTP — Curiously Recurring Template Pattern

> **One sentence:** A class inherits from a template instantiated with itself as the argument — giving you the power of virtual dispatch with zero runtime cost.

```cpp
template<typename Derived>
class Base { ... };

class MyClass : public Base<MyClass> { ... };  // ← "curiously recurring"
```

Also called: **static polymorphism**, **F-bound polymorphism**.

## 🟢 Beginner — What Problem Does CRTP Solve?

### The Problem: Virtual Functions Have a Cost

Virtual functions let a base class call overridden methods in derived classes. But they come at a price:

```cpp
struct Animal {
    virtual void speak() = 0;   // runtime dispatch via vtable
    virtual ~Animal() = default;
};

struct Dog : Animal {
    void speak() override { std::cout << "Woof\n"; }
};

void makeSpeak(Animal& a) {
    a.speak();   // lookup in vtable at runtime — pointer chase every call
}
```

Every virtual call:
1. Loads the vtable pointer from the object
2. Indexes into the vtable
3. Follows another pointer to the actual function
4. Calls it — **never inlined**

For hot loops processing thousands of objects per frame, this adds up. You also can't use virtual functions with templates easily — template instantiation happens at compile time, virtual dispatch happens at runtime.

---

### The CRTP Fix: Dispatch at Compile Time

CRTP lets the base class call derived class methods **without a vtable** — the compiler resolves everything at compile time:

```cpp
template<typename Derived>
struct Animal {
    void speak() {
        // Cast to the derived type — safe because Derived IS the derived class
        static_cast<Derived*>(this)->speak_impl();
    }
};

struct Dog : Animal<Dog> {
    void speak_impl() { std::cout << "Woof\n"; }
};

struct Cat : Animal<Cat> {
    void speak_impl() { std::cout << "Meow\n"; }
};

template<typename T>
void makeSpeak(Animal<T>& a) {
    a.speak();   // resolved at compile time — can be inlined ✓
}
```

No vtable. No pointer chase. The call is resolved at compile time and can be fully inlined.

---

### The Trick Explained

The key insight is the `static_cast`:

```cpp
template<typename Derived>
struct Base {
    void interface() {
        // We know this object IS a Derived (because of how CRTP works)
        // so the cast is safe
        static_cast<Derived*>(this)->implementation();
    }
};
```

When `Dog` inherits from `Base<Dog>`, `this` in `Base` is actually pointing at a `Dog` object. The `static_cast<Derived*>` is a downcast that the compiler validates at compile time — no runtime check needed.

### Virtual vs CRTP Side by Side

```cpp
// ── Virtual (runtime polymorphism) ──────────────────────
struct Shape {
    virtual double area() const = 0;   // vtable entry
};

struct Circle : Shape {
    double r;
    double area() const override { return 3.14159 * r * r; }
};

void printArea(Shape& s) { std::cout << s.area(); }   // runtime dispatch

// ── CRTP (compile-time polymorphism) ────────────────────
template<typename Derived>
struct Shape {
    double area() const {
        return static_cast<const Derived*>(this)->area_impl();
    }
};

struct Circle : Shape<Circle> {
    double r;
    double area_impl() const { return 3.14159 * r * r; }
};

template<typename S>
void printArea(Shape<S>& s) { std::cout << s.area(); }  // compile-time dispatch
```

## 🟡 Intermediate — Common CRTP Patterns

### Pattern 1 — Adding a Common Interface

CRTP lets you add methods to all derived classes from a single base — without repeating code in each derived class:

```cpp
template<typename Derived>
struct Printable {
    void print() const {
        std::cout << static_cast<const Derived*>(this)->toString() << "\n";
    }

    void printUpperCase() const {
        std::string s = static_cast<const Derived*>(this)->toString();
        std::transform(s.begin(), s.end(), s.begin(), ::toupper);
        std::cout << s << "\n";
    }
};

struct Color : Printable<Color> {
    int r, g, b;
    std::string toString() const {
        return "rgb(" + std::to_string(r) + ","
                      + std::to_string(g) + ","
                      + std::to_string(b) + ")";
    }
};

struct Point : Printable<Point> {
    double x, y;
    std::string toString() const {
        return "(" + std::to_string(x) + ", " + std::to_string(y) + ")";
    }
};

Color c{255, 128, 0};
c.print();           // rgb(255,128,0)
c.printUpperCase();  // RGB(255,128,0)

Point p{3.0, 4.0};
p.print();           // (3.000000, 4.000000)
```

Both `Color` and `Point` get `print()` and `printUpperCase()` for free — defined once in `Printable`.

### Pattern 2 — Static Interface Enforcement

CRTP acts like a compile-time abstract base class. If the derived class forgets to implement a required method, you get a clear compile error:

```cpp
template<typename Derived>
struct Serializable {
    std::string serialize() const {
        // This fails to compile if Derived doesn't have serialize_impl()
        return static_cast<const Derived*>(this)->serialize_impl();
    }

    static Derived deserialize(const std::string& s) {
        return Derived::deserialize_impl(s);
    }
};

struct Config : Serializable<Config> {
    std::string host;
    int         port;

    std::string serialize_impl() const {
        return host + ":" + std::to_string(port);
    }

    static Config deserialize_impl(const std::string& s) {
        auto colon = s.find(':');
        Config c;
        c.host = s.substr(0, colon);
        c.port = std::stoi(s.substr(colon + 1));
        return c;
    }
};

Config cfg{"localhost", 8080};
std::string data = cfg.serialize();         // "localhost:8080"
Config cfg2 = Config::deserialize(data);    // restored ✓
```

### Pattern 3 — Counting Instances (CRTP for per-type state)

CRTP gives each derived type its **own static data**, because each instantiation of the template is a distinct class:

```cpp
template<typename Derived>
struct InstanceCounter {
    static int count;

    InstanceCounter()  { ++count; }
    ~InstanceCounter() { --count; }

    static int instanceCount() { return count; }
};

// Each instantiation gets its OWN static variable
template<typename Derived>
int InstanceCounter<Derived>::count = 0;

struct Widget : InstanceCounter<Widget> { std::string name; };
struct Dialog : InstanceCounter<Dialog> { int id; };

Widget w1, w2, w3;
Dialog d1;

std::cout << Widget::instanceCount();  // 3
std::cout << Dialog::instanceCount();  // 1
// Each type has its own counter — they don't share ✓
```

### Pattern 4 — `enable_shared_from_this` (Standard Library CRTP)

`std::enable_shared_from_this` is the most famous CRTP class in the standard library:

```cpp
class Node : public std::enable_shared_from_this<Node> {
public:
    std::shared_ptr<Node> getShared() {
        return shared_from_this();   // safe — returns a shared_ptr to *this
    }
};

auto node = std::make_shared<Node>();
auto same = node->getShared();   // both point to the same Node ✓

// Without CRTP (wrong way):
class BadNode {
    std::shared_ptr<BadNode> getShared() {
        return std::shared_ptr<BadNode>(this);   // ❌ double-free!
    }
};
```

## 🔴 Advanced — CRTP for Mixins, Policy Classes & Operator Generation

### Mixins — Composing Behaviour Without Inheritance Hell

CRTP mixins let you add orthogonal features to a class by layering multiple base classes, each independently testable:

```cpp
// Mixin 1: adds comparison operators
template<typename Derived>
struct Comparable {
    friend bool operator==(const Derived& a, const Derived& b) {
        return !(a < b) && !(b < a);
    }
    friend bool operator!=(const Derived& a, const Derived& b) { return !(a == b); }
    friend bool operator> (const Derived& a, const Derived& b) { return b < a; }
    friend bool operator<=(const Derived& a, const Derived& b) { return !(b < a); }
    friend bool operator>=(const Derived& a, const Derived& b) { return !(a < b); }
    // Derived only needs to provide operator<
};

// Mixin 2: adds arithmetic operators
template<typename Derived>
struct Arithmetic {
    friend Derived operator+(Derived lhs, const Derived& rhs) {
        lhs += rhs;   // Derived must provide operator+=
        return lhs;
    }
    friend Derived operator-(Derived lhs, const Derived& rhs) {
        lhs -= rhs;   // Derived must provide operator-=
        return lhs;
    }
};

// Mixin 3: adds stream output
template<typename Derived>
struct Streamable {
    friend std::ostream& operator<<(std::ostream& os, const Derived& d) {
        return os << d.toString();   // Derived must provide toString()
    }
};

// Compose all mixins — Vector2D gets all operators for free
struct Vector2D
    : Comparable<Vector2D>
    , Arithmetic<Vector2D>
    , Streamable<Vector2D>
{
    double x, y;

    Vector2D(double x, double y) : x(x), y(y) {}

    bool operator<(const Vector2D& o) const {   // for Comparable
        return (x*x + y*y) < (o.x*o.x + o.y*o.y);
    }

    Vector2D& operator+=(const Vector2D& o) {   // for Arithmetic
        x += o.x; y += o.y; return *this;
    }

    Vector2D& operator-=(const Vector2D& o) {   // for Arithmetic
        x -= o.x; y -= o.y; return *this;
    }

    std::string toString() const {              // for Streamable
        return "(" + std::to_string(x) + ", " + std::to_string(y) + ")";
    }
};

Vector2D a{3, 4}, b{1, 2};

std::cout << (a < b);    // false — a is longer ✓
std::cout << (a == b);   // false ✓
std::cout << a + b;      // (4.000000, 6.000000) ✓
```

Each mixin is a separate, reusable class. Adding `Comparable` to any type only requires implementing `operator<`. No code duplication.

### Policy Classes via CRTP

CRTP is useful for swapping behaviour at compile time without virtual overhead:

```cpp
// Policy: how to allocate memory
template<typename Derived>
struct HeapAllocPolicy {
    void* allocate(std::size_t n) { return ::operator new(n); }
    void  deallocate(void* p)     { ::operator delete(p); }
};

template<typename Derived>
struct PoolAllocPolicy {
    void* allocate(std::size_t n) { return pool_.alloc(n); }
    void  deallocate(void* p)     { pool_.free(p); }
private:
    MemoryPool pool_;
};

// Container picks its allocation policy at compile time — zero overhead
template<typename T, typename AllocPolicy = HeapAllocPolicy<void>>
class Container : public AllocPolicy {
public:
    T* create() {
        void* mem = this->allocate(sizeof(T));
        return new(mem) T();
    }
    void destroy(T* p) {
        p->~T();
        this->deallocate(p);
    }
};

Container<Widget>                   heapContainer;    // uses heap
Container<Widget, PoolAllocPolicy<void>> poolContainer;   // uses pool
// No virtual calls — policy chosen at compile time ✓
```

### Generating All Comparison Operators from One

CRTP is so useful for operator generation that C++20 standardised this pattern as `operator<=>` (spaceship operator). But understanding the CRTP version teaches you what `<=>` does under the hood:

```cpp
template<typename Derived>
struct TotalOrder {
    // Derived provides:  int compare(const Derived&) const
    //   returns: < 0 if less, 0 if equal, > 0 if greater

    bool operator==(const Derived& o) const { return derived().compare(o) == 0; }
    bool operator!=(const Derived& o) const { return derived().compare(o) != 0; }
    bool operator< (const Derived& o) const { return derived().compare(o) <  0; }
    bool operator<=(const Derived& o) const { return derived().compare(o) <= 0; }
    bool operator> (const Derived& o) const { return derived().compare(o) >  0; }
    bool operator>=(const Derived& o) const { return derived().compare(o) >= 0; }

private:
    const Derived& derived() const {
        return *static_cast<const Derived*>(this);
    }
};

struct Version : TotalOrder<Version> {
    int major, minor, patch;

    int compare(const Version& o) const {
        if (major != o.major) return major - o.major;
        if (minor != o.minor) return minor - o.minor;
        return patch - o.patch;
    }
};

Version v1{1, 2, 3}, v2{1, 3, 0};
std::cout << (v1 < v2);    // true ✓
std::cout << (v1 == v2);   // false ✓
std::cout << (v2 > v1);    // true ✓
```

## 🔵 Expert — CRTP vs Concepts, Deducing `this` (C++23) & Pitfalls

### Protecting the Base Constructor

A subtle problem: nothing stops someone from writing `class Foo : public Base<Bar>` — inheriting from the wrong instantiation. Protect against this:

```cpp
template<typename Derived>
struct Base {
protected:
    // Only Derived (and its subclasses) can construct Base<Derived>
    Base() {
        // Compile-time check: Derived must actually derive from Base<Derived>
        static_assert(std::is_base_of_v<Base<Derived>, Derived>,
            "Derived must inherit from Base<Derived>");
    }

private:
    // Private constructor + friend: ONLY Derived can instantiate Base<Derived>
    friend Derived;
};
```

---

### Preventing Object Slicing in CRTP

With virtual functions, slicing is a known hazard. CRTP has its own version:

```cpp
template<typename Derived>
struct Base {
    Derived clone() const {
        return *static_cast<const Derived*>(this);   // fine
    }
};

struct Child : Base<Child> { int extra; };

Base<Child> b = Child{};     // ❌ slices — extra is lost
                              // but Base<Child> IS-A Child... this is confusing

// Better: make Base non-copyable directly
template<typename Derived>
struct Base {
protected:
    Base() = default;
    Base(const Base&) = default;
};
```

### CRTP and Virtual Together (Mixing the Two)

You can combine CRTP with virtual functions — CRTP for the performance-critical hot path, virtual for the rare plugin/extension path:

```cpp
template<typename Derived>
struct FastRenderer {
    // Hot path — compile-time dispatch, inlineable
    void renderFrame() {
        static_cast<Derived*>(this)->renderFrame_impl();
    }
};

struct VulkanRenderer : FastRenderer<VulkanRenderer> {
    void renderFrame_impl() { /* Vulkan draw calls */ }

    // Cold path — virtual for plugin system
    virtual void onWindowResize(int w, int h) { ... }
};
```

### C++20 Concepts: A Cleaner Alternative for Interface Enforcement

One major use of CRTP is enforcing that derived classes implement certain methods. C++20 Concepts do this more cleanly for that specific use case:

```cpp
// CRTP approach (pre-C++20)
template<typename Derived>
struct Serializable {
    std::string serialize() const {
        return static_cast<const Derived*>(this)->serialize_impl();
    }
};

// C++20 Concepts approach — cleaner, no inheritance needed
template<typename T>
concept Serializable = requires(const T& t) {
    { t.serialize_impl() } -> std::convertible_to<std::string>;
};

template<Serializable T>
void save(const T& obj) {
    auto data = obj.serialize_impl();
    // ...
}
```

| Use Case | CRTP | Concepts (C++20) |
|----------|:----:|:----------------:|
| Interface enforcement | ✅ | ✅ Better |
| Code injection (adding methods) | ✅ | ❌ Not possible |
| Operator generation (mixins) | ✅ | ❌ Not possible |
| Per-type static state | ✅ | ❌ Not possible |
| Zero-overhead dispatch | ✅ | ✅ |
| Readable error messages | ⚠️ Cryptic | ✅ Clear |

**Rule:** Use Concepts for constraint checking. Use CRTP when you need to *inject* shared code into derived classes.

### C++23: Deducing `this` — The CRTP Killer?

C++23 introduces **explicit object parameters** (`deducing this`), which can replace many CRTP uses with simpler syntax:

```cpp
// CRTP (C++11)
template<typename Derived>
struct Printable {
    void print() const {
        std::cout << static_cast<const Derived*>(this)->toString() << "\n";
    }
};

struct Color : Printable<Color> {
    std::string toString() const { return "red"; }
};

// Deducing this (C++23) — no template base class needed!
struct Printable {
    void print(this const auto& self) {
        std::cout << self.toString() << "\n";   // self is deduced as the derived type
    }
};

struct Color : Printable {
    std::string toString() const { return "red"; }
};

Color c;
c.print();   // "red" — toString() resolved at compile time ✓
```

`deducing this` also solves the ref-qualifier duplication problem:

```cpp
// CRTP — must write three overloads
template<typename Derived>
struct Base {
    auto& value() &       { return static_cast<Derived*>(this)->value_; }
    auto& value() const & { return static_cast<const Derived*>(this)->value_; }
    auto  value() &&      { return std::move(static_cast<Derived*>(this)->value_); }
};

// Deducing this — one overload handles all
struct Base {
    auto&& value(this auto&& self) {
        return std::forward<decltype(self)>(self).value_;
    }
};
```

**When to still use CRTP over deducing this:**
- You need C++11/14/17 compatibility
- You need per-type static state (`static int count` per derived type)
- You need to inject state (data members) via base class

### Common Pitfalls

```cpp
// ❌ Pitfall 1: Calling derived method from base constructor
template<typename Derived>
struct Base {
    Base() {
        static_cast<Derived*>(this)->init();   // ❌ Derived isn't constructed yet!
    }
};
// Fix: use a two-phase init, or call init() from Derived's constructor

// ❌ Pitfall 2: Wrong template argument
struct Foo : Base<Bar> { ... };   // Foo inherits Base<Bar> — the cast will be wrong!
// Fix: static_assert(std::is_base_of_v<Base<Derived>, Derived>)

// ❌ Pitfall 3: Storing in a heterogeneous container
std::vector<Base<?>*> shapes;    // ❌ can't — Base<Circle> and Base<Square>
                                 //           are unrelated types
// Fix: if you need heterogeneous storage, use virtual functions or type erasure

// ❌ Pitfall 4: Accessing derived members before full construction
template<typename Derived>
struct Base {
    void setup() { static_cast<Derived*>(this)->data_ = 42; }  // data_ may not exist yet
};

// ❌ Pitfall 5: Accidental object slicing
void process(Base<MyClass> b) { ... }   // takes by value — slices Derived part!
// Fix: always pass CRTP objects by reference or pointer, never by value
```

## CRTP vs Virtual Dispatch

| Property | Virtual | CRTP |
|----------|:-------:|:----:|
| Dispatch time | Runtime | Compile time |
| Inlineable | ❌ Never | ✅ Always |
| Heterogeneous containers | ✅ Easy (`vector<Base*>`) | ❌ Hard (type erasure needed) |
| Runtime polymorphism | ✅ Yes | ❌ No |
| Zero overhead | ❌ vtable + pointer | ✅ Yes |
| Code in base applies to all derived | ✅ Yes | ✅ Yes |
| Works with templates | ❌ Poorly | ✅ Naturally |
| Readable errors | ✅ Clear | ⚠️ Cryptic |

**Use virtual when:** you need to store mixed derived types in a container, or decide behaviour at runtime (plugins, scripts, user input).

**Use CRTP when:** you know all types at compile time, you're in a hot loop, or you want to inject shared code without virtual overhead.

## Quick Reference

```
🟢 Beginner    → class Derived : public Base<Derived>
                  Base calls static_cast<Derived*>(this)->method()
                  No vtable, no pointer chase, fully inlineable

🟡 Intermediate → Add common interface to all derived classes from one base
                  Per-type static state (each instantiation is its own class)
                  std::enable_shared_from_this is CRTP in the standard library

🔴 Advanced    → Mixin layers: inherit multiple CRTP bases for orthogonal features
                  Policy classes: swap behaviour at compile time
                  Generate all comparison/arithmetic operators from one method

🔵 Expert      → static_assert to prevent wrong template argument
                  Never call derived methods from base constructor
                  C++20 Concepts for interface enforcement (simpler than CRTP)
                  C++23 deducing this replaces many CRTP uses — prefer it when available
                  CRTP can't do heterogeneous containers — use virtual or type erasure
```
