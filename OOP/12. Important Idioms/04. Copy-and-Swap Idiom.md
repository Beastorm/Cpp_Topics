# Copy-and-Swap Idiom

> **One sentence:** Make a copy first, swap it with `*this`, then let the old data destroy itself — giving you exception-safe assignment in one clean trick.

## 🟢 Beginner — What Problem Does It Solve?

### The Naive Copy Assignment — Three Hidden Bugs

When you write a copy assignment operator for a resource-managing class, you face three problems at once:

```cpp
// A simple buffer class
class Buffer {
public:
    explicit Buffer(int size)
        : data_(new int[size]), size_(size) {}

    ~Buffer() { delete[] data_; }

private:
    int* data_;
    int  size_;
};
```

A naive assignment operator looks like this:

```cpp
// ❌ Naive — has all three problems
Buffer& operator=(const Buffer& other) {
    delete[] data_;                       // Problem 1: self-assignment
    data_ = new int[other.size_];         // Problem 2: exception here leaks nothing
    size_ = other.size_;                  //            but *this is already broken
    std::copy(other.data_,                // Problem 3: code duplication
              other.data_ + size_,            with copy constructor
              data_);
    return *this;
}
```

---

### Problem 1 — Self-Assignment Crash

```cpp
Buffer a(10);
a = a;   // self-assignment

// What happens inside operator=:
delete[] data_;          // deletes a's own memory
data_ = new int[other.size_];  // other IS a — other.size_ is now garbage!
// undefined behavior, likely crash
```

The fix everyone learns first:

```cpp
Buffer& operator=(const Buffer& other) {
    if (this == &other) return *this;   // guard — but this is a patch, not a solution
    ...
}
```

---

### Problem 2 — Exception Safety

```cpp
Buffer& operator=(const Buffer& other) {
    if (this == &other) return *this;
    delete[] data_;                      // old data is GONE
    data_ = new int[other.size_];        // ← what if this throws std::bad_alloc?
    // *this now has: data_ = dangling (deleted), size_ = old value
    // Object is CORRUPTED — destructor will double-delete ❌
    ...
}
```

The assignment either fully succeeds or leaves `*this` in a broken state. This is called the **basic guarantee** — no leaks, but object may be in an invalid state. What we want is the **strong guarantee** — either fully succeeds or `*this` is completely unchanged.

---

### Problem 3 — Code Duplication

The copy constructor and copy assignment do the same thing — allocate and copy. Writing it twice means two places to update when internals change.

---

### The Copy-and-Swap Solution

The idiom solves all three problems in one move:

```
1. Make a copy of the right-hand side (using the copy constructor you already wrote)
2. Swap the copy's guts with *this
3. Let the copy (now holding the old data) destroy itself at the end of the function
```

```cpp
Buffer& operator=(Buffer other) {   // ← 'other' is a COPY (by value!)
    swap(*this, other);             // swap internals
    return *this;
    // 'other' goes out of scope here → destructor frees the OLD data ✓
}
```

That's the whole idiom. Everything else is details.

## 🟡 Intermediate — Implementing It Correctly

### Step 1 — Write a Non-Throwing `swap`

The swap function is the heart of the idiom. It must be:
- A **free function** (enables ADL — Argument Dependent Lookup)
- **`noexcept`** — swapping pointers/ints never throws
- Declared as a `friend` so it can access private members

```cpp
class Buffer {
public:
    ...

    friend void swap(Buffer& a, Buffer& b) noexcept {
        using std::swap;          // allow std::swap as fallback
        swap(a.data_, b.data_);   // swap the pointer
        swap(a.size_, b.size_);   // swap the size
    }

private:
    int* data_;
    int  size_;
};
```

Rule: **swap each member individually** — never `std::swap(*this, other)` (infinite recursion).

---

### Step 2 — Write the Copy Constructor

The copy-and-swap assignment delegates to your copy constructor. Write it carefully:

```cpp
Buffer(const Buffer& other)
    : data_(new int[other.size_]), size_(other.size_) {
    std::copy(other.data_, other.data_ + size_, data_);
}
```

If the copy constructor throws (e.g. `new` fails), the assignment operator never even starts — `*this` is untouched. That's the strong guarantee in action.

---

### Step 3 — Write the Assignment Operator

```cpp
Buffer& operator=(Buffer other) {   // pass by VALUE — triggers copy constructor
    swap(*this, other);             // swap internals with the copy
    return *this;
}   // 'other' (now holding old data) is destroyed here
```

**Why pass by value?**

```
a = b
│
└── 'other' is constructed as a copy of b (copy constructor runs)
    If copy constructor throws → *this is UNTOUCHED ✓ (strong guarantee)
    If it succeeds → swap (noexcept) → old data destroyed in 'other' ✓
```

---

### Full Working Example

```cpp
#include <algorithm>   // std::copy
#include <utility>     // std::swap

class Buffer {
public:
    // Default constructor
    Buffer() : data_(nullptr), size_(0) {}

    // Parameterised constructor
    explicit Buffer(int size)
        : data_(new int[size]), size_(size) {
        std::fill(data_, data_ + size_, 0);
    }

    // Destructor
    ~Buffer() {
        delete[] data_;
    }

    // Copy constructor
    Buffer(const Buffer& other)
        : data_(other.size_ ? new int[other.size_] : nullptr),
          size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // swap — free function, noexcept
    friend void swap(Buffer& a, Buffer& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

    // Copy assignment — copy-and-swap
    Buffer& operator=(Buffer other) {   // by value!
        swap(*this, other);
        return *this;
    }

    int  size()        const { return size_; }
    int& operator[](int i)   { return data_[i]; }
    int  operator[](int i) const { return data_[i]; }

private:
    int* data_;
    int  size_;
};
```

Testing all the edge cases:

```cpp
Buffer a(5);
a[0] = 10;

Buffer b(3);
b = a;          // copy assignment — b gets its own copy of a's data ✓
a = a;          // self-assignment — works fine (copies then swaps with itself) ✓

Buffer c;
c = Buffer(8);  // temporary — move would be better here (see Advanced) ✓
```

---

### Why Self-Assignment Is Free

With copy-and-swap, `a = a` just works — no explicit guard needed:

```cpp
a = a;
// 1. 'other' is copy-constructed from a  ← full deep copy (wasteful but safe)
// 2. swap(*this, other)                  ← swaps a with its own copy
// 3. 'other' destroyed                   ← frees the copy
// Result: a is unchanged ✓
```

It's slightly wasteful (does a full copy for nothing) but correct. If self-assignment is frequent in your code, you can add the guard back as an optimisation:

```cpp
Buffer& operator=(Buffer other) {
    if (this == &other) return *this;  // optional optimisation
    swap(*this, other);
    return *this;
}
```

## 🔴 Advanced — Move Assignment, noexcept & Performance

### The Unified Assignment Operator

One of the elegant properties of copy-and-swap: **a single assignment operator handles both copy and move**:

```cpp
Buffer& operator=(Buffer other) {   // by value
    swap(*this, other);
    return *this;
}

Buffer a(100), b(200), c(300);

a = b;               // 'other' constructed via COPY constructor → deep copy
a = std::move(c);    // 'other' constructed via MOVE constructor → pointer steal
```

When the right-hand side is an rvalue (`std::move(c)` or a temporary), the compiler constructs `other` via the **move constructor** — which is just a pointer swap, O(1). No allocation needed.

This means: add a move constructor, and your existing assignment operator automatically becomes efficient for rvalues too.

---

### Adding Move Semantics

```cpp
class Buffer {
public:
    ...

    // Move constructor — steal the resource
    Buffer(Buffer&& other) noexcept
        : data_(nullptr), size_(0) {
        swap(*this, other);   // reuse swap!
    }

    // Unified assignment — handles copy AND move
    Buffer& operator=(Buffer other) noexcept {
        swap(*this, other);
        return *this;
    }

    ...
};
```

The move constructor is now just two lines — it default-initialises itself to an empty state, then swaps with `other`. Clean and reuses the swap you already wrote.

---

### `noexcept` Propagation

The assignment operator can be `noexcept` only if both the copy/move construction AND the swap are `noexcept`. In practice:

```cpp
// Move path: move ctor (noexcept) + swap (noexcept) → assignment is noexcept ✓
a = std::move(b);

// Copy path: copy ctor might throw (new can fail) → assignment is NOT noexcept
a = b;
```

To mark the assignment noexcept, you need to know your copy constructor can't throw — unusual for heap-allocating classes. The move path is always safe to mark noexcept:

```cpp
// If you want separate operators for copy and move:
Buffer& operator=(const Buffer& other) {    // copy — may throw
    Buffer temp(other);
    swap(*this, temp);
    return *this;
}

Buffer& operator=(Buffer&& other) noexcept {  // move — noexcept
    swap(*this, other);
    return *this;
}
```

---

### Performance: When Copy-and-Swap Has Extra Cost

The unified operator is elegant but always does a full construction of `other`:

```cpp
// With unified operator= (by value):
a = b;
// 1. Copy constructor for 'other' — allocates + copies    ← always happens
// 2. swap                                                  ← O(1)
// 3. destructor for 'other' (old data)                    ← always happens

// With separate copy= :
Buffer& operator=(const Buffer& other) {
    if (size_ == other.size_) {
        // Optimisation: reuse existing allocation if sizes match
        std::copy(other.data_, other.data_ + size_, data_);
        return *this;
    }
    Buffer temp(other);
    swap(*this, temp);
    return *this;
}
```

For large objects assigned frequently, the extra allocation + deallocation in the unified operator can matter. Profile before optimising, but know the tradeoff exists.

### Exception Safety Guarantees — Visualised

```
Copy-and-swap assignment:   a = b

BEFORE swap:
┌─────────────────────────────────────┐
│  Step 1: Buffer other(b)            │  ← copy constructor
│          If this throws:            │
│            *this is UNTOUCHED ✓     │  STRONG GUARANTEE
│            'other' is never created │
└─────────────────────────────────────┘
          ↓ success

```

## 🔵 Expert — Tradeoffs, Variations & When NOT to Use It

### Variation 1 — swap as a Member Function

Some codebases put `swap` as a member instead of a friend:

```cpp
class Buffer {
public:
    void swap(Buffer& other) noexcept {
        using std::swap;
        swap(data_, other.data_);
        swap(size_, other.size_);
    }
};

// Free function delegates to member (best of both worlds)
inline void swap(Buffer& a, Buffer& b) noexcept {
    a.swap(b);
}
```

The free function is still needed for ADL. Containers like `std::sort` and algorithms that call `swap` will find yours via ADL.

---

### Variation 2 — Copy-and-Swap Without the Idiom Name

Many codebases implement the strong guarantee manually without calling it "copy-and-swap":

```cpp
Buffer& operator=(const Buffer& other) {
    if (this == &other) return *this;

    // Perform all potentially-throwing operations FIRST
    int* newData = new int[other.size_];          // may throw — *this untouched
    std::copy(other.data_, other.data_ + other.size_, newData);

    // Now commit — noexcept from here
    delete[] data_;
    data_ = newData;
    size_ = other.size_;
    return *this;
}
```

This is equivalent in safety but more explicit about what can throw. Good when the copy is more complex than a single `new`.

---

### Interaction with Inheritance

Copy-and-swap does not work directly with inheritance because swapping base and derived parts requires care:

```cpp
class Animal {
    std::string name_;
public:
    friend void swap(Animal& a, Animal& b) noexcept {
        using std::swap;
        swap(a.name_, b.name_);
    }

    Animal& operator=(Animal other) {
        swap(*this, other);
        return *this;
    }
};

class Dog : public Animal {
    std::string breed_;
public:
    friend void swap(Dog& a, Dog& b) noexcept {
        using std::swap;
        swap(static_cast<Animal&>(a), static_cast<Animal&>(b));  // swap base part
        swap(a.breed_, b.breed_);                                  // swap derived part
    }

    Dog& operator=(Dog other) {
        swap(*this, other);
        return *this;
    }
};
```

Always cast to the base class when swapping — never rely on base's `operator=` inside the derived swap (it would slice).

---

### Interaction with `std::swap`

By providing your own `swap` as a free function in the same namespace, you ensure algorithms use the efficient version:

```cpp
namespace mylib {
    class Buffer { ... };
    void swap(Buffer& a, Buffer& b) noexcept { ... }  // your fast swap
}

std::vector<mylib::Buffer> vec;
std::sort(vec.begin(), vec.end(), cmp);
// std::sort internally calls swap → ADL finds mylib::swap → uses your fast version ✓
```

Without this, `std::swap` would use three copy/move operations instead of your O(1) pointer swap.

---

### When NOT to Use Copy-and-Swap

| Situation | Reason to avoid |
|-----------|----------------|
| **Performance-critical assignments** | Always allocates + deallocates even when sizes match |
| **Types where `new` never fails** | The strong guarantee buys nothing if allocation is infallible |
| **Move-only types** | No copy constructor → unified operator only works for moves, just write `= delete` for copy |
| **Trivially copyable types** | `memcpy`-able structs — the compiler already generates optimal code |
| **Self-assignment is very common** | Unified operator does a full copy on self-assign; add the guard |

---

### Copy-and-Swap with Standard Library Members (Rule of Zero)

If you follow the Rule of Zero and use `std::vector`, `std::string`, etc., the copy-and-swap idiom is still useful for the assignment operator even without raw pointers:

```cpp
class Config {
public:
    friend void swap(Config& a, Config& b) noexcept {
        using std::swap;
        swap(a.name_,   b.name_);    // string has noexcept swap
        swap(a.values_, b.values_);  // vector has noexcept swap
    }

    Config& operator=(Config other) {
        swap(*this, other);
        return *this;
    }

private:
    std::string       name_;
    std::vector<int>  values_;
};
```

`std::string::swap` and `std::vector::swap` are both `noexcept` in C++11+, so the whole chain is `noexcept`.

### The Complete Picture

```cpp
class Buffer {
public:
    Buffer() : data_(nullptr), size_(0) {}

    explicit Buffer(int size)
        : data_(new int[size]), size_(size) {
        std::fill(data_, data_ + size_, 0);
    }

    ~Buffer() { delete[] data_; }

    // Copy constructor (used by assignment)
    Buffer(const Buffer& other)
        : data_(other.size_ ? new int[other.size_] : nullptr),
          size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // Move constructor (used by assignment for rvalues)
    Buffer(Buffer&& other) noexcept
        : data_(nullptr), size_(0) {
        swap(*this, other);
    }

    // Unified assignment — handles copy + move via by-value parameter
    Buffer& operator=(Buffer other) noexcept {
        swap(*this, other);
        return *this;
    }

    // swap — the engine of the idiom
    friend void swap(Buffer& a, Buffer& b) noexcept {
        using std::swap;
        swap(a.data_, b.data_);
        swap(a.size_, b.size_);
    }

    int  size()        const { return size_; }
    int& operator[](int i)   { return data_[i]; }

private:
    int* data_;
    int  size_;
};
```

## Quick Reference

```
🟢 Beginner    → Naive assignment has 3 bugs: self-assign crash,
                  exception corruption, code duplication
                  Copy-and-swap fixes all three in one pattern

🟡 Intermediate → Write swap() as a noexcept friend free function
                  Pass by VALUE in operator= to trigger copy/move constructor
                  Old data is freed automatically by 'other's destructor

🔴 Advanced    → Add move constructor → unified operator handles move for free
                  Copy path may throw (new), move path is noexcept
                  Unified operator always allocates — know the cost

🔵 Expert      → Swap member + free function delegation for ADL
                  Derived classes must swap base part explicitly
                  Provide free swap so std algorithms use your O(1) version
                  Skip copy-and-swap for trivially copyable / move-only types
```
