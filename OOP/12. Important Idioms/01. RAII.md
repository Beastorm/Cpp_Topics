# RAII — Resource Acquisition Is Initialization

> **One sentence:** When an object is created, it grabs a resource. When the object dies, it automatically releases it. That's RAII.

## 🟢 Beginner — What Problem Does RAII Solve?

### The Problem: Forgetting to Clean Up

Imagine you open a file to read it. You **must** close it when you're done, or the OS keeps it locked forever.

```cpp
void readFile() {
    FILE* f = fopen("notes.txt", "r");

    // ... read stuff ...

    fclose(f);   // you MUST remember this
}
```

Looks fine. But what if your code grows?

```cpp
void readFile() {
    FILE* f = fopen("notes.txt", "r");

    if (fileIsEmpty(f)) {
        return;          // ❌ forgot fclose — file is now leaked!
    }

    if (someError()) {
        throw "oops";    // ❌ forgot fclose — file is now leaked!
    }

    fclose(f);           // only reached on the happy path
}
```

Every exit point needs its own `fclose`. Miss one → **resource leak**.

---

### The RAII Fix

Tie the file's lifetime to an **object's** lifetime. Objects on the stack are destroyed automatically when they go out of scope — no matter how you leave.

```cpp
#include <cstdio>

class File {
public:
    File(const char* path) {
        f_ = fopen(path, "r");   // 📥 acquire in constructor
    }

    ~File() {
        if (f_) fclose(f_);      // 📤 release in destructor — ALWAYS runs
    }

    FILE* get() { return f_; }

private:
    FILE* f_;
};
```

Now using it is safe:

```cpp
void readFile() {
    File f("notes.txt");         // opens file

    if (fileIsEmpty(f.get())) {
        return;                  // ✅ destructor closes file
    }

    if (someError()) {
        throw "oops";            // ✅ destructor closes file
    }

}   // ✅ destructor closes file
```

You **cannot** forget to close it. The language does it for you.

---

### The Golden Rule

```
Constructor  →  Acquire resource
Destructor   →  Release resource
```

| Resource | Acquire | Release |
|----------|---------|---------|
| File | `fopen` | `fclose` |
| Memory | `new` | `delete` |
| Mutex (lock) | `lock()` | `unlock()` |
| Network socket | `connect()` | `close()` |
| Database connection | `open()` | `close()` |

---

### Use the Standard Library — Don't Write It Yourself

For most common cases, C++ already has RAII wrappers:

```cpp
// Memory — use smart pointers, never raw new/delete
auto dog = std::make_unique<Dog>("Rex");   // allocates
// dog is freed automatically when it goes out of scope

// Files — use fstream
{
    std::ifstream file("notes.txt");       // opens
    // read...
}   // closes automatically

// Mutex — use lock_guard
{
    std::lock_guard<std::mutex> lock(mtx); // locks
    // modify shared data...
}   // unlocks automatically
```

## 🟡 Intermediate — Writing Your Own RAII Class

You'll need to write your own RAII wrapper when you're managing a resource the standard library doesn't cover — a GPU buffer, a C library handle, a custom OS resource, etc.

### Template Structure

```cpp
class MyResource {
public:
    // Constructor: acquire the resource
    explicit MyResource(/* params */) {
        handle_ = acquire_resource(/* params */);
        if (!handle_) {
            throw std::runtime_error("Failed to acquire resource");
        }
    }

    // Destructor: release the resource
    ~MyResource() {
        release_resource(handle_);
    }

    // Accessor
    ResourceHandle get() const { return handle_; }

    // Prevent accidental copying (see Advanced section)
    MyResource(const MyResource&) = delete;
    MyResource& operator=(const MyResource&) = delete;

private:
    ResourceHandle handle_;
};
```

---

### Real Example — OpenGL Buffer

```cpp
class GLBuffer {
public:
    GLBuffer() {
        glGenBuffers(1, &id_);          // ask GPU for a buffer
    }

    ~GLBuffer() {
        glDeleteBuffers(1, &id_);       // return it to GPU
    }

    GLuint id() const { return id_; }

    GLBuffer(const GLBuffer&) = delete;
    GLBuffer& operator=(const GLBuffer&) = delete;

private:
    GLuint id_;
};

void render() {
    GLBuffer vbo;                        // GPU buffer created
    glBindBuffer(GL_ARRAY_BUFFER, vbo.id());
    // upload data, draw...
}   // GPU buffer deleted — no glDeleteBuffers needed!
```

---

### Real Example — Mutex Lock

```cpp
class LockGuard {
public:
    explicit LockGuard(std::mutex& mtx) : mtx_(mtx) {
        mtx_.lock();     // acquire lock
    }

    ~LockGuard() {
        mtx_.unlock();   // release lock
    }

    LockGuard(const LockGuard&) = delete;
    LockGuard& operator=(const LockGuard&) = delete;

private:
    std::mutex& mtx_;
};

// This is exactly what std::lock_guard does!
```

---

### RAII in Class Members

RAII composes naturally. If your class has RAII members, you often don't need to write a destructor at all:

```cpp
class Server {
public:
    Server(const std::string& addr)
        : socket_(addr),              // socket acquired
          logFile_("server.log") {}  // file opened

    // No destructor needed!
    // When Server is destroyed, members are destroyed in reverse order:
    //   logFile_ closed first, then socket_ closed

private:
    Socket   socket_;
    FileLog  logFile_;
};
```

## 🔴 Advanced — Exception Safety, Rule of Five & Move Semantics

### Exception Safety

This is where RAII truly shines. When an exception is thrown, C++ **unwinds the stack** — it destroys all local objects in reverse order. RAII guarantees cleanup even in the face of exceptions.

```cpp
void process() {
    std::lock_guard<std::mutex> lock(mtx);   // locked here
    std::ifstream file("data.txt");          // file opened here

    riskyOperation();                        // throws std::exception!

    // We never reach here...
    // But that's fine — both lock and file are cleaned up
    // during stack unwinding ✅
}
```

Without RAII:
```cpp
void process() {
    mtx.lock();
    FILE* f = fopen("data.txt", "r");

    riskyOperation();   // throws!
    // ❌ mtx.unlock() never called → deadlock
    // ❌ fclose(f) never called   → file leak

    mtx.unlock();
    fclose(f);
}
```

---

### The Rule of Five

If your class manages a resource (owns memory, holds a handle, etc.), you must define — or explicitly delete — these five special functions:

```cpp
class Resource {
public:
    Resource(const char* path);                          // 1. Constructor

    ~Resource();                                         // 2. Destructor

    Resource(const Resource& other);                     // 3. Copy constructor
    Resource& operator=(const Resource& other);          // 4. Copy assignment

    Resource(Resource&& other) noexcept;                 // 5. Move constructor
    Resource& operator=(Resource&& other) noexcept;      // 6. Move assignment
};
```

> **Why?** If you define a destructor but forget the copy constructor, C++ generates a shallow copy — both objects point to the same resource, and the destructor runs twice → **double free / crash**.

---

### Copy vs Move

**Copy** = duplicate the resource (expensive)  
**Move** = transfer ownership (cheap, original becomes empty)

```cpp
class Buffer {
public:
    explicit Buffer(size_t size)
        : data_(new int[size]), size_(size) {}

    ~Buffer() { delete[] data_; }

    // ---- Copy: duplicate the data ----
    Buffer(const Buffer& other)
        : data_(new int[other.size_]), size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    Buffer& operator=(const Buffer& other) {
        if (this == &other) return *this;       // self-assignment guard
        delete[] data_;
        data_ = new int[other.size_];
        size_ = other.size_;
        std::copy(other.data_, other.data_ + size_, data_);
        return *this;
    }

    // ---- Move: steal the pointer ----
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;   // leave other in a valid empty state
        other.size_ = 0;
    }

    Buffer& operator=(Buffer&& other) noexcept {
        if (this == &other) return *this;
        delete[] data_;
        data_ = other.data_;
        size_ = other.size_;
        other.data_ = nullptr;
        other.size_ = 0;
        return *this;
    }

private:
    int*   data_;
    size_t size_;
};
```

Usage:
```cpp
Buffer a(1000);
Buffer b = a;               // copy — both have their own data
Buffer c = std::move(a);    // move — c owns the data, a is now empty
```

### The Rule of Zero (Modern Preference)

If you can use standard library members (`unique_ptr`, `vector`, `string`), you often don't need to write **any** of the five special functions:

```cpp
// ❌ Old style — manual resource management
class OldStyle {
    int* data_;
public:
    OldStyle() : data_(new int[100]) {}
    ~OldStyle() { delete[] data_; }
    OldStyle(const OldStyle&);
    OldStyle& operator=(const OldStyle&);
    OldStyle(OldStyle&&) noexcept;
    OldStyle& operator=(OldStyle&&) noexcept;
};

// ✅ Modern style — delegate to RAII types
class NewStyle {
    std::vector<int> data_;          // vector handles everything
public:
    NewStyle() : data_(100) {}
    // No destructor, no copy/move — compiler generates correct ones
};
```

> **Guideline:** Aim for the Rule of Zero. Only write the Rule of Five when wrapping a raw C API or OS handle.

## 🔵 Expert — Custom Deleters, Scope Guards & Real Patterns

### Custom Deleters with `unique_ptr`

`std::unique_ptr` accepts a custom deleter — perfect for wrapping C APIs:

```cpp
// Wrap FILE* using unique_ptr with custom deleter
auto fileDeleter = [](FILE* f) { if (f) fclose(f); };
using UniqueFile = std::unique_ptr<FILE, decltype(fileDeleter)>;

UniqueFile openFile(const char* path) {
    return UniqueFile(fopen(path, "r"), fileDeleter);
}

void process() {
    auto f = openFile("data.txt");
    // use f.get() ...
}   // fclose called automatically
```

### Scope Guard — RAII for Arbitrary Cleanup

Sometimes you need RAII cleanup for a one-off action, not a whole class. A **scope guard** runs a lambda when it goes out of scope:

```cpp
template<typename F>
class ScopeGuard {
public:
    explicit ScopeGuard(F fn) : fn_(std::move(fn)), active_(true) {}

    ~ScopeGuard() {
        if (active_) fn_();
    }

    void dismiss() { active_ = false; }   // cancel the cleanup

    ScopeGuard(const ScopeGuard&) = delete;
    ScopeGuard& operator=(const ScopeGuard&) = delete;

private:
    F    fn_;
    bool active_;
};

template<typename F>
ScopeGuard<F> makeScopeGuard(F fn) {
    return ScopeGuard<F>(std::move(fn));
}
```

Usage:
```cpp
void doWork() {
    beginTransaction(db);
    auto guard = makeScopeGuard([&]{ rollbackTransaction(db); });

    insertRow(db, data);    // might throw
    updateIndex(db);        // might throw

    commitTransaction(db);
    guard.dismiss();        // success — don't rollback
}
// If anything throws, rollback runs automatically
```

> C++23 adds `std::scope_exit`, `std::scope_fail`, and `std::scope_success` to the standard library.

### RAII and `shared_ptr` with Custom Deleters

When ownership is shared across multiple owners, use `shared_ptr`:

```cpp
auto makeSharedFile(const char* path) {
    return std::shared_ptr<FILE>(
        fopen(path, "r"),
        [](FILE* f) { if (f) fclose(f); }
    );
}

void example() {
    auto f1 = makeSharedFile("data.txt");
    auto f2 = f1;   // both own the file now

    // file is closed only when BOTH f1 and f2 go out of scope
}
```

### Exception Safety Guarantees

RAII is the foundation for reasoning about exception safety. There are three levels:

| Level | Guarantee | Meaning |
|-------|-----------|---------|
| **Basic** | No leaks | Resources are always released, but state may change |
| **Strong** | Commit-or-rollback | Operation either fully succeeds or has no effect |
| **No-throw** | Never throws | Operation always succeeds (`noexcept`) |

```cpp
// Strong guarantee using copy-and-swap
Buffer& Buffer::operator=(Buffer other) noexcept {  // other is a copy
    swap(*this, other);   // cheap pointer swap — can't throw
    return *this;
    // old data destroyed in 'other's destructor
}
```

### Common Pitfalls

```cpp
// ❌ Raw pointer in constructor argument — exception between allocations leaks
Foo(new Bar(), new Baz());

// ✅ Use make_unique — safe
Foo(std::make_unique<Bar>(), std::make_unique<Baz>());

// ❌ Holding raw pointer after a move
auto p = std::make_unique<int>(42);
int* raw = p.get();
auto q = std::move(p);   // p is now empty
*raw = 10;               // ❌ undefined behavior

// ❌ Forgetting noexcept on move — std::vector falls back to copy (slow)
MyClass(MyClass&&);          // bad
MyClass(MyClass&&) noexcept; // ✅ always mark moves noexcept if possible
```

## Standard Library Cheat Sheet

| Need | Use | Header |
|------|-----|--------|
| Sole ownership of heap object | `std::unique_ptr<T>` | `<memory>` |
| Shared ownership of heap object | `std::shared_ptr<T>` | `<memory>` |
| Non-owning reference to shared object | `std::weak_ptr<T>` | `<memory>` |
| Lock a mutex for a scope | `std::lock_guard<Mutex>` | `<mutex>` |
| Lock with early unlock | `std::unique_lock<Mutex>` | `<mutex>` |
| Lock multiple mutexes safely | `std::scoped_lock` | `<mutex>` |
| File I/O | `std::fstream` | `<fstream>` |
| Dynamic array | `std::vector<T>` | `<vector>` |
| Scope cleanup (C++23) | `std::scope_exit` | `<scope>` |
