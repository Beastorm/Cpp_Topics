# Virtual Constructor / Clone Idiom

> **One sentence:** Since C++ constructors cannot be virtual, use a virtual `clone()` method to copy — and a virtual `create()` method to default-construct — any object through a base class pointer, without knowing its concrete type.

## 🟢 Beginner — Why Constructors Can't Be Virtual

### The Problem: You Have a Base Pointer, You Need a Copy

In C++, if you only have a pointer or reference to the base class, you cannot copy the real derived object correctly:

```cpp
struct Animal {
    std::string name;
};

struct Dog : Animal {
    std::string breed;
};

struct Cat : Animal {
    int lives;
};

void duplicate(const Animal& a) {
    Animal copy = a;   // ❌ SLICING — copies only the Animal part!
                       //    breed and lives are silently thrown away
}

Dog d{"Rex", "Labrador"};
duplicate(d);   // d.breed is lost — you just made a bare Animal
```

This is called **object slicing** — the derived part is sliced off because the compiler only sees `Animal`.

---

### Why You Can't Just Make the Constructor Virtual

```cpp
struct Animal {
    virtual Animal(const Animal&);   // ❌ compile error — constructors cannot be virtual
};
```

Constructors run before the vtable pointer is set up — so the mechanism that makes virtual dispatch work doesn't exist yet during construction. C++ simply forbids virtual constructors.

---

### The Fix: Virtual `clone()`

The idiom wraps construction in a regular virtual function — which *can* be overridden in derived classes:

```cpp
struct Animal {
    virtual Animal* clone() const = 0;   // "virtual constructor" — copy me
    virtual ~Animal() = default;
};

struct Dog : Animal {
    std::string breed;
    Dog* clone() const override {        // returns a Dog*, not Animal*
        return new Dog(*this);           // calls Dog's copy constructor — no slicing ✓
    }
};

struct Cat : Animal {
    int lives;
    Cat* clone() const override {
        return new Cat(*this);
    }
};

void duplicate(const Animal& a) {
    Animal* copy = a.clone();    // calls Dog::clone() or Cat::clone() at runtime ✓
    // copy is the correct derived type — no slicing
    delete copy;
}
```

Now `duplicate()` works correctly regardless of the actual type:

```cpp
Dog d;  d.breed = "Labrador";
duplicate(d);   // clone() makes a real Dog copy ✓

Cat c;  c.lives = 9;
duplicate(c);   // clone() makes a real Cat copy ✓
```

---

### The Two Halves of the Idiom

| Virtual Method | What It Does | Equivalent To |
|----------------|-------------|---------------|
| `clone()`  | Copies an existing object | Copy constructor |
| `create()` | Creates a new default object | Default constructor |

```cpp
struct Animal {
    virtual Animal* clone()  const = 0;   // copy this object
    virtual Animal* create() const = 0;   // make a fresh default object of same type
    virtual ~Animal() = default;
};
```

`create()` is useful when you need a blank new object of the same type but don't want to copy existing state — for example, initialising a pool of objects from a prototype.

## 🟡 Intermediate — Implementing `clone()` and `create()`

### Complete Base + Derived Example

```cpp
#include <iostream>
#include <memory>
#include <string>

// ── Base class ───────────────────────────────────────────
struct Shape {
    std::string color;

    explicit Shape(const std::string& c) : color(c) {}
    virtual ~Shape() = default;

    virtual Shape* clone()  const = 0;   // copy this shape
    virtual Shape* create() const = 0;   // make a blank shape of the same type
    virtual void   draw()   const = 0;
    virtual double area()   const = 0;
};

// ── Derived: Circle ──────────────────────────────────────
struct Circle : Shape {
    double radius;

    Circle(const std::string& c, double r) : Shape(c), radius(r) {}

    Circle* clone()  const override { return new Circle(*this); }
    Circle* create() const override { return new Circle(color, 0.0); }
    void    draw()   const override {
        std::cout << "Circle [" << color << "] r=" << radius << "\n";
    }
    double area() const override { return 3.14159 * radius * radius; }
};

// ── Derived: Rectangle ───────────────────────────────────
struct Rectangle : Shape {
    double width, height;

    Rectangle(const std::string& c, double w, double h)
        : Shape(c), width(w), height(h) {}

    Rectangle* clone()  const override { return new Rectangle(*this); }
    Rectangle* create() const override { return new Rectangle(color, 0.0, 0.0); }
    void       draw()   const override {
        std::cout << "Rect [" << color << "] "
                  << width << "x" << height << "\n";
    }
    double area() const override { return width * height; }
};
```

Usage through base pointer — no knowledge of concrete type needed:

```cpp
Shape* original = new Circle("red", 5.0);
Shape* copy     = original->clone();    // makes a Circle, even through Shape*

original->draw();   // Circle [red] r=5
copy->draw();       // Circle [red] r=5

Shape* blank = original->create();     // makes a new Circle with radius=0
blank->draw();                          // Circle [red] r=0

delete original;
delete copy;
delete blank;
```

---

### The Rule: Every Derived Class Must Override Both

If a derived class forgets to override `clone()`, it silently inherits the base's (or grandparent's) version — which copies the wrong type. Make them pure virtual to force each class to implement:

```cpp
struct Shape {
    virtual Shape* clone()  const = 0;   // pure virtual — must override
    virtual Shape* create() const = 0;   // pure virtual — must override
};

// ✅ Catches the mistake at compile time:
struct Triangle : Shape {
    // forgot clone() and create()
};
// error: Triangle is abstract — clone() and create() not implemented ✓
```

---

### Using a Macro to Reduce Boilerplate

Every derived class has identical clone/create implementations. A macro keeps things DRY:

```cpp
#define DECLARE_CLONE(ClassName)                              \
    ClassName* clone()  const override {                      \
        return new ClassName(*this);                          \
    }                                                         \
    ClassName* create() const override {                      \
        return new ClassName();                               \
    }

struct Circle : Shape {
    DECLARE_CLONE(Circle)
    // ... rest of Circle
};

struct Rectangle : Shape {
    DECLARE_CLONE(Rectangle)
    // ... rest of Rectangle
};
```

Or a CRTP base to avoid macros entirely (see Advanced section).

---

### Copying a Heterogeneous Container

The most common real-world use — deep-copying a `vector` of polymorphic objects:

```cpp
// Deep copy a vector of shapes — each element keeps its real derived type
std::vector<Shape*> copyShapes(const std::vector<Shape*>& src) {
    std::vector<Shape*> result;
    result.reserve(src.size());
    for (const Shape* s : src) {
        result.push_back(s->clone());   // correct type every time ✓
    }
    return result;
}

std::vector<Shape*> shapes;
shapes.push_back(new Circle("red", 3.0));
shapes.push_back(new Rectangle("blue", 4.0, 5.0));
shapes.push_back(new Circle("green", 1.5));

auto copy = copyShapes(shapes);   // deep copy — all types preserved ✓
```

## 🔴 Advanced — Covariant Return Types, Smart Pointers & Deep Copy

### Covariant Return Types

C++ allows overriding functions to return a *more derived* pointer type — called a **covariant return type**. This is the standard C++ mechanism that lets `Dog::clone()` return `Dog*` while overriding `Animal* clone()`:

```cpp
struct Animal {
    virtual Animal* clone() const = 0;   // base returns Animal*
};

struct Dog : Animal {
    Dog* clone() const override {        // ✅ covariant — Dog* is more derived than Animal*
        return new Dog(*this);
    }
};
```

Why this matters:

```cpp
Dog d{"Rex", "Labrador"};

Animal* a = d.clone();   // returns Dog*, stored as Animal* — fine
Dog*    d2 = d.clone();  // ✅ no cast needed — compiler knows clone() returns Dog*
```

Without covariant return types, you'd need a cast every time you called `clone()` on a concrete type. Covariant returns make the interface both safe and ergonomic.

**Rules for covariant returns:**
- Overriding function's return type must be a pointer or reference to the overridden function's return type
- The derived type must be publicly and unambiguously derived from the base return type
- Does NOT work with smart pointers — `unique_ptr<Dog>` is not covariant with `unique_ptr<Animal>`

---

### Smart Pointer Version — Avoiding Raw `new`

Raw `new` in `clone()` is a memory management burden. Wrap in `unique_ptr`:

```cpp
struct Shape {
    virtual std::unique_ptr<Shape> clone()  const = 0;
    virtual std::unique_ptr<Shape> create() const = 0;
    virtual ~Shape() = default;
};

struct Circle : Shape {
    double radius;

    // ⚠️ Can't use covariant return with unique_ptr
    // Must return unique_ptr<Shape>, not unique_ptr<Circle>
    std::unique_ptr<Shape> clone() const override {
        return std::make_unique<Circle>(*this);
    }

    std::unique_ptr<Shape> create() const override {
        return std::make_unique<Circle>();
    }
};
```

Usage — no manual `delete` needed:

```cpp
auto original = std::make_unique<Circle>();
original->radius = 5.0;

auto copy = original->clone();   // unique_ptr<Shape> pointing to a Circle ✓
// both deleted automatically when they go out of scope
```

---

### Getting Covariant Returns WITH Smart Pointers

You can have both covariant-style return and smart pointers using a two-function pattern — a public non-virtual wrapper and a private virtual implementation:

```cpp
struct Shape {
    // Public interface — non-virtual, returns unique_ptr<Shape>
    std::unique_ptr<Shape> clone() const {
        return std::unique_ptr<Shape>(clone_impl());   // wraps raw ptr in unique_ptr
    }

    std::unique_ptr<Shape> create() const {
        return std::unique_ptr<Shape>(create_impl());
    }

    virtual ~Shape() = default;

protected:
    // Private virtual — returns raw pointer (covariant works here)
    virtual Shape* clone_impl()  const = 0;
    virtual Shape* create_impl() const = 0;
};

struct Circle : Shape {
    double radius;

protected:
    Circle* clone_impl()  const override { return new Circle(*this); }
    Circle* create_impl() const override { return new Circle(); }
};
```

This pattern gives you:
- `unique_ptr` lifetime management (public interface) ✓
- Covariant return types internally ✓
- No raw pointer exposure to callers ✓

---

### CRTP Base to Eliminate Boilerplate

Instead of writing identical `clone()` and `create()` in every derived class, use CRTP:

```cpp
// CRTP mixin — inject clone() and create() automatically
template<typename Derived, typename Base>
struct Cloneable : Base {
    std::unique_ptr<Base> clone() const override {
        return std::make_unique<Derived>(
            static_cast<const Derived&>(*this)   // copy-construct the Derived
        );
    }

    std::unique_ptr<Base> create() const override {
        return std::make_unique<Derived>();
    }
};

// Base class defines the interface
struct Shape {
    virtual std::unique_ptr<Shape> clone()  const = 0;
    virtual std::unique_ptr<Shape> create() const = 0;
    virtual void draw() const = 0;
    virtual ~Shape() = default;
};

// Derived classes inherit clone/create for free via CRTP
struct Circle : Cloneable<Circle, Shape> {
    double radius = 0;
    void draw() const override {
        std::cout << "Circle r=" << radius << "\n";
    }
};

struct Rectangle : Cloneable<Rectangle, Shape> {
    double w = 0, h = 0;
    void draw() const override {
        std::cout << "Rect " << w << "x" << h << "\n";
    }
};

// No clone() or create() written in Circle or Rectangle — CRTP injects them ✓
```

---

### Deep Copy vs Shallow Copy in `clone()`

`clone()` should always perform a **deep copy** — owned resources must be duplicated:

```cpp
struct Document : Cloneable<Document, Shape> {
    std::vector<std::string> pages;       // owned — must deep copy
    std::shared_ptr<Style>   style;       // shared — copy increases ref count
    std::string*             rawPtr;      // ❌ dangerous — must deep copy manually

    // Compiler-generated copy constructor handles:
    // vector → deep copy (vector copies its elements) ✓
    // shared_ptr → increments ref count ✓
    // raw pointer → SHALLOW copy — both Documents point to same string ❌

    // If you have raw pointer members, write a copy constructor:
    Document(const Document& other)
        : pages(other.pages),
          style(other.style),
          rawPtr(other.rawPtr ? new std::string(*other.rawPtr) : nullptr) {}
};
```

`clone()` uses the copy constructor — so if your copy constructor is correct, `clone()` is correct.

## 🔵 Expert — Prototype Registry, Type Erasure & Factory Patterns

### Prototype Registry

A registry maps type names (or IDs) to prototype objects. To create a new object of any registered type, you call `clone()` on its prototype — without knowing the concrete type:

```cpp
#include <unordered_map>
#include <functional>

class ShapeRegistry {
public:
    // Register a prototype under a name
    void registerShape(const std::string& name,
                       std::unique_ptr<Shape> prototype) {
        registry_[name] = std::move(prototype);
    }

    // Create a new instance of a registered shape by name
    std::unique_ptr<Shape> create(const std::string& name) const {
        auto it = registry_.find(name);
        if (it == registry_.end())
            throw std::invalid_argument("Unknown shape: " + name);
        return it->second->clone();   // clone the prototype ✓
    }

    // List all registered names
    std::vector<std::string> names() const {
        std::vector<std::string> result;
        for (auto& [k, v] : registry_) result.push_back(k);
        return result;
    }

private:
    std::unordered_map<std::string, std::unique_ptr<Shape>> registry_;
};

// Setup
ShapeRegistry reg;
reg.registerShape("circle",    std::make_unique<Circle>());
reg.registerShape("rectangle", std::make_unique<Rectangle>());

// Runtime creation — type not known at compile time
std::string userInput = "circle";   // could come from config, network, UI
auto shape = reg.create(userInput);
shape->draw();   // Circle r=0 ✓

// This is the Prototype Design Pattern
```

---

### Self-Registering Types

Remove manual registration by having each type register itself at startup:

```cpp
template<typename T>
struct AutoRegister {
    AutoRegister(const std::string& name) {
        ShapeRegistry::global().registerShape(
            name, std::make_unique<T>()
        );
    }
};

struct Triangle : Cloneable<Triangle, Shape> {
    void draw() const override { std::cout << "Triangle\n"; }

private:
    // Static member — constructed before main(), registers Triangle automatically
    static AutoRegister<Triangle> reg_;
};

// In triangle.cpp:
AutoRegister<Triangle> Triangle::reg_("triangle");

// Now anywhere in the program:
auto t = ShapeRegistry::global().create("triangle");   // works ✓
```

### Virtual Clone with Type Erasure (`std::any`)

When you don't want a base class at all — pure type erasure with clone:

```cpp
struct AnyClonable {
    template<typename T>
    AnyClonable(T value)
        : data_(std::make_shared<Model<T>>(std::move(value))) {}

    AnyClonable clone() const { return data_->clone(); }

    template<typename T>
    T& get() { return static_cast<Model<T>&>(*data_).value; }

private:
    struct Concept {
        virtual AnyClonable clone() const = 0;
        virtual ~Concept() = default;
    };

    template<typename T>
    struct Model : Concept {
        T value;
        explicit Model(T v) : value(std::move(v)) {}
        AnyClonable clone() const override {
            return AnyClonable(value);   // copy-constructs T
        }
    };

    std::shared_ptr<Concept> data_;
};

AnyClonable a = Circle{};         // stores a Circle
AnyClonable b = a.clone();        // clones the Circle — no base class needed ✓
AnyClonable c = std::string{"hi"};
AnyClonable d = c.clone();        // clones the string ✓
```

### Integration with Standard Algorithms

With a proper `clone()`, polymorphic containers work with standard algorithms:

```cpp
// Transform: clone and modify each shape
std::vector<std::unique_ptr<Shape>> scaled;
std::transform(
    shapes.begin(), shapes.end(),
    std::back_inserter(scaled),
    [](const std::unique_ptr<Shape>& s) {
        auto copy = s->clone();
        copy->scale(2.0);    // virtual method
        return copy;
    }
);

// Copy-construct a container preserving polymorphism
struct ShapeCollection {
    std::vector<std::unique_ptr<Shape>> items;

    ShapeCollection(const ShapeCollection& other) {
        items.reserve(other.items.size());
        for (const auto& s : other.items)
            items.push_back(s->clone());   // deep, polymorphic copy ✓
    }
};
```

### Common Pitfalls

```cpp
// ❌ Pitfall 1: Returning a Base* when the object is Derived — hidden slicing
struct Dog : Animal {
    Animal* clone() const override {
        return new Animal(*this);   // ❌ slices! Creates an Animal, not a Dog
    }
};
// Fix: return new Dog(*this)

// ❌ Pitfall 2: Forgetting override in a deep hierarchy
struct GuideDog : Dog {
    std::string handler;
    // forgot clone() — inherits Dog::clone() which makes a Dog, not GuideDog!
};
// Fix: always override in every concrete class, or use CRTP

// ❌ Pitfall 3: Shallow copy in clone()
struct Board : Cloneable<Board, Shape> {
    Tile* tiles;   // raw array — copy constructor copies the pointer, not the data!
    // Fix: write a proper copy constructor that deep-copies tiles

// ❌ Pitfall 4: clone() without a virtual destructor — UB on delete
struct Animal {
    virtual Animal* clone() const = 0;
    // forgot: virtual ~Animal() = default;
};
Animal* a = new Dog();
delete a;   // ❌ UB — calls Animal::~Animal, not Dog::~Dog

// ❌ Pitfall 5: unique_ptr covariance — doesn't work
struct Dog : Animal {
    std::unique_ptr<Dog> clone() const override { ... }
    // ❌ error: unique_ptr<Dog> is not covariant with unique_ptr<Animal>
    // Fix: use the two-function pattern (clone_impl returns raw ptr)
};
```

### Checklist for a Correct Clone Implementation

```
✅ Base class has a virtual destructor
✅ clone() is pure virtual in the base
✅ Every concrete class overrides clone() — not just the direct child
✅ clone() returns new Derived(*this) — uses the copy constructor
✅ Copy constructor performs a deep copy of owned resources
✅ Covariant return type used (Dog* instead of Animal*) for ergonomics
✅ If using smart pointers: use the two-function pattern for covariance
✅ If hierarchy is deep: use CRTP Cloneable<> to avoid repeating implementations
```

## Clone vs Alternatives

| Approach | Polymorphic copy | No slicing | No raw `new` | Works with containers |
|----------|:---------------:|:----------:|:------------:|:--------------------:|
| **Virtual `clone()`** | ✅ | ✅ | ⚠️ needs wrapper | ✅ |
| **`std::variant`** | ✅ | ✅ | ✅ | ✅ |
| **Type erasure** | ✅ | ✅ | ✅ | ✅ |
| **Just copy the derived** | ❌ (need type) | ✅ | ✅ | ❌ |
| **Plain copy through base** | ❌ | ❌ slices | ✅ | ❌ |

Use `std::variant` when the set of types is **closed and known at compile time**.
Use virtual `clone()` when the set of types is **open** (plugins, user-defined extensions, runtime registration).
