# PImpl — Pointer to Implementation

> **One sentence:** Hide the private details of a class behind a pointer so that changing internals never forces recompilation of the code that uses it.

Also called: **Cheshire Cat idiom**, **Compiler Firewall**, **d-pointer** (Qt convention).

## 🟢 Beginner — What Problem Does PImpl Solve?

### The Problem: Header Pollution & Slow Builds

In C++, the header file (`.h`) is a **contract** — everything that includes it must recompile whenever it changes.

```cpp
// engine.h  ← included by 50 other files
#include <string>
#include <vector>
#include <openssl/ssl.h>      // heavy external dependency
#include <internal_types.h>   // your private internal header

class Engine {
public:
    void start();
    void stop();

private:
    // ❌ private details exposed in the public header
    std::string        name_;
    std::vector<int>   cache_;
    SSL_CTX*           sslCtx_;     // now everyone must have OpenSSL installed
    InternalHelper     helper_;     // now everyone depends on internal_types.h
};
```

**Problems with this:**

| Problem | Effect |
|---------|--------|
| Private members sit in the public header | Users can see implementation details they shouldn't |
| Every `#include` in the header bleeds into user code | Adding `<openssl/ssl.h>` forces every user to have OpenSSL |
| Changing a private member recompiles all 50 files | A tiny refactor triggers a huge rebuild |
| `sizeof(Engine)` is exposed | Users can accidentally depend on the object size |

---

### The PImpl Fix

Move all private data into a **separate implementation struct** defined only in the `.cpp` file. The header holds just a pointer to it.

**Before:**
```
engine.h   →  has all private members  →  50 files include it
                                           all 50 recompile on any change
```

**After:**
```
engine.h   →  has only a pointer       →  50 files include it
engine.cpp →  has all private members     only engine.cpp recompiles on change
```

---

### The Simplest PImpl

**Header (`engine.h`) — what users see:**
```cpp
#pragma once
#include <memory>
#include <string>

class Engine {
public:
    explicit Engine(const std::string& name);
    ~Engine();

    void start();
    void stop();

private:
    struct Impl;                      // forward declaration only
    std::unique_ptr<Impl> pImpl_;     // pointer to hidden implementation
};
```

**Source (`engine.cpp`) — hidden from users:**
```cpp
#include "engine.h"
#include <openssl/ssl.h>       // ✅ only here, not in the header
#include "internal_types.h"    // ✅ only here, not in the header
#include <vector>
#include <iostream>

struct Engine::Impl {
    std::string        name;
    std::vector<int>   cache;
    SSL_CTX*           sslCtx = nullptr;
    InternalHelper     helper;
};

Engine::Engine(const std::string& name)
    : pImpl_(std::make_unique<Impl>()) {
    pImpl_->name = name;
}

Engine::~Engine() = default;   // must be here, not in the header

void Engine::start() {
    std::cout << "Starting " << pImpl_->name << "\n";
}

void Engine::stop() {
    std::cout << "Stopping " << pImpl_->name << "\n";
}
```

Now you can add fields to `Impl`, include heavy headers, or refactor internals — and **none of the 50 files that include `engine.h` need to recompile**.

---

### Why `~Engine() = default` Must Be in the `.cpp`

This trips up almost everyone writing PImpl for the first time.

```cpp
// ❌ If you write this in the header:
class Engine {
    ~Engine() = default;           // compiler generates destructor HERE
    std::unique_ptr<Impl> pImpl_;  // but Impl is incomplete at this point!
};
// Error: unique_ptr's deleter calls delete on an incomplete type
```

```cpp
// ✅ Declare in header, define in .cpp where Impl is complete:
// engine.h
~Engine();

// engine.cpp
Engine::~Engine() = default;   // Impl is fully defined here ✓
```

The same rule applies to move constructor and move assignment — **declare in header, `= default` in `.cpp`**.

## 🟡 Intermediate — Writing PImpl Correctly

### Full Boilerplate (Move-Only)

Most PImpl classes should be move-only (like `unique_ptr` itself). Here's the complete, correct pattern:

**`widget.h`**
```cpp
#pragma once
#include <memory>
#include <string>

class Widget {
public:
    explicit Widget(const std::string& name);
    ~Widget();

    // Move is fine — transfer pointer ownership
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;

    // No copy by default — see Advanced section if you need it
    Widget(const Widget&) = delete;
    Widget& operator=(const Widget&) = delete;

    void draw();
    int  width() const;

private:
    struct Impl;
    std::unique_ptr<Impl> pImpl_;
};
```

**`widget.cpp`**
```cpp
#include "widget.h"
#include <iostream>

struct Widget::Impl {
    std::string name;
    int         width = 100;
    int         height = 50;

    void draw() {
        std::cout << "Drawing " << name
                  << " [" << width << "x" << height << "]\n";
    }
};

Widget::Widget(const std::string& name)
    : pImpl_(std::make_unique<Impl>()) {
    pImpl_->name = name;
}

Widget::~Widget() = default;

Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;

void Widget::draw()      { pImpl_->draw(); }
int  Widget::width() const { return pImpl_->width; }
```

---

### Accessing Impl Cleanly

For classes with many methods, repeated `pImpl_->` gets verbose. A private helper keeps things tidy:

```cpp
// In widget.cpp — a typed helper to avoid repeating pImpl_->
Widget::Impl& Widget::impl() { return *pImpl_; }
const Widget::Impl& Widget::impl() const { return *pImpl_; }

// Now methods read cleanly
void Widget::draw()       { impl().draw(); }
int  Widget::width() const { return impl().width; }
```

---

### Null Check in Methods

`pImpl_` can be null after a move. Guard public methods defensively:

```cpp
void Widget::draw() {
    if (!pImpl_) throw std::logic_error("Widget was moved-from");
    pImpl_->draw();
}
```

Or use an assert in debug builds:

```cpp
void Widget::draw() {
    assert(pImpl_ && "Widget was moved-from");
    pImpl_->draw();
}
```

---

### PImpl in a Library

PImpl is especially valuable when writing a shared library (`.dll` / `.so`). It gives you a **stable ABI** — the binary interface doesn't change even when you add private fields, because `sizeof(MyClass)` only includes one pointer.

```
Without PImpl:  sizeof(Engine) = 8 + 24 + 8 + ... (depends on internals)
With PImpl:     sizeof(Engine) = 8  (always — just one pointer)
```

Adding a new private field in the Impl doesn't change the object size, so old binaries don't need relinking.

## 🔴 Advanced — Move Semantics, Copyable PImpl & Performance

### Making PImpl Copyable

By default, `unique_ptr` is move-only. If you want `Widget w2 = w1;` to work, you must deep-copy the Impl:

```cpp
// widget.h — add copy declarations
Widget(const Widget& other);
Widget& operator=(const Widget& other);
```

```cpp
// widget.cpp — deep copy the Impl
Widget::Widget(const Widget& other)
    : pImpl_(other.pImpl_
             ? std::make_unique<Impl>(*other.pImpl_)   // copy-construct the Impl
             : nullptr) {}

Widget& Widget::operator=(const Widget& other) {
    if (this == &other) return *this;
    if (!other.pImpl_) {
        pImpl_.reset();
    } else if (!pImpl_) {
        pImpl_ = std::make_unique<Impl>(*other.pImpl_);
    } else {
        *pImpl_ = *other.pImpl_;   // assign into existing Impl
    }
    return *this;
}
```

`Impl` must be copyable for this to work — ensure all its members have copy constructors.

---

### Move Semantics Are Free

Because PImpl stores a `unique_ptr`, moves are just pointer swaps — O(1) and `noexcept`:

```cpp
Widget a("alpha");
Widget b = std::move(a);   // just moves the pointer — no data copied
// a.pImpl_ is now null, b.pImpl_ points to the original Impl
```

This makes PImpl classes **ideal for use in `std::vector`** — reallocations move the pointer cheaply instead of copying all the data.

---

### Performance Considerations

PImpl has real costs. Know them:

| Cost | Notes |
|------|-------|
| **Extra heap allocation** | Every object does one extra `new` for the Impl |
| **Pointer indirection** | Every member access goes through a pointer (cache miss risk) |
| **No inline** | Methods in the header can't see Impl members, so nothing inlines |

**When the cost is acceptable:**
- Library / API boundary code where ABI stability matters
- Objects that live for a long time (allocation cost amortised)
- Objects that are moved but rarely copied

**When to avoid PImpl:**
- Hot inner loops or performance-critical code
- Small value types (`Point`, `Color`) — overhead is proportionally huge
- Header-only libraries — PImpl requires a `.cpp`

---

### `shared_ptr` vs `unique_ptr` for PImpl

| | `unique_ptr<Impl>` | `shared_ptr<Impl>` |
|--|-------|-------|
| Copy by default | ❌ No | ✅ Yes (shallow — shared state!) |
| Move | ✅ Fast (pointer swap) | ✅ Fast (pointer swap) |
| Overhead | None beyond pointer | Reference count + atomic ops |
| Use case | Standard PImpl | Shared / copy-on-write PImpl |

Use `shared_ptr` only if you intentionally want multiple objects to share the same Impl (rare):

```cpp
// Shallow copy — both widgets share the same Impl
class Widget {
    std::shared_ptr<Impl> pImpl_;
    ...
};
Widget w2 = w1;    // w1 and w2 now point to the same Impl
```

## 🔵 Expert — PImpl with Templates, Custom Deleters & Real Patterns

### The Problem: PImpl and Templates Don't Mix

Templates must be fully visible to the compiler at the point of instantiation — you can't hide them behind a PImpl.

```cpp
// ❌ This does NOT work
template<typename T>
class Box {
    struct Impl;
    std::unique_ptr<Impl> pImpl_;   // Impl is incomplete — template can't see it
};
```

**Workaround 1 — explicit instantiation:**
```cpp
// box.h
template<typename T>
class Box {
public:
    Box();
    ~Box();
    void set(T val);
    T    get() const;
private:
    struct Impl;
    std::unique_ptr<Impl> pImpl_;
};

// box.cpp
template<typename T>
struct Box<T>::Impl { T value; };

template<typename T> Box<T>::Box() : pImpl_(std::make_unique<Impl>()) {}
template<typename T> Box<T>::~Box() = default;
template<typename T> void Box<T>::set(T v) { pImpl_->value = v; }
template<typename T> T    Box<T>::get() const { return pImpl_->value; }

// Explicit instantiations — only these types work
template class Box<int>;
template class Box<std::string>;
```

**Workaround 2 — type-erase the Impl:**
```cpp
// Use void* + manual vtable for full type erasure (advanced)
struct ImplBase {
    virtual ~ImplBase() = default;
    virtual ImplBase* clone() const = 0;
};
```

### Custom Deleter for Non-Pointer Handles

Some C APIs use integer handles, not pointers. `unique_ptr` requires a pointer type. Use a custom deleter with a wrapper:

```cpp
// e.g. OpenGL texture handle (GLuint, not a pointer)
struct GLTextureDeleter {
    using pointer = GLuint;   // trick: tell unique_ptr the "pointer" type
    void operator()(GLuint id) const {
        glDeleteTextures(1, &id);
    }
};

using UniqueTexture = std::unique_ptr<GLuint, GLTextureDeleter>;

class Sprite {
public:
    Sprite(const char* path) {
        GLuint id;
        glGenTextures(1, &id);
        // load texture...
        pImpl_ = std::make_unique<Impl>(UniqueTexture(id));
    }
    ~Sprite() = default;

private:
    struct Impl {
        UniqueTexture texture;
        int width, height;
    };
    std::unique_ptr<Impl> pImpl_;
};
```

### Copy-on-Write PImpl (Lazy Copy)

Used when copies are common but mutations are rare. Share the Impl until a write is needed, then copy:

```cpp
class Document {
public:
    Document() : pImpl_(std::make_shared<Impl>()) {}

    // Read — just access shared Impl
    std::string title() const { return pImpl_->title; }

    // Write — detach before mutating
    void setTitle(const std::string& t) {
        detach();
        pImpl_->title = t;
    }

private:
    void detach() {
        if (pImpl_.use_count() > 1) {
            pImpl_ = std::make_shared<Impl>(*pImpl_);  // copy only when needed
        }
    }

    struct Impl {
        std::string title;
        std::string body;
    };
    std::shared_ptr<Impl> pImpl_;
};

Document a;
a.setTitle("Hello");
Document b = a;         // shares Impl — no copy yet
b.setTitle("World");    // detaches — now b has its own Impl
```

### Real-World Pattern — Qt's d-pointer

Qt uses PImpl pervasively under the name `d-pointer`. Their pattern adds a back-pointer (`q-pointer`) from Impl to the outer class:

```cpp
// widget_p.h — private header, not shipped to users
struct WidgetPrivate {
    explicit WidgetPrivate(Widget* q) : q_ptr(q) {}

    Widget* q_ptr;         // back-pointer to outer class
    int width  = 100;
    int height = 50;

    void internalDraw() {
        q_ptr->draw();     // can call public API via back-pointer
    }
};

// widget.h — public header
class Widget {
public:
    Widget();
    ~Widget();
    void draw();

protected:
    // Protected so subclasses can access d-pointer
    std::unique_ptr<WidgetPrivate> d_ptr;
};

// Convenience macros (Qt style)
#define Q_D(Class) Class##Private* const d = d_func()
#define Q_Q(Class) Class* const q = q_func()
```

---

### Checklist Before Shipping a PImpl Class

```
✅ Destructor defined in .cpp (not header)
✅ Move constructor defined in .cpp
✅ Move assignment defined in .cpp
✅ Copy constructor either deleted or deep-copies Impl
✅ Null check (or assert) in public methods after move
✅ Impl members initialized (use in-class initializers)
✅ No templates in Impl unless explicitly instantiated
✅ Private headers (like widget_p.h) not installed to users
```

## PImpl vs Alternatives

| Technique | Hides internals | Stable ABI | Runtime cost | Inheritance |
|-----------|:-:|:-:|:-:|:-:|
| **PImpl** | ✅ | ✅ | 1 heap alloc + indirection | Works |
| **Abstract base + factory** | ✅ | ✅ | vtable dispatch | Natural |
| **Modules (C++20)** | ✅ | ❌ (compiler-specific) | None | Works |
| **`#include` as-is** | ❌ | ❌ | None | Works |
| **`inline` namespace** | Partial | ❌ | None | Works |

- Use **PImpl** when you want ABI stability with a single concrete class.
- Use **abstract base + factory** when you want polymorphism and full interface hiding.
- Use **C++20 modules** when you control the full toolchain and don't need ABI stability guarantees.
