# Rule of Three / Five / Zero

> **One sentence:** If your class manually manages a resource, you must define the right set of special member functions — or bugs like double-free, memory leaks, and silent data corruption will follow.

## 🟢 Beginner — What Are Special Member Functions?

C++ automatically generates certain functions for every class. These are called **special member functions**:

```cpp
class Dog {
public:
    Dog();                              // 1. Default constructor
    ~Dog();                             // 2. Destructor
    Dog(const Dog& other);              // 3. Copy constructor
    Dog& operator=(const Dog& other);   // 4. Copy assignment operator
    Dog(Dog&& other) noexcept;          // 5. Move constructor       (C++11)
    Dog& operator=(Dog&& other) noexcept; // 6. Move assignment operator (C++11)
};
```

If you don't write them, the compiler writes them for you — doing a **member-by-member copy/move/destroy**. For simple classes that's perfect. For classes that own raw resources, it's a disaster.

---

### When the Compiler's Version Breaks Things

```cpp
class Buffer {
public:
    Buffer(int size) {
        data_ = new int[size];   // allocate on heap
        size_ = size;
    }

    ~Buffer() {
        delete[] data_;          // free on heap
    }

private:
    int* data_;
    int  size_;
};
```

Looks fine. But the compiler silently generates a copy constructor that does this:

```cpp
// What the compiler generates — a SHALLOW copy
Buffer(const Buffer& other) {
    data_ = other.data_;   // ❌ copies the POINTER, not the data
    size_ = other.size_;
}
```

Now two objects point to the **same memory**:

```cpp
Buffer a(10);
Buffer b = a;   // b.data_ == a.data_ — same pointer!

// When both go out of scope:
// ~Buffer() called on b → deletes data_  ✓
// ~Buffer() called on a → deletes data_ AGAIN  ❌ CRASH (double free)
```

This is the core problem all three rules solve.

## 🟡 Intermediate — Rule of Three

> **Rule of Three:** If you define any one of destructor, copy constructor, or copy assignment — define all three.

The logic: if you need a custom destructor, you're managing a resource. That means the compiler-generated copy will do a shallow copy of that resource — which is wrong. So you must also fix the copy constructor and copy assignment.

### The Three Functions

```
1. Destructor           →  release the resource
2. Copy constructor     →  deep copy the resource
3. Copy assignment      →  deep copy the resource (with self-assignment guard)
```

---

### Full Example — Fixed Buffer

```cpp
class Buffer {
public:
    // Constructor
    explicit Buffer(int size)
        : data_(new int[size]), size_(size) {
        std::fill(data_, data_ + size_, 0);
    }

    // 1. Destructor — release
    ~Buffer() {
        delete[] data_;
    }

    // 2. Copy constructor — deep copy
    Buffer(const Buffer& other)
        : data_(new int[other.size_]), size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // 3. Copy assignment — deep copy with self-assignment guard
    Buffer& operator=(const Buffer& other) {
        if (this == &other) return *this;       // guard: a = a

        delete[] data_;                         // free old resource
        data_ = new int[other.size_];           // allocate new
        size_ = other.size_;
        std::copy(other.data_, other.data_ + size_, data_);
        return *this;
    }

    int  size() const { return size_; }
    int& operator[](int i) { return data_[i]; }

private:
    int* data_;
    int  size_;
};
```

Now copies are safe:

```cpp
Buffer a(10);
a[0] = 42;

Buffer b = a;        // deep copy — b has its own memory
b[0] = 99;

// a[0] is still 42 ✅
// ~Buffer on b frees b's memory ✅
// ~Buffer on a frees a's memory ✅ — no double free
```

---

### The Self-Assignment Problem

Copy assignment must handle `a = a`:

```cpp
// ❌ Without the guard
Buffer& operator=(const Buffer& other) {
    delete[] data_;           // if other == *this, we just deleted our own data!
    data_ = new int[other.size_];  // other.size_ is now garbage
    ...
}

// ✅ With the guard
Buffer& operator=(const Buffer& other) {
    if (this == &other) return *this;   // detect self-assignment early
    ...
}
```

---

### Exception-Safe Assignment: Copy-and-Swap

The copy-and-swap idiom gives you strong exception safety for free:

```cpp
// Add a swap function
friend void swap(Buffer& a, Buffer& b) noexcept {
    std::swap(a.data_, b.data_);
    std::swap(a.size_, b.size_);
}

// Assignment takes by value (makes a copy), then swaps
Buffer& operator=(Buffer other) {   // 'other' is a copy
    swap(*this, other);             // steal other's data
    return *this;
    // old data destroyed in other's destructor
}
```

This is both exception-safe (the copy happens before any modification) and handles self-assignment automatically.

## 🔴 Advanced — Rule of Five (C++11)

C++11 added **move semantics** — a way to *transfer* a resource instead of copying it. This is cheap (just a pointer swap) vs copying (allocate + copy all data).

> **Rule of Five:** If you define any one of the five special functions, consider defining all five.

The two new functions are:

```
4. Move constructor      →  steal the resource from a temporary
5. Move assignment       →  steal the resource from a temporary
```

---

### Why Moves Matter

```cpp
Buffer makeBuffer(int size) {
    Buffer temp(size);
    // fill temp...
    return temp;   // without move: copies all data (expensive)
                   // with move:    transfers pointer (O(1))
}

Buffer b = makeBuffer(1000000);   // move constructor called — no data copied ✅
```

Without a move constructor, `std::vector<Buffer>` must **copy** every element when it reallocates — potentially copying megabytes of data.

---

### Full Rule of Five Example

```cpp
class Buffer {
public:
    explicit Buffer(int size)
        : data_(new int[size]), size_(size) {
        std::fill(data_, data_ + size_, 0);
    }

    // 1. Destructor
    ~Buffer() {
        delete[] data_;
    }

    // 2. Copy constructor
    Buffer(const Buffer& other)
        : data_(new int[other.size_]), size_(other.size_) {
        std::copy(other.data_, other.data_ + size_, data_);
    }

    // 3. Copy assignment
    Buffer& operator=(const Buffer& other) {
        if (this == &other) return *this;
        Buffer temp(other);        // make a copy
        swap(*this, temp);         // swap with copy (copy-and-swap)
        return *this;
    }

    // 4. Move constructor — steal the resource
    Buffer(Buffer&& other) noexcept
        : data_(other.data_), size_(other.size_) {
        other.data_ = nullptr;     // leave other in a valid empty state
        other.size_ = 0;
    }

    // 5. Move assignment — steal the resource
    Buffer& operator=(Buffer&& other) noexcept {
        if (this == &other) return *this;
        delete[] data_;            // free current resource
        data_ = other.data_;       // steal
        size_ = other.size_;
        other.data_ = nullptr;     // leave other valid but empty
        other.size_ = 0;
        return *this;
    }

    friend void swap(Buffer& a, Buffer& b) noexcept {
        std::swap(a.data_, b.data_);
        std::swap(a.size_, b.size_);
    }

    int  size() const { return size_; }
    int& operator[](int i) { return data_[i]; }

private:
    int* data_;
    int  size_;
};
```

Usage:

```cpp
Buffer a(100);
Buffer b = a;               // copy — b gets its own data
Buffer c = std::move(a);    // move — c steals a's pointer, a becomes empty

std::vector<Buffer> vec;
vec.push_back(Buffer(500)); // move constructor used — no copy ✅
```

---

### The Moved-From State

After a move, the source object must be in a **valid but unspecified state** — it must be safe to destroy and assign to, but you shouldn't read from it.

```cpp
Buffer a(100);
Buffer b = std::move(a);

// a.data_ is now nullptr, a.size_ is 0
// ✅ Safe to destroy a (destructor checks nullptr)
// ✅ Safe to assign: a = Buffer(50)
// ❌ Don't read a[0] — it's empty
```

---

### `noexcept` on Move Operations — Critical for Performance

```cpp
// ❌ Without noexcept — std::vector CANNOT use your move constructor
Buffer(Buffer&& other);

// ✅ With noexcept — std::vector WILL use your move constructor
Buffer(Buffer&& other) noexcept;
```

`std::vector` uses moves during reallocation **only if they are `noexcept`**. Otherwise it falls back to copies to preserve the strong exception guarantee. Always mark moves `noexcept` when they can't throw.

---

### Unified Move Assignment via Swap

```cpp
// Single assignment operator handles both copy and move:
Buffer& operator=(Buffer other) noexcept {  // by value: copies or moves
    swap(*this, other);
    return *this;
}

Buffer a(100), b(200), c(300);
a = b;               // calls copy constructor for 'other', then swaps
a = std::move(c);    // calls move constructor for 'other', then swaps
```

## 🔵 Expert — Rule of Zero & Modern Patterns

### Rule of Zero

> **Rule of Zero:** Design your class so it needs none of the five special functions. Let the compiler generate them all.

The trick: **use members that already implement RAII** (`unique_ptr`, `shared_ptr`, `vector`, `string`, etc.). Their special functions compose automatically.

```cpp
// ❌ Rule of Five — lots of boilerplate, easy to get wrong
class OldBuffer {
    int* data_;
    int  size_;
public:
    OldBuffer(int n) : data_(new int[n]), size_(n) {}
    ~OldBuffer()                          { delete[] data_; }
    OldBuffer(const OldBuffer& o)         : data_(new int[o.size_]), size_(o.size_)
                                          { std::copy(o.data_, o.data_+size_, data_); }
    OldBuffer& operator=(const OldBuffer&); // ... more boilerplate
    OldBuffer(OldBuffer&& o) noexcept     : data_(o.data_), size_(o.size_)
                                          { o.data_=nullptr; o.size_=0; }
    OldBuffer& operator=(OldBuffer&&) noexcept; // ... more boilerplate
};

// ✅ Rule of Zero — compiler generates everything correctly
class NewBuffer {
    std::vector<int> data_;   // vector handles copy, move, destroy
public:
    explicit NewBuffer(int n) : data_(n, 0) {}
    int  size() const          { return data_.size(); }
    int& operator[](int i)     { return data_[i]; }
    // No destructor, no copy, no move — all generated and correct ✅
};
```

---

### When to Use Which Rule

```
Does your class manage a raw resource (raw pointer, file handle, mutex, socket)?
│
├── YES → Can you replace it with a standard RAII type?
│         │
│         ├── YES → Do that. Apply Rule of Zero.
│         │         (use unique_ptr, vector, string, lock_guard, etc.)
│         │
│         └── NO  → You're writing a low-level resource wrapper.
│                   Apply Rule of Five.
│                   (e.g. wrapping a C API, a custom allocator, a GPU handle)
│
└── NO  → Apply Rule of Zero. Don't define any special functions.
```

---

### Explicitly Defaulting and Deleting

You can be explicit about what the compiler should generate — this is better than relying on implicit rules:

```cpp
class Config {
public:
    Config() = default;                           // explicitly generate default ctor
    ~Config() = default;                          // explicitly generate destructor

    Config(const Config&) = default;              // allow copy
    Config& operator=(const Config&) = default;   // allow copy assign

    Config(Config&&) = default;                   // allow move
    Config& operator=(Config&&) = default;        // allow move assign

private:
    std::string name_;
    std::vector<int> values_;
};
```

Or explicitly **disable** operations you don't want:

```cpp
class Singleton {
public:
    static Singleton& instance() {
        static Singleton s;
        return s;
    }

    Singleton(const Singleton&) = delete;             // no copy
    Singleton& operator=(const Singleton&) = delete;  // no copy assign
    Singleton(Singleton&&) = delete;                  // no move
    Singleton& operator=(Singleton&&) = delete;       // no move assign

private:
    Singleton() = default;
};
```

---

### Interaction Between the Rules — Compiler Generation Table

This is the most important table in this document. The compiler generates special functions based on what you declare:

| You declare | Default ctor | Destructor | Copy ctor | Copy assign | Move ctor | Move assign |
|---|:-:|:-:|:-:|:-:|:-:|:-:|
| Nothing | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Any constructor | ❌ | ✅ | ✅ | ✅ | ✅ | ✅ |
| Destructor only | ✅ | — | ✅ ⚠️ | ✅ ⚠️ | ❌ | ❌ |
| Copy constructor | ✅ | ✅ | — | ✅ ⚠️ | ❌ | ❌ |
| Copy assignment | ✅ | ✅ | ✅ ⚠️ | — | ❌ | ❌ |
| Move constructor | ✅ | ✅ | ❌ | ❌ | — | ❌ |
| Move assignment | ✅ | ✅ | ❌ | ❌ | ❌ | — |

**Legend:**
- ✅ = generated
- ❌ = NOT generated
- ⚠️ = generated but **deprecated** — likely wrong (shallow copy of a resource)

Key insight: **declaring a destructor suppresses move generation**. This is why:

```cpp
class MyClass {
    ~MyClass() { ... }   // you declared a destructor
    // move constructor:  NOT generated ← silent performance regression!
    // move assignment:   NOT generated ← silent performance regression!
};
```

If you declare a destructor, always explicitly `= default` the moves if you want them.

---

### Common Pitfalls

```cpp
// ❌ Pitfall 1: Declaring destructor silently kills moves
class Logger {
    ~Logger() { flush(); }     // moves are now deleted
    // fix: add Logger(Logger&&) noexcept = default;
};

// ❌ Pitfall 2: Forgetting self-assignment in copy operator
Buffer& operator=(const Buffer& other) {
    delete[] data_;            // if other == *this: deleted our own data!
    data_ = new int[other.size_]; // reading from freed memory → UB
}

// ❌ Pitfall 3: Move leaves source in invalid state
Buffer(Buffer&& other) noexcept : data_(other.data_), size_(other.size_) {
    // forgot: other.data_ = nullptr; other.size_ = 0;
    // destructor will delete the same memory twice → crash
}

// ❌ Pitfall 4: Move not marked noexcept — vector uses copy instead
Buffer(Buffer&& other);           // std::vector won't use this move!
Buffer(Buffer&& other) noexcept;  // ✅ vector will use this

// ❌ Pitfall 5: Slicing through copy
class Base {
    Base(const Base&) = default;  // copies only the Base part of a derived object
};
// Fix: if Base is polymorphic, delete copy or make it protected
```

---

### Polymorphic Base Classes

A special case: if your class is meant to be inherited and used via a base pointer, follow these rules:

```cpp
class Shape {
public:
    virtual ~Shape() = default;       // virtual destructor — REQUIRED for delete via base ptr

    Shape(const Shape&) = delete;              // prevent slicing
    Shape& operator=(const Shape&) = delete;   // prevent slicing

    virtual Shape* clone() const = 0;          // explicit deep copy instead
};

class Circle : public Shape {
public:
    Circle* clone() const override { return new Circle(*this); }
private:
    double radius_;
};
```

## The Three Rules at a Glance

```
Rule of Three (pre-C++11)
─────────────────────────
If you define:   destructor, copy ctor, or copy assignment
You must define: ALL THREE

Rule of Five (C++11+)
─────────────────────
If you define:   any of the three above, OR move ctor, OR move assignment
You must define: ALL FIVE

Rule of Zero (C++11+, preferred)
─────────────────────────────────
Design your class to need NONE of the five.
Use RAII member types (unique_ptr, vector, string) and let the compiler
generate everything correctly.
```

## Compiler Generation Cheat Sheet

```
Declare nothing                  →  all 6 generated ✅
Declare destructor               →  copy: generated (deprecated ⚠️), moves: deleted ❌
Declare copy constructor         →  copy assign: generated (deprecated ⚠️), moves: deleted ❌
Declare copy assignment          →  copy ctor: generated (deprecated ⚠️), moves: deleted ❌
Declare move constructor         →  copy: deleted ❌, move assign: NOT generated
Declare move assignment          →  copy: deleted ❌, move ctor: NOT generated
Declare both move ctor + assign  →  copy: deleted ❌
```

**Safe default for resource-managing classes:**

```cpp
class MyResource {
public:
    MyResource();

    ~MyResource();                                       // defined in .cpp
    MyResource(const MyResource&);                       // defined in .cpp
    MyResource& operator=(const MyResource&);            // defined in .cpp
    MyResource(MyResource&&) noexcept;                   // defined in .cpp
    MyResource& operator=(MyResource&&) noexcept;        // defined in .cpp
};
```

## Quick Reference

```
🟢 Beginner    → Compiler-generated copy does a shallow copy of pointers
                  That causes double-free and data corruption
                  If you have a destructor, you probably need all three

🟡 Intermediate → Rule of Three: destructor + copy ctor + copy assignment
                  Self-assignment guard in copy assignment
                  Copy-and-swap for exception-safe assignment

🔴 Advanced    → Rule of Five: add move ctor + move assignment
                  Move must leave source in a valid (empty) state
                  Always mark moves noexcept — critical for std::vector

🔵 Expert      → Rule of Zero: use RAII members, define nothing
                  Explicitly = default or = delete for clarity
                  Declaring destructor silently suppresses moves — watch out
                  Polymorphic bases need virtual destructor + deleted copy
```
