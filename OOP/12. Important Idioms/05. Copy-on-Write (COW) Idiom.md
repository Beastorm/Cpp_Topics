# Copy-on-Write (COW) Idiom

> **One sentence:** Share the same data between multiple objects until one of them needs to modify it — only then make a private copy.

Also called: **lazy copying**, **implicit sharing**.

## 🟢 Beginner — What Problem Does COW Solve?

### The Expensive Copy Problem

Imagine a `String` class holding a large buffer. Every time you copy it, you pay for a full heap allocation and `memcpy`:

```cpp
String a = loadHugeDocument();   // 10 MB in memory
String b = a;                    // ← copies 10 MB — expensive!
String c = a;                    // ← copies 10 MB again — expensive!

// But if b and c never modify the string,
// why did we pay to copy the data three times?
```

This is wasteful when copies are **read-only most of the time**.

---

### The COW Fix: Share Until You Write

Instead of copying immediately, share the same underlying data. Only copy when someone actually *writes*:

```cpp
String a = loadHugeDocument();   // 10 MB allocated once
String b = a;                    // ← NO copy — b shares a's buffer
String c = a;                    // ← NO copy — c shares a's buffer

b.append("!");                   // ← NOW b gets its own copy (write triggered)
// a and c still share the original buffer
```

```
After   a = loadHugeDocument():
  a ──→ [ "Hello World..." 10MB ]

After   b = a  and  c = a:
  a ──┐
  b ──┼──→ [ "Hello World..." 10MB ]   (shared, ref count = 3)
  c ──┘

After   b.append("!"):
  a ──┐
  c ──┴──→ [ "Hello World..." 10MB ]   (shared, ref count = 2)
  b   ──→  [ "Hello World...!" 10MB ]  (b's own copy)
```

---

### When COW Is Worth It

COW is a good fit when:

| Situation | Why COW Helps |
|-----------|--------------|
| Objects are **large** (strings, images, documents) | Avoids expensive copies when reads dominate |
| Copies are **passed around but rarely modified** | Most copies stay shared — zero allocation cost |
| You want **value semantics** with low copy cost | Callers write `obj2 = obj1` naturally, no `clone()` calls |

COW is a **bad fit** when:

| Situation | Why COW Hurts |
|-----------|--------------|
| Objects are **small** (a few ints) | Overhead of `shared_ptr` exceeds savings |
| Writes are **frequent** | Every write triggers a copy — worse than copying upfront |
| Code is **heavily multithreaded** | Detach + modify is not atomic — needs careful synchronisation |

## 🟡 Intermediate — Implementing COW with `shared_ptr`

### The Core Mechanism: `detach()`

The entire COW idiom revolves around one private helper — `detach()`:

```
detach() checks:
  Is anyone else sharing this data?  (use_count > 1)
    YES → make my own copy, point to that instead
    NO  → I'm the only owner, modify in place
```

```cpp
void detach() {
    if (data_.use_count() > 1) {
        data_ = std::make_shared<Data>(*data_);   // deep copy
    }
    // now data_.use_count() == 1 — safe to modify
}
```

Call `detach()` at the start of every **mutating** method. Never call it in `const` methods.

---

### Full Working Example — `CowString`

```cpp
#include <memory>
#include <string>
#include <iostream>

class CowString {
public:
    // Constructors
    CowString() : data_(std::make_shared<std::string>()) {}

    explicit CowString(const std::string& s)
        : data_(std::make_shared<std::string>(s)) {}

    // Copy constructor — share, don't copy
    CowString(const CowString& other) = default;            // shared_ptr copy = shared

    // Copy assignment — share, don't copy
    CowString& operator=(const CowString& other) = default; // shared_ptr assign = shared

    // Move — transfer ownership
    CowString(CowString&&) noexcept = default;
    CowString& operator=(CowString&&) noexcept = default;

    // ── Read (const) methods — no detach needed ──

    const std::string& str() const { return *data_; }
    std::size_t size()        const { return data_->size(); }
    char operator[](int i)    const { return (*data_)[i]; }   // read-only

    // ── Write (mutating) methods — detach first ──

    void append(const std::string& s) {
        detach();               // ensure we own the data
        data_->append(s);       // now safe to modify
    }

    void clear() {
        detach();
        data_->clear();
    }

    char& operator[](int i) {
        detach();               // even a single char write triggers detach
        return (*data_)[i];
    }

    // How many objects share this buffer right now
    long use_count() const { return data_.use_count(); }

private:
    std::shared_ptr<std::string> data_;

    void detach() {
        if (data_.use_count() > 1) {
            data_ = std::make_shared<std::string>(*data_);
        }
    }
};
```

Usage:

```cpp
CowString a("Hello, World!");
CowString b = a;               // shares buffer — no allocation
CowString c = a;               // shares buffer — no allocation

std::cout << a.use_count();    // 3 — all sharing

b.append("!!!");               // b detaches — gets its own copy
std::cout << a.use_count();    // 2 — a and c still share
std::cout << b.use_count();    // 1 — b is alone

c.clear();                     // c detaches
std::cout << a.use_count();    // 1 — a is alone now
```

### The `Data` Struct Pattern (for larger classes)

When the object has multiple fields, bundle them all into a nested `Data` struct so the `shared_ptr` covers everything:

```cpp
class Document {
public:
    Document() : d_(std::make_shared<Data>()) {}

    // Reads
    const std::string& title() const { return d_->title; }
    const std::string& body()  const { return d_->body; }
    int                pages() const { return d_->pages; }

    // Writes
    void setTitle(const std::string& t) { detach(); d_->title = t; }
    void setBody (const std::string& b) { detach(); d_->body  = b; }

private:
    struct Data {
        std::string title;
        std::string body;
        int         pages = 0;
    };

    std::shared_ptr<Data> d_;

    void detach() {
        if (d_.use_count() > 1)
            d_ = std::make_shared<Data>(*d_);   // copies entire Data struct
    }
};
```

This is exactly the pattern Qt uses internally (`QSharedDataPointer`).

## 🔴 Advanced — Thread Safety, `detach()` & const-Correctness

### Thread Safety: What `shared_ptr` Gives You (and What It Doesn't)

`shared_ptr`'s **reference count** is atomic — incrementing/decrementing it from multiple threads is safe.

**But the data it points to is NOT protected:**

```cpp
// Thread 1:                    Thread 2:
CowString b = a;                CowString c = a;
b.append("X");                  c.append("Y");
```

Both threads call `detach()`, which checks `use_count() > 1` and copies. But between the check and the modification, another thread may have already detached — this is a **TOCTOU race** (time-of-check to time-of-use):

```
Thread 1: use_count() == 2 → will detach
Thread 2: use_count() == 2 → will detach
Thread 1: makes copy, modifies  ← fine
Thread 2: makes copy, modifies  ← fine (both detached independently)
```

Actually this specific case is safe because each thread ends up with its own copy. The dangerous case is:

```cpp
// Thread 1 reads:              Thread 2 writes:
const char* p = a.str().c_str();  a.setTitle("new");
// Thread 2's detach() replaces data_ — but Thread 1 still holds a raw pointer
// to the OLD string's internal buffer — which is now dangling if no other
// shared_ptr keeps it alive ❌
```

**Rule:** Never hold raw pointers/references into COW data across thread boundaries. Always hold a `CowString` copy (which increments the ref count and keeps data alive).

---

### Thread-Safe COW: Per-Object Mutex

For truly thread-safe COW, protect `detach()` with a mutex:

```cpp
class ThreadSafeCowString {
public:
    void append(const std::string& s) {
        std::lock_guard<std::mutex> lock(mutex_);
        detach();
        data_->append(s);
    }

    std::string read() const {
        std::lock_guard<std::mutex> lock(mutex_);
        return *data_;   // return a copy while holding the lock
    }

private:
    mutable std::mutex           mutex_;
    std::shared_ptr<std::string> data_;

    void detach() {   // call while holding mutex_
        if (data_.use_count() > 1)
            data_ = std::make_shared<std::string>(*data_);
    }
};
```

For high-read / low-write workloads, prefer `std::shared_mutex` (C++17):

```cpp
mutable std::shared_mutex rwmutex_;

std::string read() const {
    std::shared_lock lock(rwmutex_);   // many readers at once
    return *data_;
}

void append(const std::string& s) {
    std::unique_lock lock(rwmutex_);   // exclusive for writes
    detach();
    data_->append(s);
}
```

### `const`-Correctness and the `mutable` Trap

`detach()` must never be called from `const` methods. But what about methods that *logically* don't modify content — like `operator[]` for reading?

```cpp
// ❌ Wrong — non-const operator[] always detaches, even for reads
char& operator[](int i) {
    detach();          // copies even when caller only reads!
    return (*data_)[i];
}

// ✅ Correct — provide both const and non-const overloads
char operator[](int i) const {    // read: no detach
    return (*data_)[i];
}

char& operator[](int i) {         // write: detach
    detach();
    return (*data_)[i];
}
```

Always provide **both** overloads for any accessor that could be used for reading or writing. The `const` version is called on `const` objects and inside `const` methods — never detaches.

---

### Lazy Detach vs Eager Detach

**Lazy (standard COW):** detach only when a mutating method is called.

```cpp
void append(const std::string& s) {
    detach();          // detach here, just before modifying
    data_->append(s);
}
```

**Eager detach:** detach immediately on assignment if you anticipate writes:

```cpp
// Caller explicitly requests a private copy
CowString writeable = a.detached();   // public detach method

CowString CowString::detached() const {
    CowString copy(*this);
    copy.detach_();    // force detach now
    return copy;
}
```

Eager detach avoids the `detach()` overhead inside the hot write path — useful if writes always follow immediately after copy.

## 🔵 Expert — COW in the Wild, Pitfalls & Modern Alternatives

### COW in Qt: `QSharedData` + `QSharedDataPointer`

Qt implements COW for almost all of its value types (`QString`, `QByteArray`, `QImage`, `QList`, etc.) using a pair of helper classes:

```cpp
// Your private data inherits QSharedData (which holds the ref count)
class EmployeeData : public QSharedData {
public:
    QString name;
    int     salary = 0;
};

// Your public class holds a QSharedDataPointer
class Employee {
public:
    Employee() : d(new EmployeeData) {}

    // Reads — QSharedDataPointer provides operator-> for const access
    QString name()   const { return d->name; }
    int     salary() const { return d->salary; }

    // Writes — QSharedDataPointer::detach() is called automatically
    // when you use non-const operator->
    void setName  (const QString& n) { d->name   = n; }   // auto-detaches!
    void setSalary(int s)            { d->salary = s; }    // auto-detaches!

private:
    QSharedDataPointer<EmployeeData> d;
};
```

`QSharedDataPointer` calls `detach()` automatically whenever you use the non-const `operator->`. You never call `detach()` manually.

---

### Implementing Auto-Detach with `operator->`

You can replicate Qt's magic with a custom smart pointer:

```cpp
template<typename T>
class CowPtr {
public:
    explicit CowPtr(T* p) : ptr_(p) {}
    CowPtr(const CowPtr&) = default;
    CowPtr& operator=(const CowPtr&) = default;

    // const access — no detach
    const T* operator->() const { return ptr_.get(); }
    const T& operator* () const { return *ptr_; }

    // non-const access — auto-detach
    T* operator->() {
        detach();
        return ptr_.get();
    }
    T& operator*() {
        detach();
        return *ptr_;
    }

    long use_count() const { return ptr_.use_count(); }

private:
    std::shared_ptr<T> ptr_;

    void detach() {
        if (ptr_.use_count() > 1)
            ptr_ = std::make_shared<T>(*ptr_);
    }
};
```

Usage — `detach()` is fully transparent to callers:

```cpp
struct Config { std::string host; int port = 8080; };

CowPtr<Config> a(new Config{"localhost", 80});
CowPtr<Config> b = a;              // shared

b->port = 9090;                    // non-const → auto-detaches b
std::cout << a->port;              // 80 — a unchanged ✓
std::cout << b->port;              // 9090 ✓
```

### The Iterator Invalidation Problem

COW breaks iterator safety in a subtle way:

```cpp
CowString a("Hello");
auto it = a.str().begin();    // iterator into a's buffer

CowString b = a;              // shares buffer
b[0] = 'J';                   // b detaches — b gets new buffer
                               // a's buffer is unchanged ✓

// But if YOU hold an iterator and then call a mutating method on the SAME object:
CowString c("World");
const char* p = c.str().c_str();   // raw pointer into buffer
c.append("!");                     // detaches — NEW buffer allocated
// p now points to the OLD buffer — which may or may not still be alive
// (alive if nothing else owns it... but don't rely on this) ❌
```

**Rule:** Never hold raw iterators or pointers into a COW object's data across any mutating call on that object.

### Common Pitfalls

```cpp
// ❌ Pitfall 1: Forgetting to detach before EVERY write path
void insertAt(int pos, char c) {
    // forgot detach() — modifies shared buffer, corrupts all copies!
    data_->insert(data_->begin() + pos, c);
}

// ❌ Pitfall 2: Returning a non-const reference leaks write access
std::string& mutableStr() {
    // No detach — caller can modify shared buffer silently!
    return *data_;
}
// Fix: either detach here, or only expose const ref / value

// ❌ Pitfall 3: use_count() check is not thread-safe for mutation decision
if (data_.use_count() == 1) {
    data_->modify();   // another thread may have copied between check and modify!
}
// Fix: hold a mutex around the check + modify

// ❌ Pitfall 4: COW on small objects — overhead exceeds benefit
class Point {             // only 8 bytes
    CowPtr<Data> d_;      // shared_ptr = 16 bytes + heap alloc — way worse!
};

// ❌ Pitfall 5: Detaching inside const method via mutable member
mutable std::shared_ptr<Data> data_;   // tempting for "lazy init"
// Makes every "read" potentially trigger a copy — defeats const contract
```

### COW Was Removed from `std::string` (C++11)

Before C++11, many `std::string` implementations used COW internally. C++11 **forbids** COW for `std::string` because it breaks the iterator invalidation rules and creates subtle multithreading hazards:

```cpp
std::string a = "hello";
std::string b = a;          // pre-C++11: may share buffer
char& r = b[0];             // pre-C++11: may or may not detach here
a[0] = 'H';                 // pre-C++11: undefined — did b detach or not?
```

C++11 mandates that `operator[]` on a non-const string always gives you exclusive write access — no implicit sharing allowed.

**Lesson:** COW is a valid design choice, but it must be **explicit** — not hidden inside an operator that looks like a simple array access.

---

### Modern Alternative: Immutable Data + Structural Sharing

For large, frequently-shared data structures, consider **immutable data** instead of COW:

```cpp
// Immutable approach — no need for detach() at all
class ImmutableString {
public:
    explicit ImmutableString(std::string s)
        : data_(std::make_shared<const std::string>(std::move(s))) {}

    // "modify" returns a NEW object — original untouched
    ImmutableString append(const std::string& s) const {
        return ImmutableString(*data_ + s);   // new allocation, new object
    }

    const std::string& str() const { return *data_; }

private:
    std::shared_ptr<const std::string> data_;
};

ImmutableString a("Hello");
ImmutableString b = a.append(", World!");   // a is unchanged, b is new
// a and b can be shared across threads freely — data is const ✓
```

Immutable + structural sharing is the approach used by functional languages (Clojure, Scala) and by React's state model. No detach, no race conditions, but every write allocates.

## COW vs Alternatives

| Approach | Copy cost | Write cost | Thread safe? | Complexity |
|----------|:---------:|:----------:|:------------:|:----------:|
| **Deep copy always** | High | Low | ✅ Yes | Low |
| **COW (this doc)** | Low (shared) | Medium (detach) | ⚠️ Careful | Medium |
| **Move semantics** | Low (O(1)) | Low | ✅ Yes | Low |
| **Immutable + new object** | Low (shared) | High (alloc) | ✅ Yes | Medium |
| **Shared mutable + mutex** | Low (shared) | Low + lock | ✅ Yes | High |

**Prefer move semantics** when the source won't be used again.  
**Prefer COW** when value semantics with low copy cost matter and writes are infrequent.  
**Prefer immutable** in highly concurrent, functional, or reactive designs.
