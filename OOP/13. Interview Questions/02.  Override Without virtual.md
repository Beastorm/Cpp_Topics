# What Happens When You Override Without `virtual` in C++?

> **One sentence:** Without `virtual`, the function is not overridden — it is **hidden**. Calls through a base pointer silently call the wrong version, with zero compiler warnings by default.

## 🟢 Beginner — Virtual vs Name Hiding

### What You Think Happens

```cpp
struct Animal {
    void speak() { std::cout << "...\n"; }
};

struct Dog : Animal {
    void speak() { std::cout << "Woof!\n"; }   // looks like an override
};

Animal* a = new Dog;
a->speak();   // you expect: "Woof!"
              // you get:    "..."    ← silent bug
```

No compiler error. No warning (by default). Just the wrong function called.

---

### Why It Happens: Static vs Dynamic Dispatch

Without `virtual`, the compiler resolves function calls at **compile time** based on the static type of the pointer — not the actual runtime type of the object.

```
With virtual:
  a->speak()  →  look up vtable at runtime  →  calls Dog::speak()   ✓

Without virtual:
  a->speak()  →  resolved at compile time   →  calls Animal::speak() ✗
                  ("a is Animal* so call Animal::speak")
```

The Dog object is there — but the compiler never asks it which `speak()` to use. It already decided.

---

### The Fix: Add `virtual`

```cpp
struct Animal {
    virtual void speak() { std::cout << "...\n"; }   // ← virtual
};

struct Dog : Animal {
    void speak() override { std::cout << "Woof!\n"; }  // ← override
};

Animal* a = new Dog;
a->speak();   // "Woof!" ✓ — correct, runtime dispatch via vtable
```

Two keywords, two purposes:

| Keyword | Where | What It Does |
|---------|-------|-------------|
| `virtual` | Base class | Enables runtime dispatch — vtable entry created |
| `override` | Derived class | Verifies you are actually overriding something — compile error if not |

## 🟡 Intermediate — The Three Behaviours

### Case 1: With `virtual` — Correct Polymorphism

```cpp
struct Base {
    virtual void f() { std::cout << "Base::f\n"; }
    virtual ~Base() = default;
};

struct Derived : Base {
    void f() override { std::cout << "Derived::f\n"; }
};

Base* p = new Derived;
p->f();      // "Derived::f" ✓ — vtable lookup, correct

Base& r = *p;
r.f();       // "Derived::f" ✓ — references also dispatch virtually

delete p;    // safe — virtual destructor ✓
```

---

### Case 2: Without `virtual` — Name Hiding

```cpp
struct Base {
    void f() { std::cout << "Base::f\n"; }   // no virtual
};

struct Derived : Base {
    void f() { std::cout << "Derived::f\n"; }  // hides Base::f, doesn't override
};

Derived d;
d.f();           // "Derived::f" ✓ — direct call, no ambiguity

Base* p = &d;
p->f();          // "Base::f"    ✗ — compile-time dispatch, silent wrong call

Base& r = d;
r.f();           // "Base::f"    ✗ — same problem through reference
```

`Derived::f` completely hides `Base::f` inside the `Derived` scope — but the base version is still reachable through a base pointer or reference.

---

### Case 3: Accessing the Hidden Base Function

The hidden function is not gone — you can still call it explicitly:

```cpp
Derived d;
d.f();           // Derived::f  (hides base)
d.Base::f();     // Base::f     (explicit scope resolution)

Base* p = &d;
p->f();          // Base::f     (static dispatch through base ptr)
```

---

### The `override` Keyword Turns Silent Bugs Into Compile Errors

```cpp
struct Base {
    virtual void process(int x);
};

struct Derived : Base {
    // ❌ Typo — different signature, silently becomes name hiding
    void process(double x) { ... }

    // ✅ override catches this at compile time
    void process(double x) override { ... }
    // error: 'process(double)' marked override but does not override
};
```

Common mistakes `override` catches:

```cpp
struct Base {
    virtual void update(int x);
    virtual void render() const;
    virtual void reset();
};

struct Derived : Base {
    void update(float x) override;   // ❌ wrong param type — caught
    void render() override;          // ❌ missing const — caught
    void Reset() override;           // ❌ wrong capitalisation — caught
};
```

**Always write `override`.** It costs nothing and prevents an entire class of bugs.

---

### Name Hiding Across Overload Sets

Name hiding applies to the entire overload set, not just one signature:

```cpp
struct Base {
    void f(int x)    { std::cout << "int\n"; }
    void f(double x) { std::cout << "double\n"; }
    void f()         { std::cout << "void\n"; }
};

struct Derived : Base {
    void f(int x) { std::cout << "Derived int\n"; }
    // Derived::f(int) hides ALL three Base::f overloads
};

Derived d;
d.f(1);      // "Derived int" ✓
d.f(3.14);   // ❌ error: no matching f(double) — Base::f(double) is hidden!
d.f();       // ❌ error: no matching f()       — Base::f() is hidden!

// Fix: bring them back with using
struct Derived2 : Base {
    using Base::f;           // re-introduces all Base::f overloads
    void f(int x) { ... }    // this one is replaced
};
```

## 🔴 Advanced — The Destructor Trap & UB

### The Most Dangerous Case: Missing Virtual Destructor

If your base destructor is not `virtual` and you `delete` through a base pointer, the derived destructor is **never called** — this is undefined behaviour:

```cpp
struct Base {
    ~Base() { std::cout << "~Base\n"; }   // NOT virtual — dangerous!
};

struct Derived : Base {
    int* data = new int[1000];
    ~Derived() { delete[] data; }         // ← this never runs
};

Base* p = new Derived;
delete p;
// Output:  ~Base
// Missing: ~Derived   ← 1000 ints leaked
// Technically:        undefined behaviour
```

The compiler chose `Base::~Base` at compile time because `p` is `Base*`. Without `virtual`, it doesn't know there's a `Derived::~Derived` to call.

---

### Fix: Always Virtual Destructor on Polymorphic Base

```cpp
struct Base {
    virtual ~Base() = default;   // ✓ enables correct destruction
};

struct Derived : Base {
    int* data = new int[1000];
    ~Derived() override { delete[] data; }  // ✓ now guaranteed to run
};

Base* p = new Derived;
delete p;
// Output: ~Derived then ~Base — correct order ✓
```

**Rule:** Any class with at least one `virtual` function must have a `virtual` destructor.

---

### When Name Hiding Is Intentional

Occasionally, hiding a base function is deliberate — for example, to restrict the interface or change the return type:

```cpp
struct Stream {
    void write(const char* s) { ... }
    void write(int n) { ... }
};

struct BinaryStream : Stream {
    // Intentionally hide text write — binary stream only accepts raw bytes
    void write(const void* buf, std::size_t len) { ... }
    // Stream::write(const char*) and Stream::write(int) are now hidden
    // Callers can't accidentally use the text overloads
};
```

This is rare and should always be clearly commented. The compiler warning `-Woverloaded-virtual` (GCC/Clang) will flag accidental hiding.

---

### `final` — Preventing Further Overrides

`final` on a class prevents any further derivation. `final` on a method prevents further overriding in subclasses:

```cpp
struct Base {
    virtual void f();
};

struct Middle : Base {
    void f() override final;   // no further override allowed
};

struct Leaf : Middle {
    void f() override;         // ❌ compile error — f() is final in Middle
};

struct Sealed final : Base {   // Sealed cannot be inherited from
    void f() override;
};

struct Child : Sealed { };     // ❌ compile error — Sealed is final
```

`final` also enables a compiler optimisation — when it can prove no further override exists, it can devirtualise the call (skip the vtable lookup entirely).

## 🔵 Expert — How the Vtable Works & `override` Enforcement

### What Actually Happens in Memory

Every class with at least one `virtual` function gets a **vtable** — a static array of function pointers. Every instance gets a hidden `vptr` pointing to its class's vtable:

```
Memory layout of a Dog object:

  [ vptr ] ──→  Dog's vtable:
  [ name ]        [0] = &Dog::speak
                  [1] = &Dog::~Dog

Memory layout of an Animal object:

  [ vptr ] ──→  Animal's vtable:
  [ name ]        [0] = &Animal::speak
                  [1] = &Animal::~Animal
```

A virtual call `p->speak()` compiles to roughly:

```cpp
// What p->speak() actually does:
(*p->vptr[0])(p);   // load vptr, index into vtable, call the function pointer
```

A non-virtual call `p->speak()` (where `p` is `Base*`) compiles directly to:

```cpp
Animal::speak(p);   // hardcoded at compile time — no vtable involved
```

This is why non-virtual dispatch is faster (one less indirection) but loses polymorphism entirely.

---

### Compiler Flags That Catch Hidden Functions

Turn these on in your build to catch accidental hiding at compile time:

```bash
# GCC / Clang
-Woverloaded-virtual   # warns when a derived function hides a base virtual overload
-Wsuggest-override     # warns when override keyword is missing on a virtual override

# Full recommended set for clean polymorphism
-Wall -Wextra -Woverloaded-virtual -Wsuggest-override -Werror
```

Example warning output:

```
warning: 'Derived::f(double)' hides overloaded virtual function [-Woverloaded-virtual]
note: hidden by 'Derived::f(double)' [-Woverloaded-virtual]
```

### Pure Virtual Functions and Abstract Classes

`= 0` makes a function **pure virtual** — the base class provides no implementation, and derived classes must override it or they become abstract (cannot be instantiated):

```cpp
struct Shape {
    virtual double area() const = 0;    // pure virtual — must override
    virtual void   draw() const = 0;    // pure virtual — must override
    virtual ~Shape() = default;
};

struct Circle : Shape {
    double radius;
    double area() const override { return 3.14159 * radius * radius; }
    void   draw() const override { std::cout << "O\n"; }
};

// Shape s;     ❌ cannot instantiate abstract class
Circle c;       // ✓ all pure virtuals overridden
```

If a derived class forgets to override a pure virtual:

```cpp
struct BadCircle : Shape {
    double area() const override { return 0; }
    // forgot draw() — BadCircle is still abstract
};

// BadCircle b;  ❌ error: cannot instantiate abstract class BadCircle
//                         because draw() is pure virtual and not overridden
```

### The Covariant Return Type Exception

A derived `override` can return a more derived pointer/reference type — this is called a **covariant return type** and is the only case where the signatures are allowed to differ:

```cpp
struct Base {
    virtual Base* clone() const { return new Base(*this); }
};

struct Derived : Base {
    Derived* clone() const override {    // ✓ covariant — Derived* is more derived than Base*
        return new Derived(*this);
    }
};

Derived d;
Derived* copy = d.clone();   // no cast needed — compiler knows it's Derived* ✓
Base*    copy2 = d.clone();  // also fine — implicit upcast ✓
```

This only works for raw pointer and reference types — `unique_ptr<Derived>` is not covariant with `unique_ptr<Base>`.

---

### Complete Rules Summary

```
Function in derived has same name as base — what happens?

Base function is virtual:
  ├── Derived uses override  →  Override ✓ — vtable updated, runtime dispatch
  ├── Derived omits override →  Override ✓ — still works, but fragile (use override!)
  └── Derived has wrong sig  →  Hiding ✗  — override keyword would catch this

Base function is NOT virtual:
  ├── Any derived function   →  Hiding ✗  — compile-time dispatch through base ptr
  └── override keyword used  →  Compile error ✓ — caught immediately

Destructor rules:
  ├── Base has virtual ~     →  Safe to delete through base pointer ✓
  └── Base has no virtual ~  →  UB on delete through base pointer ✗
```
