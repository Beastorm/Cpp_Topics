Imagine writing a big book where each chapter is written by a different author in a separate room. Each author writes their chapter without seeing the others. When the book is assembled, an editor (the linker) stitches everything together. What is a Symbol? (The Simple Answer)

🔤 A Symbol is Just a Name + a Location

Think of a symbol like an entry in a phonebook. It has a name (like "John Smith") and an address (where to find it in the binary).

When you write:

int value = 42;
int get_value() { return value; }

The compiler creates two symbols in the binary:

  • 📍 value → "this variable lives at byte offset 0 in the data section"
  • 📍 get_value → "this function lives at byte offset 0 in the code section"
💡 Why do we need symbols?
Your program is split into many files. When main.c wants to call get_value() which is in values.c, it needs to know where that function is. Symbols provide that directory/lookup — without them, files can't find each other!
⚡ When do you care about symbols in real life?
  • You get undefined reference to 'get_value' → missing symbol
  • You get multiple definition of 'value' → duplicate symbol
  • You're building a library and want to hide private functions from users

How Programs Are Built — Step by Step

🔄 The Build Pipeline

Building a C/C++ program has two big stages: compiling and linking.

Fig 1: The two stages of building a C/C++ program

📝 What "Separate Compilation" Means

  • Each .c or .cpp file is compiled completely independently. The compiler can't see into other files.
  • Each file produces an object file (.o) — a bundle of machine code + data + a symbol table.
  • The linker then takes all .o files and merges them into a final executable.
✅ Benefit
If you have 1000 source files and you change only one, you only need to recompile that one file and re-link. Huge speed win for big projects!

Think of an object file like a filing cabinet with labelled drawers:

🔑 The Most Important Insight

The machine code in .text and the symbol names in .symtab are completely separate. This has a big consequence:

💡 Why this matters
You can remove the symbol table (using a tool called strip) and the actual code still works! The code doesn't need its own name to run. Removing symbol names just means no one can look up where things are — useful for smaller, harder-to-reverse-engineer binaries.

Relocations — "Fill This In Later"

🍕 Real World Analogy
Imagine writing a party invitation before you've booked the venue. You write: "Party at [VENUE — TBD]" and leave a sticky note: "Fill in the venue address before printing." A relocation is exactly that sticky note — a placeholder in machine code that says "someone please fill in this address later."

⚙️ What Happens Step by Step

  • Compiler compiles main.c. It sees get_value() but doesn't know where it lives (it's in another file!).
  • Compiler writes the call instruction with zeros as the address: E8 00 00 00 00
  • Compiler also writes a relocation note in .rela.text: "At offset 0xA, please fill in the address of get_value"
  • The linker reads all relocation notes, looks up each symbol in the symbol tables, and patches in the real addresses.
Fig 3: Before and after the linker resolves a relocation
❌ What if the symbol is missing?
If you strip the symbol table from values.o before linking, the linker can't find get_value. You get the famous error:
undefined reference to 'get_value' Now you know exactly why this happens!

The static Keyword & Internal Linkage

🤫 Making a Function "Private to This File"

When you put static in front of a function or global variable at file scope (not inside a class), it gets LOCAL binding. This means:

  • ✅ You can use it in this file
  • 🚫 No other file can link to it
  • ✅ You can have the SAME name in multiple files — no conflict!
// values.c
static int helper() { return 42; }  ← LOCAL, only this file can use it

// main.c
static void helper() { printf("help\n"); }  ← ALSO LOCAL, different function

// No conflict! The linker treats these as completely separate things.

🚫 The Classic Bug: Definitions in Header Files

A common beginner mistake is putting definitions (not just declarations) in .h files:

// ❌ WRONG — values.h with a definition
int value = 42;          ← definition: creates storage
int get_value() { ... }   ← definition: creates code

// If main.c and utils.c both #include "values.h",
// BOTH files will have their own copy of value and get_value.
// Linker error: "multiple definition of 'value'"
// ✅ CORRECT — values.h with only declarations
extern int value;     ← declaration: "it exists somewhere"
int get_value();      ← declaration only

// values.c — ONE definition lives here
int value = 42;
int get_value() { return value; }
💡 Declaration vs Definition
Declaration = "Hey, this thing exists somewhere." (just tells the compiler about it)
Definition = "Here is the actual code/data." (allocates memory, creates the symbol)

Templates & Member Functions — Weak Symbols

🤔 The Problem with Templates

C++ templates must have their full definition in the header file. But if 10 different .cpp files include that header and use the template, each one compiles its own copy of the function body. Won't that cause "multiple definition" errors?

✅ The Clever Solution: WEAK Symbols
The compiler automatically marks template functions and class member functions (defined in headers) as WEAK. The linker rule for WEAK is: "keep one, throw away the rest." Problem solved!
Fig 5: Weak symbols let the linker deduplicate template instantiations

🔧 One Section Per Weak Symbol — How the Linker Discards Them

You might wonder: the linker merges entire sections at once — how can it throw away individual functions?

The compiler is smart about this. For each template instantiation or member function definition, it creates a brand new section just for that one function. The linker can then discard entire sections.

# objdump shows separate sections for each weak function
.text.MyClass::get_value()      ← its own private section
.text._Z8get_valueIiEv          ← another private section (template int)
.text.main                      ← normal main section

⚠️

The One Definition Rule (ODR) — Watch Out!

📏 The Rule in Plain English

If a function or variable has multiple copies (weak symbols), they must all be exactly identical. The linker picks one and throws the rest away — it does not check if they're the same.

💥 The Danger — Silent Undefined Behavior
If two files include the same header but one of them has a #define that changes the function body, you have an ODR violation. The linker silently picks one version. Your program may behave randomly — and no warning is shown! 
// ❌ NEVER do this in a header used by multiple files
#ifdef NEW_VERSION
  template<typename T> T get_value() { return 2; }
#else
  template<typename T> T get_value() { return 1; }
#endif

// file1.cpp → defines NEW_VERSION → get_value returns 2
// file2.cpp → no NEW_VERSION   → get_value returns 1
// Linker picks one silently. The other is GONE.
// This is UNDEFINED BEHAVIOR. Your program is broken.
✅ Safe Rule
All copies of a weak symbol across your entire project must have identical source code. Don't change template bodies with macros or ifdefs.

Weak Symbols — Making Overridable Functions

🛠️ A Manual Use of Weak: Allow Users to Override Your Function

You can mark any function as weak on purpose using __attribute__((weak)). This lets users of your library replace your default implementation with their own.

🍕 Analogy
Imagine a pizza kit company that includes a default tomato sauce packet (WEAK). But if you make your own sauce and put it in the box, the kit uses yours instead (GLOBAL). The kit's sauce is just the fallback. 
// library.c — your default (weak) implementation
Resource* load_resource() __attribute__((weak));
Resource* load_resource() {
    return load_from_file();   ← default fallback
}

// user.c — user's custom version (GLOBAL, no weak attribute)
Resource* load_resource() {
    return load_from_database();  ← overrides the default!
}

Linker rule: ONE GLOBAL beats ALL WEAK copies. The user's version wins.

Note: __attribute__((weak)) is a Linux/GCC/Clang specific feature, not standard C++.

Dynamic Libraries (.so files) & Visibility

📚 Two Kinds of Libraries

📦 Static Library (.a)

The library code is copied into your executable at link time. One big self-contained file.

Like buying ingredients and baking them into your cake.

🔗 Dynamic Library (.so / .dll)

The library stays separate. Your program loads it at runtime. Many programs can share one copy.

Like ordering from a restaurant that many people use.

⚡ Important: C/C++ Standards Don't Know About Dynamic Libraries!
The entire C++23 standard (2,124 pages!) says nothing about .so or .dll files. Dynamic linking is a feature of the operating system, not the language. What follows is Linux-specific.

👁️ Binding vs Visibility — Two Different Things

When shipping a shared library, you have two layers of control:

ConceptControls…Matters when…
Binding
(Global/Local)		 Can other .o files inside your library see this symbol? Building the library (static linking of .o files)
Visibility
(Default/Hidden)		 Can users who link against your .so see this symbol? When the user links their program to your .so
💡 The key insight
Inside a shared library, you need GLOBAL binding so all your internal .o files can call each other. But you also want HIDDEN visibility so external users can't access your private internals. These two goals don't conflict!
Fig 6: Inside a .so — all functions can see each other, but only DEFAULT visibility is exposed externally

🛡️ Best Practice: The Visibility Pattern

# Step 1: compile everything hidden by default
gcc -fvisibility=hidden -c library.c

// Step 2: define a portable macro for your public API
#define MYLIB_API __attribute__((visibility("default")))

// Step 3: mark only your public functions with it
MYLIB_API int api_function();     ← users can link to this
int internal_function();          ← hidden from users

Why hide first, then reveal? Say you have 2000 internal functions and 23 public ones. It's much easier to explicitly mark 23 things as public than to remember to hide 1977 things!

🪟 Windows Version of the Same Idea

// Windows uses __declspec instead of __attribute__
#ifdef _WIN32
  #ifdef BUILDING_DLL
    #define MYLIB_API __declspec(dllexport)  ← when building the DLL
  #else
    #define MYLIB_API __declspec(dllimport)  ← when using the DLL
  #endif
#else
  #define MYLIB_API __attribute__((visibility("default")))  ← Linux
#endif

This lets you ship one header that works for both building and using the library, on both Windows and Linux.

🗂️ The Core Rule

Something gets a symbol if it's a single, findable thing that other code needs to locate. Things that are temporary or relative (like variables on the stack) don't need symbols.

Tools to Inspect Your Binaries

🔍 nm — List Symbols

The quickest way to see what symbols are in a file.

# See all symbols
nm file.o

# C++ friendly names (demangle)
nm -C file.o

# See dynamic symbols (.so)
nm -D libfoo.so

Letter guide: T=global code, t=local code, D=global data, d=local data, U=undefined, W=weak, u=unique

🔍 objdump — Disassemble

See the actual machine code and more.

# List all sections
objdump -s file.o

# Disassemble code (see assembly)
objdump -d file.o

# Show relocations too
objdump -dr file.o

🔍 readelf — Deep Inspection

More detailed than nm — shows visibility info too!

# List all sections
readelf -S -W file.o

# Show symbols WITH visibility
readelf -s file.o
# nm does NOT show visibility!

✂️ strip — Remove Symbol Tables

Shrink your binary by removing symbol info you don't need in production.

# Remove static symbol table
strip myapp

# Remove debug info only
strip --strip-debug myapp

# Verify: nm shows nothing
nm myapp   ← "no symbols"
nm -D myapp ← still shows API

Final Summary

Fig 9: All concepts — how they connect together

✅ The 10 Things to Remember

  1. A symbol is a name → location entry in a table (like a phonebook for your binary).
  2. Each .c/.cpp file is compiled in isolation into a .o (object file).
  3. When a file can't find another function at compile time, it leaves zeros + a relocation note. The linker fills it in.
  4. If the linker can't find a symbol → undefined reference error. Now you know why!
  5. GLOBAL binding = visible to all files. LOCAL = private to one file (static).
  6. WEAK = many copies allowed, linker picks one. Used for templates & member functions.
  7. All WEAK copies of a symbol must be identical — ODR violation = undefined behavior with no warning.
  8. For shared libraries: Binding = who inside the .so can see it. Visibility = who outside can link to it.
  9. Best practice: compile with -fvisibility=hidden, then explicitly export your public API.
  10. Only singular, findable things get symbols. Stack variables, params, and class types don't.

🟢 BEGINNER LEVEL Expect these in entry-level / fresher interviews

Q1 🟢 — What is a symbol in the context of C/C++ compilation?

Also asked as: "What does a linker use to connect object files?"

💡 Show Model Answer

A symbol is a named entry in a symbol table that maps a name (like a function or variable name) to a location inside the binary — specifically, a section name and an offset within that section.

When you write int value = 42;, the compiler creates a symbol called value pointing to the .data section. When you write int get_value() {...}, it creates a symbol called get_value pointing to the .text (code) section.

Symbols are what allow separately compiled files to find and call each other's functions and variables — the linker uses the symbol table as a directory to resolve all cross-file references.

🌟 Bonus point to mention
Symbol names and machine code live in separate sections of the object file — so you can strip symbols without breaking the code itself.

Q2 🟢 — What is the difference between a declaration and a definition in C/C++?

Also asked as: "Why should you not put definitions in header files?"

💡 Show Model Answer

A declaration tells the compiler that something exists somewhere — it doesn't create any storage or machine code. A definition actually creates the thing — storage for a variable, or machine code for a function — and produces a symbol in the object file.

// Declaration — no symbol created
extern int value;
int get_value();

// Definition — symbol IS created
int value = 42;
int get_value() { return value; }

If you put a definition in a header file and that header is included by multiple .cpp files, each translation unit gets its own copy of the symbol, and the linker will raise a "multiple definition" error. Declarations are safe to put in headers because they don't create symbols.

Q3 🟢 — What causes an "undefined reference" linker error? How do you fix it?

💡 Show Model Answer

An "undefined reference" error means the linker found a relocation entry (a reference to a symbol) in one object file but could not find the matching definition (the actual symbol) in any of the object files or libraries being linked.

Common causes:

  • You declared a function but forgot to write the definition (or forgot to compile/link the file containing it)
  • You stripped the symbol table from an object file before linking
  • You forgot to link a required library (-lm, -lpthread, etc.)
  • Name mangling mismatch — a C++ function called from C without extern "C"

Fix: Ensure the .cpp/.c file containing the definition is compiled and passed to the linker, or link the correct library.

Q4 🟢 — What does the static keyword mean at file scope? How does it affect linking?

💡 Show Model Answer

At file scope, static gives a function or variable internal linkage. This means the symbol gets LOCAL binding in the ELF symbol table (shown as lowercase t or d in nm), making it invisible to the linker when resolving references from other files.

Key effects:

  • Other translation units cannot link to this function/variable
  • You can have the same name in multiple files with no conflict — each is a completely separate LOCAL symbol
  • The machine code is identical — only the symbol table entry changes
// Both files can have "helper" — no linker conflict
static int helper() { return 1; }  // file1.c
static int helper() { return 2; }  // file2.c

Note: static inside a class means something different (class-level member). Inside a function it means static storage duration. Context matters!

Q5 🟢 — What are the key sections inside an ELF object file?

💡 Show Model Answer
SectionContents
.textMachine code (CPU instructions) — the compiled functions
.dataInitialized global and static variables
.bssZero-initialized global variables (no space needed in file)
.symtabSymbol table — names and locations of all symbols
.rela.textRelocation entries — fix-it notes for the linker
.debug_*DWARF debug information (file names, line numbers, types)

Key insight: these sections are completely separate. The machine code in .text has no symbol names embedded — those live only in .symtab. This is why strip can remove symbols without breaking the code.

Q6 🟢 — What is a relocation? Why does the compiler need it?

💡 Show Model Answer

A relocation is a note stored in a separate section (.rela.text) that tells the linker: "at this byte offset in the code, please fill in the address of this symbol."

The compiler needs it because each file is compiled in isolation. When main.c calls get_value() from another file, the compiler doesn't know the final address of that function. So it writes a placeholder (zeros) in the call instruction and leaves a relocation entry.

The linker later loops over all relocation tables, looks up each symbol, computes the correct address, and patches it into the binary. After linking, all relocations are resolved and no zeros remain.

Q7 🟡 — What are the four ELF symbol binding types? How do they differ?

💡 Show Model Answer

Critical difference WEAK vs UNIQUE: A GLOBAL symbol can override a WEAK one (enabling the plugin/override pattern). A UNIQUE symbol cannot be overridden — it's used for inline variables to guarantee exactly one instance without allowing user overrides.

Q8 🟡 — Why do C++ templates produce WEAK symbols? How does the linker handle them?

💡 Show Model Answer

Template definitions must live in header files so every translation unit can instantiate them. When multiple .cpp files include the same header and use the same template, each one compiles its own copy of the instantiation — producing potentially many identical definitions.

The compiler automatically marks template instantiations (and member functions defined in headers) as WEAK symbols. The linker rule for WEAK is: keep exactly one, silently discard all duplicates.

Clever trick: The linker normally merges entire sections — it can't remove individual functions from inside a section. To work around this, the compiler creates a brand-new section for each weak symbol (e.g., .text.MyClass::get_value()). The linker can then discard entire sections it doesn't need, keeping only one copy.

⚠️ Watch out
All weak copies must be identical (ODR rule). The linker does NOT verify this. Silent undefined behavior if they differ!

Q9 🟡 — What is the One Definition Rule (ODR)? What happens if you violate it?

💡 Show Model Answer

The One Definition Rule states: in an entire program, there must be exactly one definition of every non-inline function, variable, class, template, etc. For things that are allowed to have multiple definitions (templates, inline functions, member functions defined in headers), all copies must be identical.

What happens if you violate it: The linker silently picks one version and discards the rest — with no warning or error. This leads to undefined behavior. Your program may produce wrong results, crash, or behave differently on different builds/platforms.

// ODR violation — DO NOT do this
#ifdef FEATURE_X
  template<typename T> T compute() { return 2; }
#else
  template<typename T> T compute() { return 1; }
#endif
// Different TUs may see different bodies → UB!

Real-world danger: ODR violations are notoriously hard to debug because there's no compiler/linker error. Tools like -fsanitize=address or gold linker's --detect-odr-violations can sometimes catch them.

Q11 🟡 — How would you limit the public API surface of a shared library (.so)?

💡 Show Model Answer

The recommended approach is:

  • Compile all objects with -fvisibility=hidden — this makes every symbol hidden by default
  • Define an export macro (e.g. MYLIB_API) that expands to __attribute__((visibility("default"))) on Linux
  • Tag only your public API functions/classes with the macro
// In CMake:
target_compile_options(mylib PRIVATE -fvisibility=hidden)

// In your header:
#define MYLIB_API __attribute__((visibility("default")))
MYLIB_API int public_function();   ← exported
int internal_helper();             ← hidden

Why "hide-by-default"?

  • Smaller .dynsym table → faster startup, smaller binary
  • Prevents users from accidentally depending on internal functions
  • Reduces symbol conflicts when multiple libraries define the same internal names
  • It's easier to remember to mark 20 public things than to hide 2000 private ones

Q12 🟡 — Why does a shared library have two symbol tables (.symtab and .dynsym)? Can you remove either?

💡 Show Model Answer

They serve different purposes at different stages:

TablePurposeUsed whenCan strip?
.symtabAll symbols — used during static linking to build the .soBuild time only✅ Yes — strip libfoo.so
.dynsymPublic API only (default-visibility) — used by the dynamic linkerEvery program startup❌ No — runtime requires it

After the .so is built and shipped, you can safely strip .symtab — linking is already done, only dynamic symbol resolution happens at runtime. Stripping it shrinks the binary and hides internal implementation details (a security-through-obscurity benefit).

.dynsym is always needed because ld.so reads it every time the program starts to resolve cross-.so symbol references.

Q15 🔴 — How does __attribute__((weak)) enable a plugin/override pattern in C? Walk through the mechanics.

💡 Show Model Answer

When you mark a function with __attribute__((weak)), the compiler emits it with WEAK binding in the ELF symbol table. The linker rule for symbol resolution is:

  1. If there is exactly one GLOBAL symbol with this name → use it, discard all WEAK copies
  2. If there are only WEAK symbols → pick one (usually the first encountered)
  3. You cannot have two GLOBAL symbols with the same name — that's a linker error
// library.c — shipped default
void on_event() __attribute__((weak));
void on_event() { /* default no-op */ }

// user.c — user's override (no weak → GLOBAL)
void on_event() {
    printf("custom handler!\n");  ← this wins
}

This enables a simple plug-in system for static linking. The library ships a safe default; users can silently override it just by providing their own definition. It's used extensively in embedded C for interrupt handlers — the HAL provides weak defaults, and user code overrides only the ones it needs.

⚠️ Linux/GCC/Clang only
This is not standard C/C++. It's an ELF-specific extension. Windows has no direct equivalent (you'd use function pointers or DLL import tables instead).

Q16 🔴 — Why can't the linker simply remove individual weak-symbol functions from a merged .text section? How does it actually do it?

💡 Show Model Answer

The linker works at section granularity — it copies or discards entire sections, not individual bytes within them. So if a function is in the middle of a .text section along with 50 other functions, the linker can't surgically remove just that one function.

The compiler's solution: For every weak/unique symbol (template instantiation, member function defined in a header, inline variable), the compiler creates a dedicated section containing only that one symbol:

# objdump -t main.o | grep text
.text.main                       ← normal code section
.text._ZN7MyClass8get_valueEv    ← section for ONE member fn
.text._Z9get_valueIiEv           ← section for ONE template inst.

Now the linker can discard the entire section when it encounters a duplicate. This is the mechanism that makes the C++ COMDAT (COMmon DATa) feature work — what GCC/Clang call "section groups."

Additional benefit: This same mechanism enables dead code elimination (--gc-sections flag). If a function is never called, its entire dedicated section can be garbage-collected by the linker, reducing binary size.

Q17 🔴 — What is UNIQUE symbol binding (ELF extension)? How does it differ from WEAK, and what C++ feature uses it?

💡 Show Model Answer

UNIQUE is a newer ELF extension to the GNU ABI. Like WEAK, it allows a symbol to exist in multiple object files with the linker keeping only one copy. The critical difference:

PropertyWEAKUNIQUE
Multiple copies allowed?✅ Yes✅ Yes
GLOBAL overrides it?✅ Yes — GLOBAL wins❌ No — UNIQUE cannot be overridden
NM letterW / wu (lowercase)
Used forTemplates, member fns, manual __attribute__((weak))C++17 inline variables

C++17 introduced inline variables to solve the problem of defining variables in headers:

// Before C++17: had to put definition in a .cpp file
// C++17+: define directly in the header
inline int global_count = 0;  ← UNIQUE symbol

struct Config {
    static inline int max_size = 100;  ← also UNIQUE
};

The UNIQUE binding guarantees one copy across the entire program, while preventing anyone from accidentally or intentionally overriding it with a GLOBAL definition — which would be wrong for a language-level inline variable.

Confusing NM output note: u (lowercase) = UNIQUE, and U (uppercase) = UNDEFINED. These are completely unrelated and the naming is unfortunately confusing.

Q18 🔴 — You have a .so library. A user reports that calling an undocumented internal function (by casting a function pointer from dlsym) works. How would you prevent this?

💡 Show Model Answer

If dlsym() can find a symbol, it means that symbol is in the .dynsym (dynamic symbol table) with DEFAULT visibility. The fix is to ensure internal functions have HIDDEN visibility so they never appear in .dynsym.

Step-by-step fix:

  • Add -fvisibility=hidden to your compile flags for all .o files going into the .so. Now every symbol defaults to HIDDEN — they won't appear in .dynsym.
  • Create an export macro and tag only your intentionally public functions: 
    #define MYLIB_API __attribute__((visibility("default")))
MYLIB_API int documented_function();  ← in .dynsym
int internal_function();              ← NOT in .dynsym
  • Optionally run strip to remove .symtab as well, so even tools like nm can't list internal symbol names.
  • Verify with: nm -D libmylib.so — only your public API should appear.

With HIDDEN visibility, dlsym("internal_function") returns NULL — the symbol simply doesn't exist in the dynamic symbol table. The code is still there and runs, but it's unreachable from the outside.

Q19 🔴 — Explain how the separate compilation model and relocations work together. Why does the compiler emit zeros in the call instruction?

💡 Show Model Answer

The core problem: In the separate compilation model, each .c/.cpp file is compiled in complete isolation. When the compiler processes main.c and sees call get_value(), it has no idea:

  • Which object file get_value is in
  • Where in the final executable the linker will place that object file's code
  • Therefore: what the final absolute or relative address of get_value will be

What the compiler does:

  1. Emits the call opcode (E8) followed by 4 zero bytes as a placeholder for the target address
  2. Writes a relocation entry in .rela.text: "At byte offset 0xA in this section, fill in the PC-relative address of symbol get_value, type R_X86_64_PLT32"

What the linker does:

  1. Decides where every section from every object file will be placed in the final binary
  2. Loops over all relocation tables
  3. For each relocation: looks up the symbol in the symbol table, computes the correct address (relative or absolute, depending on type), and patches the zero bytes with the real value

Critically, the linker doesn't need to understand assembly — it just needs to know "compute distance to symbol X and write 4 bytes at offset Y." This clean separation is what makes the compilation model so powerful and parallelizable.

Q20 🔴 — How would you create a cross-platform (Linux + Windows) export macro for a shared library? What are the differences between the two platforms?

💡 Show Model Answer

Platform differences:

AspectLinux (.so)Windows (.dll)
Default visibilityAll symbols exported (unless -fvisibility=hidden)Nothing exported unless explicitly marked
Export syntax__attribute__((visibility("default")))__declspec(dllexport)
Import syntax(same as export or nothing)__declspec(dllimport)
Same header for build+use?Yes — same attribute worksNo — need dllexport when building, dllimport when using

Standard cross-platform pattern:

// mylib_export.h
#ifdef _WIN32
  #ifdef MYLIB_BUILDING_DLL
    #define MYLIB_API __declspec(dllexport)
  #else
    #define MYLIB_API __declspec(dllimport)
  #endif
#else
  #define MYLIB_API __attribute__((visibility("default")))
#endif

// mylib.h — public header shipped to users
MYLIB_API int do_something(int x);

Usage:

  • When building the DLL/so: define MYLIB_BUILDING_DLL (via CMake target_compile_definitions) + pass -fvisibility=hidden on Linux
  • When using the library: just include the header — the macro automatically uses dllimport on Windows and default visibility on Linux

This single header works correctly in both build contexts, which is why you often see this pattern in open-source libraries like OpenCV, Qt, and Boost.

🧩 TRICKY SCENARIO QUESTIONS Think through these — often asked in system-design / debugging rounds

🧩 Scenario A — The Mystery Bug

Your team has a shared library used by 10 programs. After upgrading the library, one program started returning wrong results — but only on some machines. No code in that program changed. What could be causing this, and how would you investigate?

💡 Think Through It

Likely causes:

  • ODR violation: Two different versions of a weak symbol exist in the process (e.g., the library and the executable both define a template function differently). The linker or dynamic linker picks one.
  • Symbol interposition: With default visibility on Linux, a global symbol in the executable can override one in the .so — if the program defines a function with the same name as an internal library function, and that library function changed behavior.
  • ABI break: The struct/class layout changed in the new library but the program was compiled against the old header.

Investigation steps:

  • Run nm -D libmylib.so before and after upgrade — did any public symbol disappear or change?
  • Use readelf -s to check for duplicate symbol names between the executable and .so
  • Compile with -Wl,--detect-odr-violations (gold linker) or -fsanitize=address
  • Use -fvisibility=hidden going forward to prevent symbol interposition

🧩 Scenario B — Duplicate Symbol

You're getting multiple definition of 'process' at link time. You search your codebase and find process() defined in only ONE file. How is this possible, and how do you fix it?

💡 Think Through It

Most likely cause: The definition of process() is in a header file, not a .cpp file. Every .cpp file that includes that header gets its own copy of the definition, each producing a GLOBAL symbol — and the linker rejects multiple GLOBAL definitions.

Fix options:

  • Move the definition to a single .cpp file; keep only the declaration in the header
  • Mark it inline — inline functions are allowed to be defined in multiple TUs (they become WEAK symbols)
  • Mark it static — gives LOCAL binding; each TU gets its own private copy (fine if unshared state is OK)
  • Put it in an anonymous namespace — equivalent to static in C++

To diagnose: Run nm *.o | grep ' T process' — if multiple object files show T process (global), that confirms the duplicate definitions.

🧩 Scenario C — Shrinking a Binary

Your embedded system has tight flash memory constraints. Your .so is 4MB but you estimate the actual code is only 2MB. What techniques would you use to reduce the size?

💡 Think Through It

Symbol-table related:

  • strip libfoo.so — removes .symtab (can save 10–30% in debug builds)
  • strip --strip-debug libfoo.so — removes DWARF debug info (often very large)

Dead code elimination:

  • Compile with -ffunction-sections -fdata-sections — puts each function/variable in its own section
  • Link with -Wl,--gc-sections — linker garbage-collects unused sections
  • Use -fvisibility=hidden — hidden symbols are better candidates for elimination

Compiler optimizations:

  • -Os — optimize for size instead of speed
  • -flto (Link Time Optimization) — cross-file inlining and dead code removal

Verify what's taking space:

nm --size-sort -r libfoo.so | head -20   ← largest symbols
readelf -S libfoo.so                     ← section sizes
