# 1. Introduction to STL

**STL** = Standard Template Library — a collection of pre-built, highly optimized tools in C++:

- **Containers** — data structures (vector, map, set, …)
- **Algorithms** — functions like sort, search, copy
- **Iterators** — unified way to traverse any container

> **Analogy:** STL is like IKEA furniture — ready-made, high-quality pieces instead of building everything from raw wood.

**Recommended setup:**

```cpp
#include <bits/stdc++.h>
using namespace std;
```

# 2. Sequence Containers

Sequence containers store elements in a **linear** order.

---

## 🧱 std::array

**Fixed-size array.** Size is known at compile time. Stored on the stack; no dynamic allocation.

> **Analogy:** A row of fixed chairs in a classroom — you know exactly how many seats there are.

**Internal structure:** Fixed-size contiguous array (compile-time size), usually stack allocated.

### Functions

| Function       | Description               | Time |
|----------------|---------------------------|------|
| `at(i)`        | Bounds-checked access     | O(1) |
| `operator[]`   | Fast access (no check)    | O(1) |
| `front()`      | First element             | O(1) |
| `back()`       | Last element              | O(1) |
| `size()`       | Number of elements        | O(1) |
| `fill(val)`    | Fill all elements         | O(n) |
| `data()`       | Raw pointer to buffer     | O(1) |

### Example

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    array<int, 5> arr{10, 20, 30, 40, 50};
    cout << arr.size() << " " << arr.front() << " " << arr.back() << endl;
    arr.fill(99);
    return 0;
}
```

### std::array vs std::vector

| Feature        | `std::array`                            | `std::vector`                           |
|----------------|-----------------------------------------|-----------------------------------------|
| Size           | Fixed at compile time                   | Dynamic (grows/shrinks)                 |
| Memory         | Usually stack                           | Heap                                    |
| `push_back`    | Not available                           | Available                               |
| Performance    | Slightly better (no dynamic allocation) | Good with `reserve()`                   |
| When to use    | Size is known and fixed                 | Most general cases                      |

### C-style array vs std::array

| Feature              | C-style array (`int arr[5]`)         | `std::array<int, 5>`                     |
|----------------------|---------------------------------------|-------------------------------------------|
| Size info            | Not stored (decays to pointer)        | Stored (`size()` available)               |
| Bounds checking      | None — `[]` is unchecked               | `at(i)` is bounds-checked                 |
| Assignment           | Cannot assign (`arr2 = arr1` invalid) | Copyable/assignable like any object       |
| Passing to functions | Decays to pointer, loses size info    | Passes full type, size preserved          |
| STL algorithm support| Works via pointers (`begin`/`end`)    | Works directly (`arr.begin()`, `arr.end()`)|
| Safety               | Easy to misuse (no checks)             | Safer, modern C++ idiomatic choice         |
| Performance          | Identical (zero overhead)              | Identical (zero overhead)                  |
| When to use          | Legacy C code / C interop              | Modern C++ — always prefer this           |

## 📦 std::vector

**Dynamic array.** Size can grow or shrink at runtime. The most commonly used STL container.

> **Analogy:** A stretchable rubber band that automatically expands when you add more items.

**Internal structure:** Dynamic contiguous array on the heap. Reallocates (1.5×–2× growth) when capacity is exceeded.

### Why `reserve()` is Critical

When `push_back()` fills the current capacity, the vector allocates a new larger block, copies all elements, and frees the old block. Without `reserve()`, this can make large insertions expensive.

- `v.reserve(n)` pre-allocates memory for at least `n` elements upfront
- After the call: `size()` stays the same, `capacity() >= n`
- No reallocations happen until the reserved capacity is exceeded

```cpp
// Without reserve — repeated reallocations
vector<int> bad;
for (int i = 0; i < 100000; i++) bad.push_back(i);

// With reserve — single allocation
vector<int> good;
good.reserve(100000);
for (int i = 0; i < 100000; i++) good.push_back(i);
```

**Rule:** Always use `reserve()` when the final size can be estimated.

### Functions

| Function              | Description                    | Time Complexity  |
|-----------------------|--------------------------------|------------------|
| `push_back(val)`      | Add at end                     | Amortized O(1)   |
| `pop_back()`          | Remove last                    | O(1)             |
| `insert(pos, val)`    | Insert at iterator position    | O(n)             |
| `erase(pos)`          | Remove at iterator position    | O(n)             |
| `size()`              | Current number of elements     | O(1)             |
| `capacity()`          | Allocated memory slots         | O(1)             |
| `reserve(n)`          | Pre-allocate memory            | O(n)             |
| `at(i)` / `[]`        | Access element                 | O(1)             |
| `front()` / `back()`  | Access ends                    | O(1)             |
| `clear()`             | Remove all elements            | O(n)             |
| `emplace_back(args)`  | Construct in-place at end      | Amortized O(1)   |

### Example

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    vector<int> v{1, 2, 3};
    v.push_back(4);
    v.insert(v.begin() + 1, 99);
    v.erase(v.begin() + 2);
    v.reserve(100);

    cout << "Size: " << v.size() << ", Capacity: " << v.capacity() << endl;
    for (int x : v) cout << x << " ";
    cout << endl;
    return 0;
}
```

### std::vector vs std::deque

| Feature                      | `std::vector`               | `std::deque`                    |
|------------------------------|-----------------------------|---------------------------------|
| `push_front` / `pop_front`   | O(n)                        | O(1)                            |
| Best use case                | General purpose + appends   | Frequent operations at both ends|

### std::vector vs std::list

| Feature                  | `std::vector`            | `std::list`                    |
|--------------------------|--------------------------|--------------------------------|
| Random access            | O(1)                     | O(n)                           |
| Insert/Delete in middle  | O(n)                     | O(1) (with iterator)           |
| Best for                 | Random access + appends  | Frequent middle insert/delete  |

## 🚂 std::deque

**Double-ended queue.** Fast push/pop at both front and back.

> **Analogy:** A train with engines on both sides — you can board or exit from either end quickly.

**Internal structure:** Multiple fixed-size contiguous blocks (chunks) linked together on the heap. Allows efficient insertion at both ends without shifting elements.

### Functions

| Function              | Description           | Time Complexity |
|-----------------------|-----------------------|-----------------|
| `push_front(val)`     | Add to front          | O(1)            |
| `push_back(val)`      | Add to back           | O(1)            |
| `pop_front()`         | Remove from front     | O(1)            |
| `pop_back()`          | Remove from back      | O(1)            |
| `front()` / `back()`  | Access ends           | O(1)            |
| `operator[]` / `at()` | Random access         | O(1)            |
| `size()`              | Number of elements    | O(1)            |
| `insert(pos, val)`    | Insert anywhere       | O(n)            |

### Example

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    deque<int> dq;
    dq.push_back(10);
    dq.push_front(5);
    dq.push_back(20);

    cout << "Front: " << dq.front() << ", Back: " << dq.back() << endl;
    dq.pop_front();
    dq.pop_back();

    cout << "After pops: ";
    for (int x : dq) cout << x << " ";
    cout << endl;
    cout << "dq[0]: " << dq[0] << endl;
    return 0;
}
```

### std::deque vs std::vector

| Feature                     | `std::deque`                      | `std::vector`                      |
|-----------------------------|-----------------------------------|------------------------------------|
| `push_front` / `pop_front`  | O(1)                              | O(n)                               |
| Memory layout               | Multiple fixed blocks             | Single contiguous block            |
| Random access               | O(1)                              | O(1)                               |
| Memory overhead             | Slightly higher (block pointers)  | Lower                              |
| Best for                    | Operations at both ends           | Mostly appends + random access     |

## 🔗 std::list

**Doubly linked list.** Best for frequent middle insertions/deletions.

> **Analogy:** A chain of friends holding hands — each knows the previous and next person. You can insert or remove someone in the middle without disturbing the whole line.

**Internal structure:** Doubly linked list (nodes with `prev` + `next` pointers), heap allocated. No contiguous memory, so no random access.

### Functions

| Function                              | Description                      | Time Complexity |
|---------------------------------------|----------------------------------|-----------------|
| `push_front(val)` / `push_back(val)`  | Add to front/back                | O(1)            |
| `pop_front()` / `pop_back()`          | Remove from front/back           | O(1)            |
| `insert(pos, val)`                    | Insert at iterator position      | O(1)            |
| `erase(pos)`                          | Remove at iterator               | O(1)            |
| `front()` / `back()`                  | Access ends                      | O(1)            |
| `size()`                              | Number of elements               | O(1)            |
| `begin()` / `end()`                   | Forward iterators                | —               |
| `rbegin()` / `rend()`                 | Reverse iterators                | —               |

### Example

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    list<int> l{1, 2, 3};
    l.push_front(0);
    l.push_back(4);

    auto it = l.begin();
    advance(it, 2);
    l.insert(it, 99);

    cout << "After insert: ";
    for (int x : l) cout << x << " ";
    cout << endl;

    l.pop_front();
    l.erase(next(l.begin()));
    return 0;
}
```

### std::list vs std::vector

| Feature                  | `std::list`                       | `std::vector`                  |
|--------------------------|-----------------------------------|--------------------------------|
| Random access            | O(n)                              | O(1)                           |
| Insert/Delete in middle  | O(1) (with iterator)              | O(n)                           |
| Memory layout            | Non-contiguous (linked nodes)     | Contiguous                     |
| Best for                 | Frequent middle insert/delete     | Random access + appends        |

### std::list vs std::deque

| Feature                  | `std::list`                       | `std::deque`                       |
|--------------------------|-----------------------------------|------------------------------------|
| Middle insert/delete     | O(1)                              | O(n)                               |
| Random access            | O(n)                              | O(1)                               |
| End operations           | O(1)                              | O(1)                               |
| Memory overhead          | Higher (per-node pointers)        | Lower                              |
| Best for                 | Many middle modifications         | Both-end operations + random access|

## ➡️ std::forward_list

**Singly linked list.** Lightweight, forward-only. Use when you only need forward traversal and want to minimize memory overhead.

> **Analogy:** A one-way street of friends — each knows only the next person ahead.

**Internal structure:** Singly linked list (each node has only a `next` pointer). Lower per-node overhead than `list`.

### Functions

| Function              | Description                       | Time Complexity |
|-----------------------|-----------------------------------|-----------------|
| `push_front(val)`     | Add to front                      | O(1)            |
| `pop_front()`         | Remove front                      | O(1)            |
| `insert_after(pos)`   | Insert after iterator position    | O(1)            |
| `erase_after(pos)`    | Remove after iterator position    | O(1)            |
| `front()`             | Access first element              | O(1)            |
| `before_begin()`      | Iterator before first (special)   | —               |

### Example

```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    forward_list<int> fl{1, 2, 3};
    fl.push_front(0);

    auto it = fl.begin();
    fl.insert_after(it, 99);

    cout << "Elements: ";
    for (int x : fl) cout << x << " ";
    cout << endl;
    return 0;
}
```

### std::forward_list vs std::list

| Feature                    | `std::forward_list`               | `std::list`                         |
|----------------------------|-----------------------------------|-------------------------------------|
| Direction                  | Forward only                      | Forward + backward                  |
| Memory overhead            | Lower (one pointer per node)      | Higher (two pointers per node)      |
| `push_back`                | Not available                     | Available                           |
| `insert` / `erase`         | `insert_after` / `erase_after`    | Direct `insert` / `erase`           |
| Best for                   | Memory-critical, forward-only     | Full bidirectional linked list      |

# 3. Associative Containers

Elements are stored in **sorted order** based on keys. Backed by a **Red-Black Tree** (self-balancing BST). All core operations are **O(log n)**.

---

## 📚 std::set & multiset

- **`set`** — Unique values, sorted ascending
- **`multiset`** — Duplicate values allowed, sorted ascending

> **Analogy:** A perfectly sorted bookshelf where no two books have the same title (`set`), vs allowing duplicate titles (`multiset`).

**Internal structure:** Red-Black Tree (self-balancing BST), heap allocated nodes.

### Functions

| Function          | `set` (unique)                  | `multiset` (duplicates)            | Time     |
|-------------------|---------------------------------|------------------------------------|----------|
| `insert(val)`     | Ignores duplicates              | Always inserts                     | O(log n) |
| `erase(val)`      | Removes the one occurrence      | Removes **all** occurrences        | O(log n) |
| `find(val)`       | Returns iterator or `end()`     | Same                               | O(log n) |
| `count(val)`      | 0 or 1                          | Actual count                       | O(log n) |
| `lower_bound(val)`| First element >= val            | Same                               | O(log n) |

### Example

```cpp
set<int> s{3, 1, 2, 2};         // becomes {1, 2, 3}
multiset<int> ms{1, 1, 2, 2};

s.insert(10);
cout << s.count(2);              // 1

ms.insert(1);
cout << ms.count(1);             // 3
```

### std::set vs std::multiset

| Feature              | `std::set`                         | `std::multiset`                         |
|----------------------|------------------------------------|-----------------------------------------|
| Duplicates allowed?  | No                                 | Yes                                     |
| `insert(10)` twice   | Second insert is ignored           | Both values are stored                  |
| `count(10)`          | Returns 0 or 1                     | Returns actual number of occurrences    |
| `erase(10)`          | Removes the value (if exists)      | Removes **all** occurrences             |
| Use case             | Unique elements in sorted order    | Duplicates allowed, sorted              |

### std::set vs std::map

| Feature           | `std::set`                       | `std::map`                              |
|-------------------|----------------------------------|-----------------------------------------|
| Stores            | Only unique values               | Key + Value pairs                       |
| Access            | Values act as their own keys     | Access value using `m[key]`             |
| Iteration         | Just the values                  | Pairs (`first` = key, `second` = value) |
| Common use case   | Unique sorted collection         | Dictionary / key-value lookup           |

### std::multiset vs std::multimap

| Feature           | `std::multiset`                     | `std::multimap`                            |
|-------------------|-------------------------------------|--------------------------------------------|
| Stores            | Only values (with duplicates)       | Key + Value pairs (duplicate keys allowed) |
| Access by key     | Values act as keys                  | Use key to get all values                  |
| Common use case   | Sorted collection with duplicates   | One key having multiple values             |
| `equal_range()`   | Less commonly needed                | Very useful                                |

---

## 🗺️ std::map & multimap

- **`map`** — Unique keys, sorted by key, one value per key
- **`multimap`** — Duplicate keys allowed

> **Analogy:** Dictionary (word → meaning). `map` = one meaning per word. `multimap` = multiple meanings allowed.

**Internal structure:** Red-Black Tree (key → value pairs), heap allocated, sorted by key.

### Functions

| Function           | Description                           | Time     |
|--------------------|---------------------------------------|----------|
| `m[key] = val`     | Insert / update (creates key if absent) | O(log n) |
| `insert({k, v})`   | Insert (does not overwrite)           | O(log n) |
| `find(key)`        | Search — returns iterator             | O(log n) |
| `count(key)`       | 0 or 1 (map) / actual (multimap)      | O(log n) |
| `erase(key)`       | Remove                                | O(log n) |
| `lower_bound(key)` | First key >= given key                | O(log n) |
| `equal_range(key)` | Range of all entries for a key        | O(log n) |

### Example

```cpp
map<string, int> marks;
marks["Rohit"] = 95;
marks["Priya"] = 88;
marks.insert({"Amit", 92});

cout << marks["Rohit"] << endl;

for (auto& p : marks)
    cout << p.first << " → " << p.second << endl;
```

### std::map vs std::multimap

| Feature                  | `std::map`                          | `std::multimap`                             |
|--------------------------|-------------------------------------|---------------------------------------------|
| Duplicate keys?          | No                                  | Yes                                         |
| `m[key] = value`         | Allowed (overwrites if key exists)  | **Not allowed**                             |
| `insert({key, val})`     | Ignores if key already exists       | Always inserts (even if key exists)         |
| `count(key)`             | 0 or 1                              | Actual number of values for the key         |
| Get all values for key   | `m[key]` or `find()`                | Use `equal_range(key)`                      |
| Common use case          | Unique key → value mapping          | One key can have multiple values            |

# 4. Unordered Containers

Use **hashing** instead of trees → **O(1) average** for insert/find/erase. No ordering guaranteed.

**Internal structure:** Hash Table (buckets + hash function), heap allocated.

> **Worst case: O(n)** — rare, caused by hash collisions (poor hash function or adversarial input).

### Ordered vs Unordered — Key Difference

| Feature        | `set` / `map`      | `unordered_set` / `unordered_map` |
|----------------|--------------------|------------------------------------|
| Order          | Always sorted      | No guaranteed order                |
| Speed (avg)    | O(log n)           | **O(1)**                           |
| Implementation | Red-Black Tree     | Hash Table                         |

### Functions (all unordered variants)

| Function    | Description      | Time (avg) |
|-------------|------------------|------------|
| `insert()`  | Add element      | O(1)       |
| `find()`    | Search           | O(1)       |
| `erase()`   | Remove           | O(1)       |
| `count()`   | Existence check  | O(1)       |

### Example

```cpp
unordered_set<int> us{10, 20, 30};
unordered_map<string, int> scores;
scores["Rohit"] = 95;

if (us.find(20) != us.end())
    cout << "Found!";
```

### std::set vs std::unordered_set

| Feature         | `std::set`           | `std::unordered_set`    |
|-----------------|----------------------|-------------------------|
| Order           | Always sorted        | No guaranteed order     |
| Average time    | O(log n)             | **O(1)**                |
| Worst case      | O(log n)             | O(n) (rare)             |
| Implementation  | Red-Black Tree       | Hash Table              |
| When to use     | Need sorted order    | Fast lookup, order unimportant |

### std::map vs std::unordered_map

| Feature         | `std::map`                         | `std::unordered_map`             |
|-----------------|------------------------------------|----------------------------------|
| Order           | Keys always sorted                 | No guaranteed order              |
| Average time    | O(log n)                           | **O(1)**                         |
| Implementation  | Red-Black Tree                     | Hash Table                       |
| When to use     | Need sorted keys or range queries  | Fast lookup, order unimportant   |
| Worst case      | O(log n)                           | O(n) (rare)                      |

### std::unordered_set vs std::unordered_map

| Feature     | `unordered_set`                     | `unordered_map`                      |
|-------------|-------------------------------------|--------------------------------------|
| Stores      | Only unique values                  | Key + Value pairs                    |
| Access      | Check existence (`find`, `count`)   | Get value using key (`um[key]`)      |
| Use case    | Fast unique membership test         | Fast key-to-value lookup             |
| Iteration   | Just the values                     | Pairs (`first` = key, `second` = val)|

### std::unordered_set vs std::unordered_multiset

| Feature              | `unordered_set`         | `unordered_multiset`                    |
|----------------------|-------------------------|-----------------------------------------|
| Duplicates allowed?  | No                      | Yes                                     |
| `count(10)`          | 0 or 1                  | Actual number of occurrences            |
| `insert(10)` twice   | Second is ignored       | Both are stored                         |
| Use case             | Fast unique membership  | Fast frequency count without ordering  |

# 5. Container Adapters

Wrappers around other containers that **restrict the interface** to enforce a specific access pattern. You cannot iterate directly over them.

---

## 📚 stack (LIFO)

**Last In, First Out.** Only the top element is accessible.

> **Analogy:** Stack of plates — last plate placed on top is the first you can take off.

**Internal structure:** Container adapter using `deque` by default.

### Functions

| Function  | Description       | Time |
|-----------|-------------------|------|
| `push()`  | Add to top        | O(1) |
| `pop()`   | Remove top        | O(1) |
| `top()`   | View top element  | O(1) |
| `size()`  | Number of items   | O(1) |

### Example

```cpp
stack<int> st;
st.push(10);
st.push(20);
cout << st.top();   // 20
st.pop();
```

**Common uses:** Undo functionality, expression evaluation, call stack simulation, bracket matching.

### std::stack vs std::queue

| Feature       | `std::stack`                          | `std::queue`                          |
|---------------|----------------------------------------|----------------------------------------|
| Order         | LIFO (Last In, First Out)              | FIFO (First In, First Out)             |
| Access        | `top()`                                | `front()`                              |
| Real-world    | Stack of plates, undo, call stack      | Ticket queue, print jobs, BFS          |
| Common use    | Backtracking, parentheses matching     | Task scheduling, message queues        |

### std::stack vs std::priority_queue

| Feature              | `std::stack`                          | `std::priority_queue`                     |
|----------------------|----------------------------------------|--------------------------------------------|
| Order                | LIFO (Last In First Out)               | Highest priority first                      |
| Access               | Always the most recently pushed         | Always the largest (or smallest)            |
| Use case             | Undo, call stack, bracket matching      | Top-K elements, Dijkstra, priority tasks    |
| Underlying           | deque (default)                         | Heap (vector by default)                    |

---

## 🎫 queue (FIFO)

**First In, First Out.** Access only at front (dequeue) and back (enqueue).

> **Analogy:** Queue at a ticket counter — first come, first served.

**Internal structure:** Container adapter using `deque` by default.

### Functions

| Function    | Description        | Time |
|-------------|--------------------|------|
| `push()`    | Add to back        | O(1) |
| `pop()`     | Remove from front  | O(1) |
| `front()`   | View front element | O(1) |
| `back()`    | View back element  | O(1) |
| `size()`    | Number of items    | O(1) |

### Example

```cpp
queue<int> q;
q.push(10);
q.push(20);
cout << q.front();  // 10
q.pop();
```

**Common uses:** Task scheduling, message queues, BFS.

### std::queue vs std::deque

| Feature                  | `std::queue`                             | `std::deque`                              |
|--------------------------|------------------------------------------|-------------------------------------------|
| Interface                | Restricted (push, pop, front, back)      | Full interface (iterators, `[]`, etc.)    |
| Can iterate              | No                                       | Yes                                       |
| Can access middle        | No                                       | Yes                                       |
| Use case                 | Pure FIFO semantics                      | Queue behavior + full flexibility         |
| Implementation (default) | Uses `deque` internally                  | Direct container                          |

---

## ⭐ priority_queue (Max-Heap by default)

**Highest-priority element always at top.** Backed by a heap.

> **Analogy:** VIP queue — highest priority person is always served first.

**Internal structure:** Max-heap using `std::vector` + heap algorithms by default.

### Functions

| Function  | Description                         | Time     |
|-----------|-------------------------------------|----------|
| `push()`  | Add element                         | O(log n) |
| `pop()`   | Remove top element                  | O(log n) |
| `top()`   | View top element (max by default)   | O(1)     |
| `size()`  | Number of elements                  | O(1)     |

### Example

```cpp
priority_queue<int> pq;          // max-heap
pq.push(10);
pq.push(30);
pq.push(20);
cout << pq.top();                // 30

// Min-heap
priority_queue<int, vector<int>, greater<int>> minHeap;
```

**Common uses:** Top-K elements, Dijkstra's algorithm, priority scheduling.

### std::priority_queue vs std::queue

| Feature                 | `priority_queue`                     | `queue`                           |
|-------------------------|--------------------------------------|-----------------------------------|
| Order                   | Highest priority first               | FIFO (First In First Out)         |
| Access                  | `top()` (largest/smallest)           | `front()`                         |
| Use case                | Top-K, Dijkstra, priority scheduling | Simple task queue, BFS            |
| Underlying structure    | Heap (vector by default)             | deque (by default)                |

### std::priority_queue vs std::set

| Feature                     | `priority_queue`                     | `std::set`                               |
|-----------------------------|--------------------------------------|------------------------------------------|
| Element access              | Only the top (max/min)               | All elements in sorted order             |
| Duplicates                  | Allowed                              | Not allowed (use `multiset`)             |
| Get second largest          | Not directly possible                | Easy (reverse iterators / `rbegin`)      |
| `push` / `pop` time         | O(log n)                             | O(log n)                                 |
| Best when you need          | Only the maximum/minimum element     | Full sorted access to all elements       |

### std::priority_queue vs std::vector (as heap)

| Feature         | `priority_queue`                    | `vector` + `make_heap` / `push_heap`     |
|-----------------|-------------------------------------|------------------------------------------|
| Interface       | Limited (`push`, `pop`, `top`)      | Full vector interface                    |
| Heap management | Automatic                           | Manual (`push_heap`, `pop_heap`)         |
| Convenience     | Much easier                         | More flexible but verbose                |
| When to use     | Only need max/min                   | Need both heap behavior + other ops      |

---

## Stack vs Queue vs Priority Queue

| Feature         | `stack`                   | `queue`                    | `priority_queue`               |
|-----------------|---------------------------|----------------------------|--------------------------------|
| Order           | LIFO                      | FIFO                       | Highest priority first         |
| Access          | `top()`                   | `front()`                  | `top()`                        |
| Core operations | `push`, `pop`, `top`      | `push`, `pop`, `front`     | `push`, `pop`, `top`           |
| Use case        | Undo, recursion, brackets | Task scheduling, BFS       | Dijkstra, top-K elements       |
| Implementation  | deque (default)           | deque (default)            | vector + heap                  |

# 7. STL Algorithms

All algorithms operate on **iterator ranges** (`begin`, `end`).

---

## Sorting Algorithms

| Algorithm         | Purpose                          | Time       | Example                                               |
|-------------------|----------------------------------|------------|-------------------------------------------------------|
| `sort()`          | Sort entire range                | O(n log n) | `sort(v.begin(), v.end());`                           |
| `stable_sort()`   | Sort preserving relative order   | O(n log n) | `stable_sort(v.begin(), v.end());`                    |
| `is_sorted()`     | Check if sorted                  | O(n)       | `is_sorted(v.begin(), v.end())`                       |
| `partial_sort()`  | Sort first k elements            | O(n log k) | `partial_sort(v.begin(), v.begin()+k, v.end());`      |
| `nth_element()`   | Place nth element in position    | O(n) avg   | `nth_element(v.begin(), v.begin()+n, v.end());`       |

### std::sort vs std::stable_sort

| Feature          | `std::sort`                      | `std::stable_sort`                      |
|------------------|----------------------------------|-----------------------------------------|
| Stability        | Not stable                       | Stable (preserves relative order)       |
| Time             | O(n log n)                       | O(n log n)                              |
| Extra memory     | O(1) in-place                    | O(n)                                    |
| When to use      | General purpose (faster)         | When order of equal elements matters    |

### std::sort vs std::partial_sort

| Feature               | `std::sort`              | `std::partial_sort`                    |
|-----------------------|--------------------------|----------------------------------------|
| What it sorts         | Entire range             | Only first k elements                  |
| Time                  | O(n log n)               | O(n log k)                             |
| Remaining elements    | Fully sorted             | Only first k sorted (rest unsorted)    |
| Use case              | Need fully sorted data   | Need only top-k elements sorted        |

### std::sort vs std::nth_element

| Feature              | `std::sort`             | `std::nth_element`                         |
|----------------------|-------------------------|--------------------------------------------|
| What it does         | Fully sorts the range   | Places nth element in correct position     |
| Time                 | O(n log n)              | O(n) average                               |
| Result               | Entire range sorted     | Only nth position is guaranteed correct    |
| Use case             | Need fully sorted data  | Need only kth smallest/largest             |

---

## Copy Algorithms

| Algorithm          | Purpose                    | Example                                                                   |
|--------------------|----------------------------|---------------------------------------------------------------------------|
| `copy()`           | Copy entire range          | `copy(v.begin(), v.end(), dest.begin());`                                 |
| `copy_if()`        | Copy matching elements     | `copy_if(v.begin(), v.end(), dest.begin(), [](int x){ return x > 5; });`  |
| `copy_n()`         | Copy n elements            | `copy_n(v.begin(), 3, dest.begin());`                                     |
| `copy_backward()`  | Copy in reverse order      | `copy_backward(v.begin(), v.end(), dest.end());`                          |

### std::copy vs std::copy_if

| Feature             | `std::copy`                   | `std::copy_if`                               |
|---------------------|-------------------------------|----------------------------------------------|
| What it copies      | All elements in range         | Only elements satisfying condition           |
| Fourth argument     | Not needed                    | Predicate (condition function)               |
| Use case            | Simple copy                   | Conditional copy (filter while copying)      |

---

## Other Useful Algorithms

| Algorithm    | Purpose                        | Example                             |
|--------------|--------------------------------|-------------------------------------|
| `iota()`     | Fill with increasing numbers   | `iota(v.begin(), v.end(), 1);`      |
| `fill()`     | Fill with a value              | `fill(v.begin(), v.end(), 0);`      |
| `find()`     | Linear search                  | `find(v.begin(), v.end(), 5);`      |

---

## Full Algorithms Example

```cpp
vector<int> v = {5, 2, 8, 1, 9, 3};

sort(v.begin(), v.end());                          // 1 2 3 5 8 9
partial_sort(v.begin(), v.begin()+3, v.end());     // first 3 sorted
nth_element(v.begin(), v.begin()+2, v.end());      // 3rd smallest at index 2

vector<int> dest(6);
copy(v.begin(), v.end(), dest.begin());

iota(v.begin(), v.end(), 10);                      // 10 11 12 13 14 15
```

# 9. emplace vs insert

`emplace` constructs objects **in-place** directly inside the container, avoiding a temporary object. Prefer `emplace` for complex objects.

```cpp
// Less efficient — creates a temporary pair then copies/moves it
vec.push_back(pair<int, string>(1, "One"));

// Better — constructs the pair directly inside the vector
vec.emplace_back(1, "One");

// Same principle for sets and maps
s.emplace(42);
m.emplace(10, "Ten");
```

### push_back vs emplace_back

| Feature               | `push_back(val)`                              | `emplace_back(args...)`                       |
|-----------------------|-----------------------------------------------|-----------------------------------------------|
| How it works          | Creates object then copies/moves into vector  | Constructs object **directly** in container   |
| Temporary object      | Usually creates one                           | Avoids temporary                              |
| Performance           | Slightly slower for complex objects           | Faster (preferred for non-trivial types)      |
| Syntax                | `v.push_back({1, "Hi"});`                     | `v.emplace_back(1, "Hi");`                    |
| Best for              | Simple types (int, string)                    | Complex objects (pairs, custom classes)       |

### insert vs emplace

| Feature        | `emplace`                              | `insert` / `push_back`                       |
|----------------|----------------------------------------|----------------------------------------------|
| How it works   | Constructs object **in-place**         | Creates temporary object then copies/moves   |
| Efficiency     | Usually better (avoids temporary)      | Slightly slower                              |
| Syntax         | `v.emplace_back(1, "Hello");`          | `v.push_back({1, "Hello"});`                 |
| Use case       | Complex objects (pairs, custom classes)| Simple types                                 |

# 10. Comparison Tables & Best Practices

## Container Choice Guide

| Requirement                    | Best Container(s)                |
|--------------------------------|----------------------------------|
| Fast random access             | `vector`, `array`, `deque`       |
| Insert/delete at both ends     | `deque`, `list`                  |
| Frequent middle insertions     | `list`                           |
| Unique + sorted                | `set`                            |
| Fast lookup (no order)         | `unordered_set`                  |
| Key-value pairs                | `map` / `unordered_map`          |
| LIFO                           | `stack`                          |
| FIFO                           | `queue`                          |
| Highest priority first         | `priority_queue`                 |

## Best Practices

1. Always use `reserve()` with `vector` when the final size is roughly known.
2. Prefer `emplace_back()` / `emplace()` over `push_back()` / `insert()` for complex objects.
3. Use range-based for loops with `const auto&` to avoid copies.
4. Understand **iterator invalidation** — especially with `vector` erase/insert.
5. For algorithms, always pass correct `begin` / `end` iterators.
6. Choose **unordered** containers when sorted order is not needed.
7. Never use `m[key]` on a `map` when you only want to check existence — it inserts a default value; use `find()` or `count()` instead.

# 12. STL Interview Questions

> **Tip:** Always mention time complexity, internal implementation, and real-world use cases. Be ready to write small code snippets.

---

## Vector & Sequence Containers

**Q1: What is the difference between `std::vector` and `std::array`?**

- `array` → Fixed size at compile time. No dynamic resizing. Stack allocated.
- `vector` → Dynamic size. Grows/shrinks at runtime. Heap allocated.

Use `array` when size is known and fixed. Use `vector` for most general cases.

---

**Q2: Why is `reserve()` important in vector? What happens without it?**

Without `reserve()`, every time capacity is exceeded, vector reallocates and copies all existing elements. With `reserve()`:

```cpp
vector<int> v;
v.reserve(100000);   // pre-allocates — no reallocations during insertion
```

---

**Q3: Explain iterator invalidation in `std::vector`. Give an example.**

Insertions and deletions in the middle or at the beginning invalidate all iterators, references, and pointers.

```cpp
vector<int> v = {1, 2, 3, 4};
auto it = v.begin() + 1;
v.insert(it, 99);     // it is now invalid!
```

**Safe approach:** Recalculate the iterator after any modification.

---

**Q4: When would you choose `std::deque` over `std::vector`?**

When you need frequent insertions/deletions at **both** ends. `deque` gives O(1) `push_front` / `pop_front` while `vector` is O(n).

---

**Q5: Difference between `std::list` and `std::forward_list`?**

- `list` = Doubly linked list. Forward + backward traversal. `push_back` supported.
- `forward_list` = Singly linked list. Forward only. Lower memory overhead.

Use `forward_list` when you only need forward iteration and memory is tight.

---

## Associative Containers

**Q6: Difference between `std::set` and `std::unordered_set`?**

| Feature        | `set`          | `unordered_set`   |
|----------------|----------------|-------------------|
| Order          | Sorted         | No order          |
| Time           | O(log n)       | O(1) average      |
| Implementation | Red-Black Tree | Hash Table        |

Use `unordered_set` when order doesn't matter and you want speed.

---

**Q7: How does `std::map` work internally?**

Implemented as a **Red-Black Tree** (self-balancing BST). Keys are always sorted. All core operations are O(log n).

---

**Q8: What is the difference between `map` and `multimap`?**

- `map` → Keys are unique
- `multimap` → Duplicate keys allowed

Use `multimap` when one key can have multiple values (e.g. multiple phone numbers per person).

---

**Q9: How do you find all values for a key in a `multimap`?**

```cpp
auto range = mm.equal_range(key);
for (auto it = range.first; it != range.second; ++it)
    cout << it->second;
```

---

**Q10: Can you use a custom comparator with `set` or `map`?**

Yes:

```cpp
struct cmp {
    bool operator()(int a, int b) const { return a > b; }
};
set<int, cmp> s;   // descending order
```

---

## Unordered Containers & Hashing

**Q11: What can cause worst-case O(n) performance in `unordered_set`?**

Hash collisions — many keys mapping to the same bucket (poor hash function or adversarial input). Provide a custom hasher when needed.

---

**Q12: How would you count frequency of words using STL?**

```cpp
unordered_map<string, int> freq;
for (const auto& word : words) freq[word]++;
```

---

## Container Adapters

**Q13: How would you implement a min-heap using `priority_queue`?**

```cpp
priority_queue<int, vector<int>, greater<int>> minHeap;
```

---

**Q14: Difference between `stack` and `queue`?**

- `stack` = LIFO (Last In First Out)
- `queue` = FIFO (First In First Out)

---

## Algorithms

**Q15: Difference between `std::sort` and `std::stable_sort`?**

- `sort` → Faster, not stable (equal elements may change relative order)
- `stable_sort` → Slower, preserves relative order of equal elements

---

**Q16: When would you use `nth_element` instead of `sort`?**

When you only need the kth smallest/largest element, not the full sorted array. Average O(n) vs O(n log n).

```cpp
nth_element(v.begin(), v.begin() + k, v.end());
```

---

**Q17: How do you copy only even numbers from one vector to another?**

```cpp
copy_if(src.begin(), src.end(), back_inserter(dest),
        [](int x){ return x % 2 == 0; });
```

---

## Modern C++ & Performance

**Q18: What is `std::span` and why is it useful?**

A non-owning view of contiguous memory. Avoids copying data; works with arrays, vectors, and raw pointers.

```cpp
void process(span<int> data);   // flexible, zero-copy API
```

---

**Q19: Explain `emplace_back` vs `push_back`. When is `emplace_back` better?**

`emplace_back` constructs the object directly in the container — avoids a temporary.

```cpp
vec.emplace_back(1, "Hello");    // constructs in-place — preferred
vec.push_back({1, "Hello"});     // creates temporary first
```

Especially useful for complex objects (pairs, custom classes).

---

**Q20: What are the benefits of `emplace` over `insert` in sets/maps?**

Avoids creating a temporary object that would then be copied/moved into the container.

---

## Miscellaneous & Tricky

**Q21: What happens when you access `m[key]` in a map if the key doesn't exist?**

It **inserts** the key with a default-constructed value (0 for int, empty string, etc.). This is often unexpected — prefer `find()` or `count()` for existence checks.

---

**Q22: How do you sort a vector of pairs by the second value?**

```cpp
sort(v.begin(), v.end(), [](const auto& a, const auto& b){
    return a.second < b.second;
});
```

---

**Q23: Can we store user-defined objects in STL containers?**

Yes, but they must be Copyable / Movable. For associative containers, must define `operator<` or provide a custom comparator.

---

**Q24: What is the time complexity of `v.size()` in vector vs list?**

Both are O(1) in C++11 and later.

---

**Q25: How would you remove all even numbers from a vector efficiently?**

```cpp
v.erase(remove_if(v.begin(), v.end(), [](int x){ return x % 2 == 0; }), v.end());
```

The erase-remove idiom: `remove_if` shifts non-matching elements to the front, returns a new logical end, then `erase` physically removes the tail.
