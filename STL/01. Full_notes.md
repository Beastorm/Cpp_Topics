### What is STL?

**STL** = Standard Template Library

A collection of **pre-built, highly optimized** tools in C++:
- **Containers** (data structures)
- **Algorithms** (functions like sort, search)
- **Iterators** (ways to traverse containers)

**Real-world analogy:**  
Think of STL as **IKEA furniture** — you get ready-made high-quality pieces instead of building everything from raw wood.

**Core purpose & benefits:**
- Ready-made high-quality data structures and algorithms
- Generic (works with any type)
- Extremely fast and battle-tested

### 🧱 std::array

**Fixed-size array** (better than C-style arrays).

**💡 Analogy:** A row of fixed chairs in a classroom — you know exactly how many seats there are.

**📌 Sub-topics covered in this section:**
- Syntax and creation
- Fixed size at compile time
- Major functions
- Internal structure
- Full practical example
- When to use

> **Underlying**: Fixed-size contiguous array (compile-time size)  
> **Memory**: Usually stack  
> **Implication**: Extremely fast, but **cannot grow** at runtime.

| Function     | Description               | Time |
|--------------|---------------------------|------|
| `at(i)`      | Bounds-checked access     | O(1) |
| `operator[]` | Fast access               | O(1) |
| `front()`    | First element             | O(1) |
| `back()`     | Last element              | O(1) |
| `size()`     | Number of elements        | O(1) |
| `fill(val)`  | Fill all                  | O(n) |
| `data()`     | Raw pointer               | O(1) |

**📦 Full Practical Example**
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    array<int, 5> arr{10, 20, 30, 40, 50};
    cout << arr.size() << " " << arr.front() << " " << arr.back() << endl;
    arr.fill(99);
    return 0;
}
```

### 📦 std::vector (The Most Important Container)

**Dynamic array** — size can grow or shrink at runtime.

**💡 Analogy:** A stretchable rubber band that automatically expands when you add more items.

**📌 Sub-topics covered in this section:**
- Basic usage with push_back and dynamic resizing
- Internal Data Structure (contiguous heap array)
- Why `reserve()` is Critical (3 key points fully explained)
- Major Functions with time complexities
- Full Practical Example (runnable code)
- Comparison tables with array, deque, and list

> **Underlying**: Dynamic contiguous array (resizable)  
> **Memory**: Heap allocated  
> **Implication**: Random access O(1), middle insert O(n), reallocates when full.

**🔥 Why `reserve()` is Critical**

**Sub-topics explained in detail:**

1. **Problem with repeated reallocations**  
   When `push_back()` is called and `capacity()` is full, vector allocates a new larger block (usually 1.5×–2× larger), copies **all** existing elements to the new block, and frees the old memory. Without `reserve()`, for n insertions the total cost can approach **O(n²)** due to repeated copying (amortized O(1) per push but quadratic worst-case without reservation).

2. **How `reserve(n)` pre-allocates memory**  
   `v.reserve(n)` allocates memory for **at least n elements upfront**. After the call:  
   - `size()` stays the same  
   - `capacity() >= n`  
   No reallocations or copies happen until you exceed the reserved capacity.

3. **Performance benefit with example**  
   Pre-allocation avoids expensive reallocations and element copies, making large insertions dramatically faster (linear total cost instead of quadratic).

   ```cpp
   // Bad: repeated reallocations (slow for large n)
   vector<int> bad;
   for(int i = 0; i < 100000; i++) bad.push_back(i);

   // Good: pre-allocate once
   vector<int> good;
   good.reserve(100000);
   for(int i = 0; i < 100000; i++) good.push_back(i);
   ```

   **Key insight:** Always use `reserve()` when you can estimate the final size in advance.

| Function            | Description                  | Time Complexity       |
|---------------------|------------------------------|-----------------------|
| `push_back(val)`    | Add at end                   | Amortized O(1)        |
| `pop_back()`        | Remove last                  | O(1)                  |
| `insert(pos, val)`  | Insert at position           | O(n)                  |
| `erase(pos)`        | Remove at position           | O(n)                  |
| `size()`            | Current number of elements   | O(1)                  |
| `capacity()`        | Allocated memory slots       | O(1)                  |
| `reserve(n)`        | Pre-allocate memory          | O(n)                  |
| `at(i)` / `[]`      | Access element               | O(1)                  |
| `front()` / `back()`| Access ends                  | O(1)                  |
| `clear()`           | Remove all elements          | O(n)                  |
| `emplace_back(args)`| Construct in place (efficient)| Amortized O(1)       |

**📦 Full Practical Example**
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    vector<int> v{1, 2, 3};
    v.push_back(4);
    v.insert(v.begin() + 1, 99);
    v.erase(v.begin() + 2);
    v.reserve(100);

    cout << "Size: " << v.size() << ", Capacity: " << v.capacity() << endl;
    for (int x : v) cout << x << " ";
    cout << endl;
    return 0;
}
```

**📊 Difference Between `std::vector` and `std::array`**

| Feature                  | `std::vector`                          | `std::array`                              |
|--------------------------|----------------------------------------|-------------------------------------------|
| Size                     | Dynamic                                | Fixed at compile time                     |
| Memory                   | Heap                                   | Usually stack                             |
| `push_back`              | Available                              | Not available                             |
| Performance              | Good with `reserve()`                  | Slightly better (no dynamic allocation)   |
| When to use              | Most cases                             | When size is known and fixed              |

**📊 Difference Between `std::vector` and `std::deque`**

| Feature                    | `std::vector`                          | `std::deque`                              |
|----------------------------|----------------------------------------|-------------------------------------------|
| `push_front` / `pop_front` | O(n)                                   | O(1)                                      |
| Best use case              | General purpose + appends              | Frequent operations at both ends          |

**📊 Difference Between `std::vector` and `std::list`**

| Feature                    | `std::vector`                          | `std::list`                               |
|----------------------------|----------------------------------------|-------------------------------------------|
| Random access              | O(1)                                   | O(n)                                      |

### 🚂 std::deque (Double Ended Queue)

**Fast operations at both ends** (push/pop front and back are efficient).

**💡 Analogy:** Train with engines on both sides — you can board or exit from either end quickly.

**📌 Sub-topics covered in this section:**
- Why use deque over vector
- Internal structure
- Major functions & operations
- Full practical example
- Comparison tables

> **Underlying**: Multiple fixed-size contiguous blocks (chunks) linked together  
> **Memory**: Heap (blocks on heap)  
> **Key trait**: Allows efficient insertion/deletion at **both ends** without shifting elements  
> **Implication**:  
> - Random access: O(1)  
> - push_front / pop_front / push_back / pop_back: O(1)  
> - Middle insert/erase: O(n)

| Function            | Description                          | Time Complexity |
|---------------------|--------------------------------------|-----------------|
| `push_front(val)`   | Add to front                         | O(1)            |
| `push_back(val)`    | Add to back                          | O(1)            |
| `pop_front()`       | Remove from front                    | O(1)            |
| `pop_back()`        | Remove from back                     | O(1)            |
| `front()` / `back()`| Access ends                          | O(1)            |
| `operator[]` / `at()` | Random access                      | O(1)            |
| `size()`            | Number of elements                   | O(1)            |
| `insert(pos, val)`  | Insert anywhere                      | O(n)            |

**📦 Full Practical Example**
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    deque<int> dq;
    
    dq.push_back(10);
    dq.push_front(5);
    dq.push_back(20);
    
    cout << "Front: " << dq.front() << ", Back: " << dq.back() << endl;
    cout << "Size: " << dq.size() << endl;
    
    dq.pop_front();
    dq.pop_back();
    
    cout << "After pops: ";
    for (int x : dq) cout << x << " ";
    cout << endl;
    
    // Random access works
    cout << "dq[0]: " << dq[0] << endl;
    return 0;
}
```

**📊 Difference Between `std::deque` and `std::vector`**

| Feature                    | `std::deque`                              | `std::vector`                             |
|----------------------------|-------------------------------------------|-------------------------------------------|
| push_front / pop_front     | O(1)                                      | O(n)                                      |
| Memory layout              | Multiple fixed blocks                     | Single contiguous block                   |
| Random access              | O(1)                                      | O(1)                                      |
| Best for                   | Frequent operations at both ends          | Mostly appends + random access            |
| Memory overhead            | Slightly higher (block pointers)          | Lower                                     |

### 🔗 std::list (Doubly Linked List)

**Best for frequent middle insertions/deletions.**

**💡 Analogy:** A chain of friends holding hands — each knows the previous and next person. You can easily insert or remove someone in the middle without disturbing the whole line.

**📌 Sub-topics covered in this section:**
- Core properties and why use list
- Internal Data Structure
- Major Functions & time complexities
- Full practical example
- Comparison tables (vs vector/deque)

> **Underlying**: Doubly linked list (nodes with prev + next pointers)  
> **Memory**: Heap allocated nodes (not contiguous)  
> **Key trait**: No random access; efficient insert/erase anywhere with iterator  
> **Implication**:  
> - push_front / push_back / insert / erase (with iterator): O(1)  
> - Random access: O(n)  
> - No `operator[]`

| Function                  | Description                          | Time Complexity |
|---------------------------|--------------------------------------|-----------------|
| `push_front(val)` / `push_back(val)` | Add to front/back                | O(1)            |
| `pop_front()` / `pop_back()` | Remove from front/back            | O(1)            |
| `insert(pos, val)`        | Insert at iterator position          | O(1)            |
| `erase(pos)`              | Remove at iterator                   | O(1)            |
| `front()` / `back()`      | Access ends                          | O(1)            |
| `size()`                  | Number of elements                   | O(1)            |
| `begin()` / `end()`       | Iterators (forward)                  | —               |
| `rbegin()` / `rend()`     | Reverse iterators                    | —               |

**📦 Full Practical Example**
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    list<int> l{1, 2, 3};
    l.push_front(0);
    l.push_back(4);
    
    cout << "Front: " << l.front() << ", Back: " << l.back() << endl;
    
    // Insert in middle
    auto it = l.begin();
    advance(it, 2);           // move to 3rd element
    l.insert(it, 99);
    
    cout << "After insert: ";
    for (int x : l) cout << x << " ";
    cout << endl;
    
    l.pop_front();
    l.erase(next(l.begin())); // erase second element
    
    return 0;
}
```

**📊 Difference Between `std::list` and `std::vector`**

| Feature                    | `std::list`                               | `std::vector`                             |
|----------------------------|-------------------------------------------|-------------------------------------------|
| Random access              | O(n)                                      | O(1)                                      |
| Insert/Delete in middle    | O(1) (with iterator)                      | O(n)                                      |
| Memory layout              | Non-contiguous (linked nodes)             | Contiguous                                |
| Best for                   | Frequent middle insert/delete             | Random access + appends                   |

**📊 Difference Between `std::list` and `std::deque`**

| Feature                    | `std::list`                               | `std::deque`                              |
|----------------------------|-------------------------------------------|-------------------------------------------|
| Middle insert/delete       | O(1)                                      | O(n)                                      |
| Random access              | O(n)                                      | O(1)                                      |
| End operations             | O(1)                                      | O(1)                                      |
| Memory overhead            | Higher (per-node pointers)                | Lower                                     |
| Best for                   | Many middle modifications                 | Both-end operations + random access       |

### ➡️ std::forward_list (Singly Linked List)

**Lightweight forward-only list.** Use when you only need forward traversal and want to save memory.

**💡 Analogy:** A one-way street of friends — each knows only the next person ahead. You can only move forward.

**📌 Sub-topics covered in this section:**
- When to choose forward_list
- Internal structure
- Major functions
- Full example
- Comparison with list

> **Underlying**: Singly linked list (each node has only next pointer)  
> **Memory**: Heap nodes (lower overhead than `list`)  
> **Key trait**: Forward traversal only; no `push_back` (only `push_front`)  
> **Implication**:  
> - push_front / insert_after / erase_after: O(1)  
> - No random access, no reverse traversal

| Function             | Description                     | Time Complexity |
|----------------------|---------------------------------|-----------------|
| `push_front(val)`    | Add to front                    | O(1)            |
| `pop_front()`        | Remove front                    | O(1)            |
| `insert_after(pos)`  | Insert after iterator           | O(1)            |
| `erase_after(pos)`   | Remove after iterator           | O(1)            |
| `front()`            | Access first element            | O(1)            |
| `before_begin()`     | Iterator before first (special) | —               |

**📦 Full Practical Example**
```cpp
#include <bits/stdc++.h>
using namespace std;

int main() {
    forward_list<int> fl{1, 2, 3};
    
    fl.push_front(0);
    
    // Insert after first element
    auto it = fl.begin();
    fl.insert_after(it, 99);
    
    cout << "Elements: ";
    for (int x : fl) cout << x << " ";
    cout << endl;
    
    return 0;
}
```

**📊 Difference Between `std::forward_list` and `std::list`**

| Feature                    | `std::forward_list`                       | `std::list`                               |
|----------------------------|-------------------------------------------|-------------------------------------------|
| Direction                  | Forward only                              | Forward + backward                        |
| Memory overhead            | Lower (one pointer per node)              | Higher (two pointers per node)            |
| `push_back`                | Not available                             | Available                                 |
| `insert` / `erase`         | `insert_after` / `erase_after`            | Direct `insert` / `erase`                 |
| Best for                   | Memory-critical forward-only needs        | Full bidirectional linked list            |

### 📚 std::set & multiset

**set** = Unique + Sorted  
**multiset** = Duplicates allowed + Sorted

**💡 Analogy:** A perfectly sorted bookshelf where no two books have the same title (`set`), vs allowing duplicate titles (`multiset`).

**Sub-topics explained in detail:**

- `set<int> s;` → unique + sorted ascending
- `set<int, greater<int>> s_desc;` → descending
- Golden rules: No duplicates (ignored), Always sorted, Red-Black Tree

- See table in "Major Functions" below

- Demonstrates insert, count, erase behavior

- All core ops: O(log n)

- Powerful for range queries: `s.lower_bound(val)`

- Red-Black Tree (balanced BST)

**💡 Analogy:** A perfectly sorted bookshelf where no two books have the same title (`set`), vs allowing duplicate titles (`multiset`).

> **Underlying**: **Red-Black Tree** (self-balancing BST)  
> **Memory**: Heap allocated nodes  
> **Key trait**: Always **sorted** + balanced  
> **Implication**:  
> - `insert` / `erase` / `find` → **O(log n)**  
> - No random access (`[]` not allowed)

**For `multiset`**: Same as `set` (Red-Black Tree), but allows duplicate values.

| Function     | set (Unique)                  | multiset (Duplicates)         | Time     |
|--------------|-------------------------------|-------------------------------|----------|
| `insert(val)`| Ignores duplicates            | Allows duplicates             | O(log n) |
| `erase(val)` | Removes the value             | Removes **all** occurrences   | O(log n) |
| `find(val)`  | Returns iterator or end()     | Same                          | O(log n) |
| `count(val)` | 0 or 1                        | Actual count                  | O(log n) |
| `lower_bound()` | First element >= val       | Same                          | O(log n) |

**Full Example (set + multiset):**
```cpp
set<int> s{3, 1, 2, 2};           // becomes {1,2,3}
multiset<int> ms{1, 1, 2, 2};

s.insert(10);
cout << s.count(2);   // 1

ms.insert(1);
cout << ms.count(1);  // 3
```

**📊 Difference Between `std::multiset` and `std::multimap`**

| Feature                  | `std::multiset`                         | `std::multimap`                           |
|--------------------------|-----------------------------------------|-------------------------------------------|
| Stores                   | Only values (with duplicates)           | Key + Value pairs (duplicate keys allowed)|
| Access by key            | Values act as keys                      | Use key to get all values                 |
| Common use case          | Sorted collection with duplicates       | One key having multiple values            |
| `equal_range()`          | Not needed                              | Very useful                               |

**📊 Difference Between `std::set` and `std::multiset`**

| Feature             | `std::set`                              | `std::multiset`                           |
|---------------------|-----------------------------------------|-------------------------------------------|
| Duplicates allowed? | No                                      | Yes                                       |
| `insert(10)` twice  | Second insert is ignored                | Both values are stored                    |
| `count(10)`         | Returns 0 or 1                          | Returns actual number of occurrences      |
| `erase(10)`         | Removes the value (if exists)           | Removes **all** occurrences of the value  |
| Use case            | Unique elements only                    | Need to store duplicates in sorted order  |

### 🗺️ std::map & multimap

**map** = Unique keys + Sorted by key  
**multimap** = Duplicate keys allowed

**💡 Analogy:** Dictionary (word → meaning). Map = one meaning per word. Multimap = multiple meanings allowed.

**Sub-topics explained in detail:**

- `m[key] = val` (most common but careful!)
- `m.insert({key, val})`
- `make_pair`

- find(), count(), erase(), lower_bound(), equal_range()

- Shows map usage + iteration

- `equal_range(key)` returns pair of iterators for all values of a key

**💡 Analogy:** Dictionary (word → meaning). Map = one meaning per word. Multimap = multiple meanings allowed.

> **Underlying**: **Red-Black Tree** (key → value pairs)  
> **Memory**: Heap allocated  
> **Key trait**: Sorted by **key**  
> **Implication**:  
> - All operations → **O(log n)**  
> - `map` → unique keys  
> - `multimap` → duplicate keys allowed

| Function           | Description                          | Time     |
|--------------------|--------------------------------------|----------|
| `m[key] = val`     | Insert / update                      | O(log n) |
| `insert({k,v})`    | Insert (safer)                       | O(log n) |
| `find(key)`        | Search                               | O(log n) |
| `count(key)`       | 0 or 1 (map) / actual (multimap)     | O(log n) |
| `erase(key)`       | Remove                               | O(log n) |
| `lower_bound()`    | First key >=                         | O(log n) |
| `equal_range()`    | Get range for a key (multimap)       | O(log n) |

**Full Practical Example (map):**
```cpp
map<string, int> marks;
marks["Rohit"] = 95;
marks["Priya"] = 88;
marks.insert({"Amit", 92});

cout << marks["Rohit"] << endl;

for (auto &p : marks) {
    cout << p.first << " → " << p.second << endl;
}
```

**📊 Difference Between `std::set` and `std::map`**

| Feature              | `std::set`                              | `std::map`                                |
|----------------------|-----------------------------------------|-------------------------------------------|
| Stores               | Only unique values                      | Key + Value pairs                         |
| Access               | Values act as their own keys            | Access value using key (`m[key]`)         |
| Iteration            | Just the values                         | Pairs (`first` = key, `second` = value)   |
| Common use case      | Unique sorted collection                | Dictionary / key-value lookup             |

**📊 Difference Between `std::set` and `std::map`**

| Feature              | `std::set`                              | `std::map`                                |
|----------------------|-----------------------------------------|-------------------------------------------|
| Stores               | Only unique values                      | Key + Value pairs                         |
| Access by key        | Values act as keys                      | Use key to get value (`m[key]`)           |
| Common use case      | Unique sorted collection                | Dictionary / key-value lookup             |
| Iteration elements   | Just the values                         | Pairs (`first` = key, `second` = value)   |

**📊 Difference Between `std::map` and `std::multimap`**

| Feature                  | `std::map`                              | `std::multimap`                           |
|--------------------------|-----------------------------------------|-------------------------------------------|
| Duplicate keys allowed?  | No                                      | Yes                                       |
| `m[key] = value`         | Allowed (overwrites if key exists)      | Not allowed                               |
| `insert({key, val})`     | Ignores if key already exists           | Always inserts (even if key exists)       |
| `count(key)`             | 0 or 1                                  | Actual number of values for the key       |
| How to get all values    | `m[key]` or `find()`                    | Use `equal_range(key)`                    |
| Common use case          | Unique key → value mapping              | One key can have multiple values          |

## 4. Unordered Containers

Use **hashing** instead of trees → **very fast** average case.

**📌 Sub-topics covered in this section:**

- unordered_multiset

### ⚡ unordered_set, unordered_map, etc.

**Key Difference from ordered versions:**

> **Underlying**: **Hash Table** (buckets + hash function)  
> **Memory**: Heap allocated  
> **Key trait**: Elements placed by hash value  
> **Implication**:  
> - Average case → **O(1)** for insert/erase/find  
> - No ordering  
> - Worst case → **O(n)** (rare, with collisions)

| Feature        | set / map          | unordered_set / unordered_map |
|----------------|--------------------|-------------------------------|
| Order          | Always sorted      | No guaranteed order           |
| Speed          | O(log n)           | **O(1) average**              |
| Implementation | Red-Black Tree     | Hash Table                    |

| Function     | Description                 | Time (avg) |
|--------------|-----------------------------|------------|
| `insert()`   | Add element                 | O(1)       |
| `find()`     | Search                      | O(1)       |
| `erase()`    | Remove                      | O(1)       |
| `count()`    | Existence check             | O(1)       |

**Full Example:**
```cpp
unordered_set<int> us{10, 20, 30};
unordered_map<string, int> scores;
scores["Rohit"] = 95;

if (us.find(20) != us.end()) {
    cout << "Found!";
}
```

**📊 Difference Between `std::set` and `std::unordered_set`**

| Feature              | `std::set`                              | `std::unordered_set`                      |
|----------------------|-----------------------------------------|-------------------------------------------|
| Ordering             | Always sorted (ascending)               | No guaranteed order                       |
| Average time         | O(log n)                                | **O(1)**                                  |
| Worst case           | O(log n)                                | O(n) (rare)                               |
| Implementation       | Red-Black Tree (BST)                    | Hash Table                                |
| Memory usage         | Higher                                  | Usually lower                             |
| When to use          | Need sorted order                       | Fast lookup, order doesn't matter         |

**📊 Difference Between `std::unordered_set` and `std::unordered_map`**

| Feature              | `unordered_set`                          | `unordered_map`                           |
|----------------------|------------------------------------------|-------------------------------------------|
| Stores               | Only unique values                       | Key + Value pairs                         |
| Access               | Check existence (`find`, `count`)        | Get value using key (`um[key]`)           |
| Use case             | Fast unique membership test              | Fast key-to-value lookup                  |

### 1. Vector & Sequence Containers

**Q1: What is the difference between `std::vector` and `std::array`?**

**Answer:**
- `array` → Fixed size at compile time. No dynamic resizing.
- `vector` → Dynamic size. Can grow/shrink at runtime.

**When to use:**
- Use `array` when size is known and fixed (better performance, no heap allocation).
- Use `vector` for most cases (flexibility).

**Q2: Why is `reserve()` important in vector? What happens without it?**

**Answer:**
Without `reserve()`, every time capacity is exceeded, vector reallocates memory and copies all existing elements (O(n) amortized cost per insertion in worst case).

```cpp
vector<int> v;
v.reserve(100000);   // Pre-allocates memory → no reallocations
```

**Q3: Explain iterator invalidation in `std::vector`. Give an example.**

**Answer:**
Insertions and deletions in the middle or at the beginning can invalidate iterators, references, and pointers.

```cpp
vector<int> v = {1,2,3,4};
auto it = v.begin() + 1;
v.insert(it, 99);     // it is now invalid!
```

**Safe way:** Recalculate iterator after modification.

**Q4: When would you choose `std::deque` over `std::vector`?**

**Answer:**
- Frequent insertions/deletions at **both** ends.
- Need random access + fast push_front/pop_front.
- Vector is better for mostly append-only + random access scenarios.

**Q5: Difference between `std::list` and `std::forward_list`?**

**Answer:**
- `list` = Doubly linked list (forward + backward traversal, `push_back` supported)
- `forward_list` = Singly linked list (only forward, lower memory overhead)

Use `forward_list` when you only need forward iteration and want to save memory.

### 2. Associative Containers

**Answer:**

| Feature       | set              | unordered_set     |
|---------------|------------------|-------------------|
| Order         | Sorted           | No order          |
| Time          | O(log n)         | O(1) average      |
| Implementation| Red-Black Tree   | Hash Table        |

Use `unordered_set` when order doesn't matter and you want speed.

**Q7: How does `std::map` work internally?**

**Answer:**
Implemented using **Red-Black Tree** (self-balancing BST). Keys are always sorted. Insertion, deletion, and search are O(log n).

**Q8: What is the difference between `map` and `multimap`?**

**Answer:**
- `map` → Keys are unique
- `multimap` → Duplicate keys allowed

Use `multimap` when one key can have multiple values (e.g., multiple phone numbers per person).

**Q9: How do you find all values for a key in a `multimap`?**

**Answer:**
```cpp
auto range = mm.equal_range(key);
for (auto it = range.first; it != range.second; ++it) {
    cout << it->second;
}
```

**Q10: Can you use a custom comparator with `set` or `map`?**

**Answer:** Yes.

```cpp
struct cmp {
    bool operator()(int a, int b) { return a > b; }
};
set<int, cmp> s;   // descending order
```

Or using lambda in C++11+.

### 3. Unordered Containers & Hashing

**Q11: What can cause worst-case O(n) performance in `unordered_set`?**

**Answer:**
- Hash collisions (poor hash function)
- Adversarial input (hash flooding attacks)

Modern compilers use good hash functions, but you can provide a custom hasher if needed.

**Q12: How would you count frequency of words using STL?**

**Answer:**
```cpp
unordered_map<string, int> freq;
for (const auto& word : words) freq[word]++;
```

### 4. Container Adapters

**Q13: How would you implement a min-heap using `priority_queue`?**

**Answer:**
```cpp
priority_queue<int, vector<int>, greater<int>> minHeap;
```

**Q14: Difference between `stack` and `queue`?**

**Answer:**
- `stack` = LIFO (Last In First Out)
- `queue` = FIFO (First In First Out)

### 5. Algorithms

**Q15: Difference between `std::sort` and `std::stable_sort`?**

**Answer:**
- `sort` → Faster, not stable (equal elements may change relative order)
- `stable_sort` → Slower, preserves relative order of equal elements

**Q16: When would you use `nth_element` instead of `sort`?**

**Answer:**
When you only need the nth smallest/largest element (or the top k), not the full sorted array. Average time is O(n) vs O(n log n).

```cpp
nth_element(v.begin(), v.begin() + k, v.end()); // kth element is in correct position
```

**Q17: How do you copy only even numbers from one vector to another?**

**Answer:**
```cpp
copy_if(src.begin(), src.end(), back_inserter(dest),
        [](int x){ return x % 2 == 0; });
```

### 6. Modern C++ & Performance

**Q18: What is `std::span` and why is it useful?**

**Answer:**
A non-owning view of contiguous memory. It avoids copying data and works with arrays, vectors, and raw pointers.

```cpp
void process(span<int> data);   // flexible API
```

**Q19: Explain `emplace_back` vs `push_back`. When is emplace better?**

**Answer:**
`emplace_back` constructs the object directly in the container (avoids temporary object creation).

```cpp
vec.emplace_back(1, "Hello");           // better
vec.push_back({1, "Hello"});            // creates temporary
```

Especially useful for complex objects.

**Q20: What are the benefits of using `emplace` over `insert` in sets/maps?**

**Answer:**
Avoids creating a temporary object that would be moved/copied into the container.

### 7. Miscellaneous & Tricky Questions

**Q21: What happens when you access `m[key]` in a map if the key doesn't exist?**

**Answer:**
It inserts the key with a default-constructed value (0 for int, empty string, etc.). This is often unexpected — prefer `find()` or `count()`.

**Q22: How do you sort a vector of pairs by the second value?**

**Answer:**
```cpp
sort(v.begin(), v.end(), [](const auto& a, const auto& b){
    return a.second < b.second;
});
```

**Q23: Can we store user-defined objects in STL containers?**

**Answer:** Yes, but they must be:
- Copyable / Movable
- For associative containers: must define `<` operator or provide comparator

**Q24: What is the time complexity of `v.size()` in vector vs list?**

**Answer:**
- `vector`: O(1)
- `list`: O(1) (C++11+)

**Q25: How would you remove all even numbers from a vector efficiently?**

**Answer:**
```cpp
v.erase(remove_if(v.begin(), v.end(), [](int x){ return x%2==0; }), v.end());
```
